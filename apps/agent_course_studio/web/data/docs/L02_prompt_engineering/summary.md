# L02 章节总结：System Prompt 与 Prompt 迭代

## 一句话总结

Prompt 是 Agent 的行为说明书。System Prompt 定义身份、能力边界、行为准则、输出格式和示例；测试用例和迭代机制决定它能否从“看起来能用”变成“稳定可复用”。

## System Prompt 五要素

| 要素 | 作用 | 常见写法 |
| --- | --- | --- |
| 角色定义 | 让模型知道自己是谁 | “你是一个代码 Review 助手” |
| 能力边界 | 明确能做什么、不能做什么 | “只审查代码，不生成业务承诺” |
| 行为准则 | 规定执行顺序和判断标准 | “先指出风险，再给修改建议” |
| 输出格式 | 让结果可读、可比、可解析 | Markdown 表格、JSON schema、固定字段 |
| 示例 | 稳定风格和边界 | Few-shot 输入输出样例 |

## 三种关键模式

- Few-shot：给模型看范例，减少格式漂移。
- CoT：适合复杂推理和多步分析，但生产中要注意不要泄露冗长推理过程。
- ReAct：让模型在思考、行动、观察之间循环，是工具型 Agent 的常见执行范式。

配套示例见 `resources/PROMPT_EXAMPLES.md`，里面包含情感分类、结构化提取、代码生成、工具调用决策、数学推理、逻辑推理、Agent 规划和代码调试。

## Few-Shot 与 CoT 使用原则

| 技术 | 适用场景 | 关键原则 |
| --- | --- | --- |
| Few-Shot | 分类、格式转换、结构化输出、工具路由 | 示例要典型、多样、格式一致，通常 2-5 个即可 |
| CoT | 数学推理、逻辑判断、多步骤规划、调试 | 推理步骤要可验证、结构化，并收束到明确结论 |
| Few-Shot + CoT | 固定格式的复杂推理 | 示例中同时展示思考路径和最终输出格式 |
| Zero-Shot CoT | 快速原型和通用推理增强 | 用“请一步一步思考”触发，但生产输出应控制冗长推理 |

## 工业级 System Prompt 的常见设计模式

| 模式 | 说明 |
| --- | --- |
| Identity & Persona | 明确身份、任务范围、语气和工作目标 |
| Tool Integration | 规定工具何时调用、参数格式、失败回退和禁止事项 |
| Output Formatting | 用固定结构降低下游解析成本 |
| Safety Guardrails | 约束越权、隐私、危险操作和高风险内容 |
| Error Recovery | 规定工具失败、信息不足、格式错误时如何处理 |
| Context Management | 说明如何使用历史消息、文件、用户偏好和中间状态 |

## 本章实操复盘

1. `practice/03_structured_output.py`
   - 目标：让模型输出可编程 JSON。
   - 关键点：Prompt 里明确 schema，代码里做 `json.loads` 和 JSON 子串提取兜底。

   - 目标：用同一组测试用例比较 Prompt V1/V2。
   - 关键点：V1 只定义角色，V2 增加职责边界和输出格式，因此更稳定。

3. `resources/SYSTEM_PROMPT_REFERENCES.md`
   - 目标：从真实产品 prompt 中学习结构，而不是直接照抄。
   - 关键点：重点看角色、工具、输出、安全、错误恢复和上下文管理。

## 常见误区

- 误区一：Prompt 写得越长越好。
  修正：长不等于清晰，好的 Prompt 应该结构明确、边界可测试。

- 误区二：只测一个样例就上线。
  修正：至少要覆盖正常输入、越界输入、模糊输入和格式压力测试。

- 误区三：要求 JSON 就一定能得到合法 JSON。
  修正：仍要做解析、校验、重试和兜底。

- 误区四：真实产品 prompt 可以直接复制。
  修正：应学习结构和模式，尊重来源许可和安全边界，为自己的场景重新设计。

## 面试题速查

| 问题 | 回答要点 |
| --- | --- |
| 如何改写“帮我总结这篇文章，越详细越好”？ | 补角色、输出结构、长度限制、证据要求和不确定性标注 |
| Prompt 很好但 token 太贵怎么办？ | 压缩规则、减少重复、保留代表性 few-shot、摘要长上下文 |
| 如何评估 Prompt 效果？ | 固定测试集 + 正确率、格式稳定性、越界率、幻觉率、成本、延迟 |
| 怎么降低幻觉？ | 禁止编造、标注不确定、接入检索/工具、要求证据、关键结论复核 |
| Few-shot 越多越好吗？ | 不是，优先选覆盖边界的代表样本，避免 token 浪费和样本冲突 |
| CoT 一定更好吗？ | 不一定，简单任务、低延迟任务或不宜暴露推理时可能有害 |
| JSON 不稳定怎么办？ | 降温、schema、few-shot、字段校验、重试、兜底解析、结构化输出能力 |
| prompt 很长时如何分配 token？ | System 放稳定规则，User 放当前目标，历史做摘要/检索，示例少而精 |
