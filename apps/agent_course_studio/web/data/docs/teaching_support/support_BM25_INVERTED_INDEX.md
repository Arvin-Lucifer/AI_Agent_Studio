# BM25 和倒排索引简单介绍

这份教辅资料用于帮助学生理解关键词检索的底层逻辑。它不追求复杂公式，而是解释 BM25 为什么比朴素关键词计数更可靠，以及倒排索引为什么能让搜索变快。

适合配合第 5 讲 RAG、Hybrid Search、关键词检索、企业知识库检索一起讲。

## 一、BM25 核心原理

BM25 做的事情很朴素：**给查询词和文档算相关性分数，分数越高越相关。**

它主要看三件事：

1. 词越稀有，权重越高。
2. 词出现次数越多，分数越高，但会封顶。
3. 文档越长，惩罚越重。

最终分数可以理解为：**每个查询词对文档的贡献加起来。**

### 1. 稀有词更重要

查询：

```text
苹果 手机
```

语料里的情况：

- “手机”：100 篇文档都有，很常见，权重低。
- “苹果”：只有 5 篇文档有，很稀有，权重高。

BM25 的行为：

- 文档出现“苹果”，分数涨得明显。
- 文档只出现“手机”，分数涨得很少。

结论：**稀有词更能区分文档。**

### 2. 词频会封顶

查询：

```text
电脑
```

文档 A：

```text
电脑
```

文档 B：

```text
电脑电脑电脑电脑电脑电脑电脑电脑电脑电脑
```

朴素 TF-IDF 容易认为：B 比 A 强很多，因为“电脑”出现了很多次。

BM25 会更克制：B 只比 A 强一点点。

原因是 BM25 有词频饱和：出现 2-3 次已经说明这篇文档和词有关，再重复很多次不会无限加分。

结论：**BM25 不鼓励堆砌关键词。**

### 3. 长文档会被惩罚

查询：

```text
人工智能
```

文档 X：

```text
100 字，出现 2 次“人工智能”
```

文档 Y：

```text
2000 字，出现 4 次“人工智能”
```

虽然 Y 里出现次数更多，但它很长，命中密度并不高。BM25 会做长度归一化，让长短文档更公平地比较。

可能结果：

```text
文档 X 分数 > 文档 Y 分数
```

结论：**长文档不会天然占便宜。**

## 二、完整小案例

查询：

```text
特斯拉 电动车
```

三篇文档：

1. 文档 1：特斯拉电动车很好看。
2. 文档 2：特斯拉电动车特斯拉电动车特斯拉电动车……重复很多。
3. 文档 3：电动车是未来趋势。

BM25 排序：

```text
文档 1 > 文档 2 > 文档 3
```

原因：

- 文档 1：短、命中两个词、没有堆砌，最理想。
- 文档 2：命中很多次，但词频被封顶，而且文档长度被惩罚。
- 文档 3：只命中“电动车”，没有命中更有区分度的“特斯拉”。

超短总结：

1. 词越稀有越重要。
2. 词重复太多不无限加分。
3. 文档太长会扣分。

## 三、BM25 和 TF-IDF 的关系

TF-IDF 的基本想法：

```text
词在本文出现频率 x 词在全局稀有程度
```

也就是：

- TF：词在当前文档出现得多不多。
- IDF：词在整个语料里稀不稀有。

BM25 可以理解成 TF-IDF 的工业级升级版。底层思路相似，都是用“词频 + 稀有度”判断相关性，但 BM25 修复了 TF-IDF 的两个常见问题。

### 1. TF 处理更稳

TF-IDF：

- 词出现越多，分数越高。
- 如果不加限制，容易被重复关键词刷分。

BM25：

- 词频会饱和。
- 出现 3 次和出现 10 次，差距不会特别大。

### 2. 文档长度更公平

TF-IDF：

- 通常不直接处理文档长度问题。
- 长文档词更多，更容易命中查询词。

BM25：

- 自带长度归一化。
- 长文档不会因为字数多就天然占便宜。

### 对比例子

查询：

```text
电动车
```

三篇文档：

- 文档 A：电动车很好开。
- 文档 B：电动车电动车电动车……重复几十遍的大长文。
- 文档 C：汽车怎么选。

TF-IDF 可能排序：

```text
B > A > C
```

BM25 更可能排序：

```text
A > B > C
```

因为 B 的重复词被封顶，又被长度惩罚；A 更短、更精准。

## 四、倒排索引是什么

搜索系统不能每次都把所有文档从头扫一遍。倒排索引就是为了让“查词”变快。

### 1. 正排索引

正排索引是按文档存内容：

```text
Doc1 -> 苹果、手机、电池
Doc2 -> 华为、手机、快充
Doc3 -> 苹果、耳机
```

它适合看一篇文档里有什么词。

但如果用户搜“苹果”，系统就要一篇篇扫，看哪些文档包含“苹果”。文档一多就很慢。

### 2. 倒排索引

倒排索引反过来，按词存文档：

```text
苹果 -> Doc1, Doc3
手机 -> Doc1, Doc2
华为 -> Doc2
耳机 -> Doc3
```

一句话：**词是 key，文档列表是 value。**

用户搜一个词时，系统直接拿到包含这个词的文档列表，不用遍历全文。

### 3. 检索例子

现有文档：

- Doc1：苹果手机
- Doc2：华为手机
- Doc3：苹果耳机

倒排索引：

```text
苹果 -> Doc1, Doc3
手机 -> Doc1, Doc2
华为 -> Doc2
耳机 -> Doc3
```

用户搜索：

```text
苹果 手机
```

检索过程：

1. 查“苹果”：得到 Doc1、Doc3。
2. 查“手机”：得到 Doc1、Doc2。
3. 取交集：Doc1 同时包含两个词。
4. 返回 Doc1，或把候选文档交给 BM25 打分排序。

## 五、BM25 和倒排索引怎么配合

倒排索引负责快速找到候选文档：

```text
查询词 -> 候选文档列表
```

BM25 负责给候选文档排序：

```text
候选文档 -> 相关性分数 -> 排序结果
```

所以它们的关系是：

- 倒排索引解决“去哪找”。
- BM25 解决“谁更相关”。

在 RAG 里，BM25 常用于关键词召回，向量检索用于语义召回。两者结合就是 Hybrid Search：

```text
BM25 召回精确关键词
向量检索召回语义相近内容
Reranker 统一重排
```

## 六、课堂讲解口径

可以这样给学生解释：

> 倒排索引像一本词典。你查“苹果”，词典直接告诉你哪些文档有“苹果”。BM25 像一个裁判，它看每篇候选文档里词是否稀有、出现多少次、文档多长，然后给它们打相关性分。

再补一句：

> BM25 不是理解语义的模型，它更擅长精确词匹配。所以在 RAG 中，它常和 embedding 向量检索搭配使用。

## 七、小练习

### 练习 1：手写倒排索引

给定：

```text
Doc1: 苹果 手机 电池
Doc2: 华为 手机 快充
Doc3: 苹果 耳机
Doc4: 手机 充电器
```

写出：

```text
苹果 -> ?
手机 -> ?
快充 -> ?
耳机 -> ?
```

### 练习 2：判断 BM25 排序

查询：

```text
LangChain Agent
```

文档：

1. `LangChain 可以构建 Agent。`
2. `Agent Agent Agent Agent Agent 是未来。`
3. `LangChain 是一个开发框架，支持工具、检索和 Agent。`

思考：哪篇可能排第一？为什么？

### 练习 3：RAG 中怎么用

如果企业知识库里有大量制度文档和 API 文档，哪些问题更适合 BM25？

- 精确术语：接口名、制度编号、错误码。
- 人名、部门名、系统名。
- 用户输入包含明确关键词的问题。

哪些问题更适合向量检索？

- 问法模糊。
- 同义表达多。
- 用户不知道准确术语。

## 八、面试表达

如果被问“BM25 是什么”，可以这样答：

> BM25 是一种关键词检索排序算法，用来衡量查询词和文档的相关性。它继承了 TF-IDF 的词频和稀有度思想，但加入了词频饱和和文档长度归一化，所以不会因为关键词重复堆砌或文档很长就给过高分。在 RAG 中，BM25 常用于关键词召回，和向量检索组成 Hybrid Search。

如果被问“倒排索引是什么”，可以这样答：

> 倒排索引是搜索系统的基础结构。它不是按文档存词，而是按词存文档列表。用户搜一个词时，系统可以直接找到包含这个词的文档，不需要遍历所有文档。BM25 通常在倒排索引召回出的候选文档上做相关性打分和排序。
