# 第 5 讲：RAG 检索增强生成 —— 让 Agent 拥有专属知识库

## 5.1 什么是 RAG？

RAG = Retrieval-Augmented Generation，中文叫“检索增强生成”。

不要被名字吓到，可以用一个很直观的比喻理解：

- 不用 RAG 的 LLM：闭卷考试。模型只能靠参数里已有的知识回答，遇到没学过、过期或私有知识时容易编造。
- 使用 RAG 的 LLM：开卷考试。回答前先翻资料，找到相关段落，再基于资料回答。

RAG 的核心思路：

```text
用户问题
  ↓
先从知识库检索相关内容
  ↓
把检索内容 + 用户问题 一起给 LLM
  ↓
LLM 基于资料生成答案
```

例子：

```text
用户问题：我们公司的年假政策是什么？

检索结果：
资料1：员工入职满 1 年享有 5 天年假。
资料2：年假需提前 3 个工作日申请。

增强后的 prompt：
根据以下资料回答问题：
资料1：员工入职满 1 年享有 5 天年假。
资料2：年假需提前 3 个工作日申请。
问题：我们公司的年假政策是什么？

最终回答：
根据公司政策，入职满 1 年可享有 5 天年假，需要提前 3 个工作日申请。
```

RAG 主要解决四类问题：

- 知识更新：模型训练后的新政策、新文档、新接口说明。
- 私有知识：公司内部手册、项目文档、客服知识库。
- 幻觉控制：让模型基于上下文回答，而不是凭空编造。
- 来源追溯：回答后能标注来自哪个文档、哪一页、哪个 chunk。

## 5.2 RAG 的完整流程

RAG 通常分成两个阶段：离线阶段和在线阶段。

### 离线阶段：准备知识库

```text
原始文档：PDF / Word / 网页 / Wiki / 数据库
  ↓
文档解析：提取文本、表格、图片、元数据
  ↓
数据清洗：去重、去页眉页脚、去乱码、权限同步
  ↓
文本分块：把长文档切成 chunk
  ↓
向量化：把 chunk 转成 embedding
  ↓
索引存储：向量库、关键词索引、结构化索引
```

### 在线阶段：回答问题

```text
用户提问
  ↓
问题理解 / Query Rewrite
  ↓
检索相关 chunk
  ↓
可选：rerank 精排
  ↓
构造上下文 prompt
  ↓
LLM 基于上下文生成答案
  ↓
返回答案 + 引用来源
```

要注意：RAG 不是“向量库 + LLM”这么简单。一个可用的 RAG 系统至少包括数据治理、切分、检索、重排、上下文构造、生成约束、引用、评估和权限控制。

## 5.3 向量和向量数据库

向量就是一组数字。Embedding 模型可以把文本转换成一组数字，这组数字代表文本语义。

```text
"猫喜欢吃鱼"  -> [0.12, 0.85, 0.33, ...]
"小猫爱吃鱼"  -> [0.13, 0.84, 0.31, ...]
"今天天气好"  -> [0.91, 0.11, 0.72, ...]
```

语义相近的文本，向量距离通常更近。RAG 就利用这一点，把用户问题和文档 chunk 都转成向量，再用相似度检索相关内容。

常见向量数据库：

- Chroma：轻量、本地、适合实验。
- FAISS：高性能向量检索库。
- Milvus：企业级向量数据库。
- pgvector：Postgres 生态里的向量检索方案。
- Elasticsearch/OpenSearch：常用于关键词 + 向量混合检索。

本章练习为了避免依赖安装卡住，先实现一个标准库版本地检索索引。它不是生产向量库，但能完整展示 RAG 的核心流程：文档、chunk、索引、检索、上下文、生成。

## 5.4 实操：从零搭建 RAG 系统

本章实践分成四步。

### Step 1：准备示例文档

运行：

```bash
python practice/11_prepare_knowledge_base.py
```

脚本会在 `data/knowledge_base/` 下创建三份文档：

- `company_policy.txt`：休假政策。
- `tech_guide.txt`：代码提交、部署、API 规范。
- `onboarding.txt`：新员工入职流程、试用期、常用系统。

这些文档模拟公司内部知识库。

### Step 2：构建本地索引

运行：

```bash
python practice/12_build_local_index.py --query "年假有几天？"
```

脚本会做：

1. 加载 `data/knowledge_base/` 中的 `.txt` 文件。
2. 清洗文本。
3. 按段落和长度切成 chunk。
4. 用标准库实现一个轻量 TF-IDF 风格索引。
5. 保存到 `data/rag_index/local_index.json`。
6. 用测试 query 检索 TopK chunk。

### Step 3：实现 RAG 问答

运行：

```bash
python practice/13_rag_agent.py --question "我入职2年了，有几天年假？" --show-context
```

脚本会：

1. 加载本地索引。
2. 检索与问题相关的 chunk。
3. 把 chunk 格式化成带来源的上下文。
4. 要求 LLM 只基于上下文回答。
5. 输出答案和引用来源。

如果只想看检索，不想调用模型，可以运行：

```bash
python practice/13_rag_agent.py --question "代码提交有什么规范？" --no-llm --show-context
```

### Step 4：比较 chunk size

运行：

```bash
python practice/14_chunking_compare.py
```

脚本会比较 `chunk_size=200/500/1000` 时同一批问题的检索结果。你会看到：

- chunk 太小：语义容易断裂。
- chunk 太大：噪声变多，占用上下文。
- overlap 能缓解断裂，但会增加重复内容。

## 5.5 分块策略对比

| 分块策略 | 特点 | 适用场景 |
| --- | --- | --- |
| 固定长度 | 按字符数硬切，简单直接 | 格式统一、结构弱的文本 |
| 递归分块 | 优先按段落、句子、标点切 | 通用场景，推荐默认方案 |
| 结构化分块 | 按标题、章节、表格、代码函数切 | 技术文档、手册、代码库 |
| 语义分块 | 按语义边界聚类 | 主题混杂、长报告 |

Chunk size 没有绝对标准。更重要的是用评估集验证：

- 正确 chunk 是否能进 TopK？
- TopK 噪声是否太多？
- 生成答案是否忠实于上下文？
- 引用是否能支撑答案？

## 5.6 Rewrite、Rerank 和 Hybrid Search

RAG 的效果不只取决于向量库。

### Query Rewrite

用户问题常常很口语，比如“这个怎么配置？”多轮对话里还可能有指代。Query Rewrite 会把问题改写成更适合检索的形式。

常见方式：

- 规则改写：同义词、缩写、术语表。
- LLM 改写：把口语问题改成文档术语。
- 多路查询：生成多个 query 后合并结果。
- 对话感知改写：结合历史，把“这个”补全成明确对象。

### Rerank

Embedding 检索适合快速召回，但不一定排序最准确。Reranker 会对候选文档做更精细的相关性判断。

典型流程：

```text
Retriever 召回 Top 50
  ↓
Reranker 精排
  ↓
选择 Top 5 放入上下文
```

### Hybrid Search

纯向量检索对语义问题友好，但对错误码、函数名、API 路径、产品型号、专有名词不一定可靠。关键词检索对精确匹配更强。

Hybrid Search 常见做法：

- 向量召回 TopK。
- BM25/关键词召回 TopK。
- 合并去重。
- 用 RRF 或加权融合排序。
- 可选 reranker 精排。

## 5.7 PDF RAG：不能简单转纯文本

PDF 是 RAG 最容易出问题的数据源之一。核心原则：

不要先把 PDF 粗暴转成纯文本再切分，而是先做版面解析和结构还原。

PDF 中的信息类型包括：

- 正文文本：进入主 chunk。
- 标题、章节、页码、文档名：进入 metadata，必要时拼入 chunk 前缀。
- 图片、图表、表格：转换成文本描述或结构化数据，不能简单丢弃。
- 页眉、页脚、水印、脚注：区分是否有语义价值，重复无效信息要清洗。

详细处理方案见 [resources/PDF_RAG_PROCESS.md](../resources/PDF_RAG_PROCESS.md)。

## 5.8 Agentic RAG

普通问答 RAG 通常是一次流程：

```text
用户问题 -> 检索 -> 生成
```

Agentic RAG 则把检索放进 Agent 决策流程：

- Agent 判断是否需要检索。
- Agent 决定检索哪个知识源。
- Agent 可以把复杂问题拆成多个子问题。
- Agent 可以多轮检索、检查证据是否足够。
- Agent 可以把 RAG 和 SQL、API、搜索、文件工具一起使用。

Agentic RAG 的难点：

- 中间一步检索错误会放大。
- 多轮检索需要状态管理。
- 多工具协作需要权限和边界。
- 证据不足时需要拒答或追问，而不是强行生成。

## 5.9 课后作业

必做：

1. 用自己准备的 3-5 个文档搭建一个 RAG 问答系统。
2. 记录至少 5 个问题的检索结果和最终回答。
3. 对一个失败问题做排查：数据、切分、检索、上下文、Prompt、生成哪一环出错？
4. 实现一个简单 rerank 或 rewrite 优化。

选做：

- 对比 `chunk_size=200/500/1000` 的召回质量。
- 把本地标准库索引替换成 Chroma/FAISS。
- 处理一个 PDF，设计 metadata 和引用格式。
- 尝试 PageIndex 或 LLM Wiki 的选型分析。
