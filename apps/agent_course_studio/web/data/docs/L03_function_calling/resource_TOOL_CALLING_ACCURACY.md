# L03 专题：如何提升 Function Calling 准确率

当系统里可用工具越来越多时，问题会从“会不会调工具”升级成“如何选对工具、少选错工具、少漏选工具，同时控制延迟、成本和安全风险”。

## 一、为什么工具一多准确率会变差

1. 工具语义重叠
   例如 `search_web`、`search_docs`、`search_order`、`search_customer` 都叫搜索，模型很难区分。

2. 描述过长、过像、过泛
   “用于查询各种信息”“支持多种检索能力”这类描述在工具少时还能凑合，工具一多就失效。

3. 暴露工具太多
   把 80 个工具一次性塞给模型，本质上是在让模型做全量搜索，注意力分散且选择空间过大。

4. 工具成本不对称
   查缓存和发邮件不是一个风险级别；读操作和写操作也不能同权暴露。

## 二、高质量回答框架

如果被问“工具很多时，如何提升 function calling 准确率”，可以按这 5 层回答：

1. 减少同时暴露给模型的工具数量。
2. 让工具边界更清晰，降低语义重叠。
3. 在模型调用前增加路由层。
4. 对高风险 / 高成本工具做额外约束。
5. 基于线上数据持续评估和优化。

## 三、方法一：动态注入工具

不要每次都把所有工具给模型。先做一层轻路由，把工具收缩到小集合。

例子：

```text
用户："帮我看看明天下午有没有时间"
路由域：Communication
候选工具：
- get_calendar_events
- check_freebusy
- create_event
```

加分表达：

```text
当工具数量扩张后，核心问题不再只是参数抽取，而是先做工具召回，再做工具选择。
```

## 四、方法二：工具边界清晰

差的工具：

- `search_data`
- `query_info`
- `get_record`

好的工具：

- `search_customer_profile`
- `search_order_records`
- `search_support_tickets`

命名经验：

```text
动作 + 对象 + 场景约束
```

## 五、方法三：前置路由层

常见路由方式：

| 方式 | 适用场景 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 规则路由 | 边界明显 | 便宜、稳定、可解释 | 覆盖有限 |
| 小模型 / 分类器 | 意图类别稳定 | 灵活且较稳 | 需要标注数据 |
| Embedding 召回 | 工具规模大 | 可扩展到大量工具 | 只能召回，不能直接执行 |

Embedding 召回工具，本质上是 RAG for tools。

## 六、方法四：工具分层治理

工具不能同权暴露。

| 层级 | 示例 | 治理方式 |
| --- | --- | --- |
| 只读低风险 | 查天气、查订单、查文档 | 可直接开放 |
| 写操作 | 建任务、发消息、更新记录 | 参数校验、预览、确认 |
| 高危工具 | 删除数据、退款、改权限 | RBAC、人工确认、审计、沙箱 |

重点：工具准确率优化不能只看是否选对，还要看选错时损失是否可控。

## 七、方法五：工具描述和 few-shot

好的 description 要写清：

1. 工具做什么。
2. 什么时候用。
3. 不什么时候用。
4. 参数含义和示例。

例子：

```text
用于根据客户姓名、手机号、客户 ID 查询客户主档信息。
当用户想查客户基本资料、客户状态、归属销售时使用。
不用于查询订单、退款、工单。
```

工具选择 few-shot 示例：

```text
"查客户余额" -> get_customer_account_balance
"查退款状态" -> search_refund_requests
"公司的报销政策" -> search_internal_docs
```

## 八、客服 Agent 案例

用户说：

```text
帮我看一下王敏最近的退款处理到哪一步了
```

稳定流程：

1. 意图路由：售后 / 退款状态查询。
2. 候选工具收缩：`search_customer_profile`、`search_refund_requests`、`search_support_tickets`。
3. 多步调用：先找 customer_id，再查退款，再查工单。
4. 总结输出：说明退款进度和关联工单状态。

如果用户说：

```text
帮我通知负责销售，这个客户退款投诉比较严重
```

这已经从查询变成“查询 + 判断 + 写操作”。稳定做法是生成通知草稿并请求确认，再执行发送。

## 九、线上验证

不要只靠主观感觉，要看：

- tool selection accuracy
- false positive / false negative
- 平均工具调用次数
- end-to-end success rate
- latency
- user satisfaction
- high-risk confirmation rate
