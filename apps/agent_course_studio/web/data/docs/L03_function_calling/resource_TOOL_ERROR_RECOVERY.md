# L03 专题：工具执行报错后，怎么让 Agent 更稳

真实线上系统里，工具调用失败不是异常情况，而是常态。高质量 Agent 不是永不出错，而是出错后可控、可恢复、可解释。

## 一、核心回答框架

1. 先对错误分类，不同错误不同处理。
2. 对可恢复错误做有限自动恢复。
3. 对不可恢复错误及时降级或中止。
4. 对写操作保证幂等和确认，避免误执行。
5. 记录观测数据，支持复盘和持续优化。

## 二、错误分类

| 类型 | 典型场景 | 稳定处理 |
| --- | --- | --- |
| 参数错误 | 日期格式错、缺字段、枚举非法 | schema 校验、自动归一化、必要时澄清 |
| 用户输入不足 | 重名、缺订单号、联系人不明确 | 向用户补充询问 |
| 下游瞬时故障 | timeout、502/503/504、限流 | 有限重试 + 指数退避 |
| 权限 / 鉴权错误 | 403、token 过期、跨租户访问 | 立即止损，不绕过权限 |
| 业务空结果 / 语义不确定 | 查无结果、多候选、字段缺失 | 消歧、fallback 或保守输出 |

## 三、恢复策略

### 1. 自动修复

适合参数格式、别名、轻微 schema 问题：

- “北京时间明天下午” -> 标准时间范围。
- “高优” -> `priority=P1`。
- “沪” -> `Shanghai`。

自动修复不能用于高风险写操作的关键参数。

### 2. 有限重试

适合 timeout、502/503、限流。

最佳实践：

- 2-3 次以内。
- 指数退避。
- 不对确定性失败重试。
- 写操作必须有幂等保障。

### 3. Fallback

主工具失败时，使用质量边界明确的降级方案：

- 实时接口失败 -> 缓存或快照。
- 向量检索失败 -> 关键词检索。
- 搜索 API 超时 -> 次级索引。

fallback 结果必须标注时效性和可信边界。

### 4. 澄清 / 人工确认

适合：

- 信息不足。
- 歧义很大。
- 涉及写操作。
- 恢复成本高且错误代价大。

高危操作如发邮件、删除记录、提交审批、退款，应要求显式确认。

## 四、Partial Success

很多任务不是非黑即白。

例子：

```text
已完成：找到客户，查到退款状态。
未完成：财务提醒发送失败。
建议下一步：是否生成提醒草稿供你手动发送？
```

成熟 Agent 不应因为最后一步失败就把整个任务都说成失败。

## 五、观测与日志

每次工具调用建议记录：

- trace_id
- session_id
- tool_name
- input args
- validation result
- error type
- retry count
- latency
- final status

线上指标：

- tool success rate
- retry rate
- timeout rate
- auth failure rate
- parameter validation failure rate
- partial success rate
- fallback usage rate

## 六、面试口径

工具执行报错后，想让 Agent 更稳，核心不是简单重试，而是先分类、再恢复、最后止损。

我会先把错误分成参数错误、用户输入不足、下游瞬时故障、权限错误、业务空结果几类。参数错误优先校验和自动修复；信息不足要向用户澄清；timeout/502/限流可以有限重试；权限错误必须中止；空结果或歧义结果要消歧或降级。系统还要支持 partial success，并把每次工具调用的错误类型、重试次数和 latency 打到日志里，支撑后续优化。
