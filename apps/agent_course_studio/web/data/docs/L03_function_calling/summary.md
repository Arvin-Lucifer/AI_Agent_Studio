# L03 章节总结：Function Calling 核心机制

## 一句话总结

Function Calling 让模型从“生成文本”升级为“选择工具并传参”。模型负责决策，代码负责执行；成熟系统还必须处理工具选择准确率、错误恢复、安全风控、观测指标和噪声结果治理。

## 四步流程

1. 定义工具：名称、描述、参数 schema。
2. 发送用户请求和工具列表给模型。
3. 模型返回 tool calls。
4. 程序执行工具，把结果写回模型，模型生成最终回复。

## 本章实操

| 脚本 | 作用 |
| --- | --- |
| `practice/05_function_calling.py` | 基础 Function Calling：天气、时间、计算 |
| `practice/06_parallel_function_calling.py` | 并发执行多个独立 tool calls |

## 工具 Schema 设计要点

- 一个工具只做一件事。
- 名称使用“动作 + 对象 + 场景约束”。
- description 写清“什么时候用”和“不什么时候用”。
- 参数类型明确，关键字段 required。
- 可枚举值使用 enum。
- 避免深层嵌套和过多可选参数。

## 工具很多时的准确率框架

1. 动态注入工具，不全量暴露。
2. 优化工具边界，减少语义重叠。
3. 增加路由层：规则、分类器、embedding top-k。
4. 高风险 / 高成本工具分层治理。
5. 用线上日志持续评估：选择准确率、参数错误率、重试率、完成率。

## 错误恢复框架

| 错误类型 | 处理方式 |
| --- | --- |
| 参数错误 | schema 校验、格式归一化、必要时澄清 |
| 用户输入不足 | 向用户补充询问，不盲重试 |
| 下游瞬时故障 | 有限重试 + 退避 |
| 权限错误 | 立即止损，不绕过权限 |
| 空结果 / 语义不确定 | 消歧、fallback 或保守输出 |

## 噪声结果治理框架

- 工具结果只是 observation，不是 truth。
- 先做 cleaning、filtering、normalization。
- 不迷信 top1，增加 rerank。
- 高风险结论做多源交叉验证。
- 明确检测冲突。
- 约束模型只基于 evidence 输出。
- 证据不足时保守回答或请求澄清。

## 安全护栏

- 工具分级：读 / 写 / 高危执行。
- 高危操作 require confirmation。
- RBAC 和权限检查必须在代码层完成。
- 写操作需要幂等键和审计日志。
- 外部内容进入模型前要防间接提示注入。

## 面试金句

- 当工具数量扩张后，核心问题不再只是参数抽取，而是 tool retrieval + tool invocation。
- 工具准确率优化不能只看“是否选对”，还要看“即使选错，损失是否可控”。
- 成熟 Agent 的稳定性，不体现在永不出错，而体现在出错后依然可控、可恢复、可解释。
- 工具返回的是 observation，不是 truth；真正的能力在于把 observation 变成可信 evidence。
