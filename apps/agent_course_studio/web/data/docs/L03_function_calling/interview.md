## 1. Function Calling 和纯 Prompt 输出 JSON 有什么区别？

纯 Prompt 输出 JSON 是让模型“写一段看起来像调用意图的数据”；Function Calling 是让模型在受约束的工具 schema 下选择工具和生成参数，由代码负责真正执行。

关键区别：

- Function Calling 有明确工具名、参数 schema 和调用协议。
- 工具执行权仍在代码侧，不在模型侧。
- 能把模型决策和业务执行分离，便于校验、审计和权限控制。

## 2. Function Schema 怎么设计？

高质量 schema 包含：

- 清晰工具名：动作 + 对象 + 场景。
- 明确 description：什么时候用、不什么时候用。
- 参数类型和必填字段。
- 枚举、范围、格式约束。
- 安全边界和示例。

坏例子是 `query_info`，好例子是 `search_order_records`、`get_customer_profile`。

## 3. Schema 太复杂会有什么问题？

- 模型更容易填错字段。
- 上下文变长，成本升高。
- 工具边界变模糊。
- 后端校验和错误恢复更困难。

工程上应拆分职责，保持单个工具能力聚焦。

## 4. 工具很多时，如何提升调用准确率？

回答框架：

1. 不一次性暴露所有工具，先做工具路由或动态注入。
2. 工具命名和描述要减少语义重叠。
3. 对高风险工具加确认、权限和审计。
4. 为工具选择提供 few-shot 示例。
5. 用线上日志评估工具选择准确率、漏选率、误选率。

加分表达：

```text
工具数量上来之后，问题不只是参数抽取，而是先做工具召回，再做工具选择，本质上是 RAG for tools。
```

## 5. 多轮 Function Calling 怎么避免死循环？

做法：

- 设置 `max_steps`。
- 对重复 tool call 做检测。
- 要求每轮观察结果必须推动状态变化。
- 工具失败后有限重试。
- 达不到条件时降级、澄清或中止。

## 6. 工具执行报错后怎么让 Agent 更稳？

核心不是简单重试，而是分类处理。

| 错误类型 | 处理方式 |
| --- | --- |
| 参数错误 | schema 校验、自动归一化、必要时澄清 |
| 用户输入不足 | 追问用户 |
| timeout / 502 / 限流 | 有限重试 + 指数退避 |
| 权限错误 | 立即中止，不绕过权限 |
| 空结果 / 多候选 | 消歧、fallback 或保守输出 |

成熟系统还要支持 partial success，例如“已查到订单，但发送提醒失败”。

## 7. 工具返回结果不可靠或噪声很大怎么办？

回答要点：

- 把工具返回当作 observation，不直接当 truth。
- 清洗、过滤、归一化。
- 对多源结果做交叉验证。
- 对冲突信息标注来源、时间和可信度。
- 生成答案时只基于可信 evidence。

面试金句：

```text
在 Agent 系统里，工具返回的是 observation，不是 truth；真正的能力在于把 observation 变成可信 evidence。
```

## 8. Function Calling 有哪些安全风险？

- 模型误调用高危工具。
- Prompt Injection 诱导调用越权工具。
- 参数中夹带危险内容。
- 写操作重复执行。
- 工具返回敏感信息后被直接泄露。

防护：

- 最小权限。
- 工具分级。
- 参数校验。
- 写操作预览和确认。
- 审计日志。
- 高危工具人工审批。

## 9. 如何评估 Function Calling？

指标：

- 工具选择准确率。
- 参数准确率。
- 漏调率。
- 误调率。
- 工具执行成功率。
- 平均步骤数。
- 延迟和成本。
- 高风险误调用次数。

评估集要覆盖正常请求、模糊请求、越权请求、工具失败和噪声结果。
