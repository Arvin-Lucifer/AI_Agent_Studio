# L03 课后小测

请先独立作答，再看参考答案。

1. Function Calling 的核心流程是哪 4 步？
2. 模型会不会直接执行函数？为什么？
3. Function Calling 和 prompt 要求输出 JSON 有什么区别？
4. function schema 设计有哪些原则？
5. schema 太复杂会带来什么问题？
6. 多轮 tool calling 如何避免死循环？
7. 工具很多时如何提升工具选择准确率？
8. 工具执行报错有哪些常见类型？
9. 哪些错误适合重试，哪些不适合？
10. 如何防止模型调用高危函数？
11. 如何评估 function calling 做得好不好？
12. 工具结果不可靠时，为什么不能直接喂给模型？

## 参考答案（示例）

1. 定义工具、发送用户请求、模型返回 tool calls、代码执行工具并回写结果。
2. 不会。模型只输出调用意图，实际执行由程序代码完成。
3. Function Calling 有 schema、工具选择机制和参数结构；纯 prompt JSON 更依赖文本生成，稳定性更弱。
4. 参数名清晰、类型明确、required 合理、enum 约束、描述清楚、一个函数只做一件事。
5. 参数更容易填错、嵌套导致稳定性下降、可选字段乱补、调试困难、工具选择变差。
6. 设置最大 tool call 次数、超时和预算限制，对无新信息调用 early stop，加入状态机或 step guard。
7. 动态注入工具、清晰工具边界、前置路由、工具分层治理、few-shot 和线上评估。
8. 选错工具、参数错误、缺字段、工具失败、空结果、权限失败、无限循环、工具结果噪声。
9. timeout、502/503、限流等瞬时错误适合有限重试；权限错误、参数缺失、用户信息不足不适合盲重试。
10. 工具分级、显式确认、RBAC、参数白名单、审计日志、沙箱、人类确认。
11. 看工具选择准确率、参数准确率、schema valid rate、tool success rate、retry rate、任务完成率、延迟和成本。
12. 工具结果只是 observation，可能弱相关、过期、缺字段或冲突，需要清洗、过滤、归一化、rerank 和证据约束。
