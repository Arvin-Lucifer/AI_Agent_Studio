# Agent 翻车场景与调优手册

Agent 调优的第一步不是改 prompt，而是先判断翻车类型。

## 常见场景

| 翻车场景 | 症状 | 修复方法 |
| --- | --- | --- |
| 幻觉 | 编造不存在的信息 | 只基于检索结果回答，无证据拒答，强制引用 |
| 死循环 | 反复调用同一个工具 | 设置最大步数，检测重复 tool call，失败后降级 |
| 工具误调 | 该查知识库却去计算 | 优化工具描述，加“何时不用”，做工具选择评测 |
| 格式错乱 | JSON 不合法或字段缺失 | 使用 schema、few-shot、解析失败有限重试 |
| Prompt 注入 | 用户绕过约束或索要系统提示词 | 输入校验、外部内容当数据、高风险确认 |
| 成本失控 | 简单问题走复杂链路 | 路由分层、缓存、限制最大步骤和 token |
| 延迟过高 | 用户等不到结果 | 流式输出、异步任务、关键路径剥离 |

## 幻觉修复 Prompt 片段

```text
## 重要约束
- 只基于知识库检索到的内容回答，不要添加知识库中没有的信息。
- 如果知识库中没有相关信息，直接回复“抱歉，我在知识库中没有找到相关信息”。
- 不要猜测或编造数据、日期、数字。
- 引用信息时注明来源文档。
```

## 死循环治理

需要同时做三件事：

- `max_steps`：硬上限。
- 重复检测：同一工具 + 同一参数连续出现时中止。
- 状态推进判断：每次工具调用必须产生新信息。

## 工具误调治理

工具 description 要写清：

- 什么时候使用。
- 输入参数是什么。
- 什么时候不要使用。
- 和相似工具有什么区别。

工具多时，应先做工具路由，不要一次性把所有工具暴露给模型。

## 评测驱动调优

每次调优都要记录：

- 修改了什么。
- 修复了哪些 case。
- 是否导致旧 case 退化。
- 是否增加了新安全风险。

调优不是“感觉好一点”，而是“同一套评测集上指标变好且安全不退化”。
