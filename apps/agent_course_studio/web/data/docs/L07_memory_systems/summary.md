# L07 章节总结：记忆系统

## 一句话总结

Agent 记忆不是把所有历史都塞给模型，而是根据记忆类型选择保存、检索、压缩、更新和遗忘策略。

## 核心概念

| 概念 | 本章理解 |
| --- | --- |
| 短期记忆 | 当前会话上下文，受上下文窗口限制 |
| 长期记忆 | 跨会话保存用户画像、偏好、事实和任务结果 |
| 外部知识 | RAG 知识库、数据库、文档系统 |
| 滑动窗口 | 只保留最近 N 轮，简单但会丢早期关键信息 |
| 摘要压缩 | 把旧对话折叠成摘要，节省 token 但会丢细节 |
| 用户画像 | key/value 型记忆，适合 upsert 覆盖 |
| 事实流 | 事件型记忆，适合 append、检索和审计 |
| 知识图谱 | 用实体-关系-实体表达结构化关系和多跳推理 |
| 遗忘机制 | 主动删除、过期、压缩或合并记忆，避免噪声和合规风险 |

## 本章代码地图

- `21_memory_window.py`：滑动窗口短期记忆。
- `22_memory_summary.py`：摘要压缩短期记忆。
- `23_long_term_memory_json.py`：JSON 长期记忆。
- `24_hybrid_memory_graph.py`：语义入口 + 知识图谱扩展。
- `25_agent_with_long_memory.py`：长期记忆工具接入 Agent。

## 必须记住的工程判断

- `MemorySaver` 不是长期记忆，它只是会话检查点。
- 长期记忆要区分 profile、preferences、facts、KG，不同类型写入语义不同。
- 检索记忆时不能只看“相似”，还要看时间、重要性、权限、冲突和可信度。
- 记忆越多不一定越好，过期和低质量记忆会污染回答。
- 用户必须能显式删除个人记忆，隐私字段要优先考虑合规。

- 用户画像为什么适合 upsert，而事实流适合 append？

- 为什么长期记忆必须有遗忘机制？
- 把长期记忆接入 Agent 时，怎样避免跨用户记忆泄漏？
