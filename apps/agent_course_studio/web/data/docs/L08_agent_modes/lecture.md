# 第8讲 Agent 模式：从 RAG/ReAct 到多 Agent 协作

第 8 讲的总题目是“Agent 模式”。这一章不是只讲“多 Agent”，而是要解决一个更前置的问题：面对一个真实任务，我们到底应该选择哪种 Agent 形态。

有些问题只需要模型直接回答，有些问题必须先检索企业知识库，有些问题需要多步推理和工具调用，还有些复杂任务适合拆成多个角色协作。模式选错了，系统就会在准确率、成本、延迟和可维护性之间一起失控。

## 8.1 为什么要学习 Agent 模式

Agent 不是越复杂越好。工程里的核心判断是：用刚好足够的模式解决问题。

一个客服机器人问“你是谁”，直接回答就够；问“我入职 3 年有几天年假”，需要查企业制度；问“P0 工单响应时间和 API 发布审批有什么共同要求”，就可能需要跨库检索和多步综合；要写一份行业研报，则适合拆成调研、分析、写作、审核多个角色。

常见模式可以粗略分成六类：

| 模式 | 核心能力 | 适合场景 | 主要风险 |
| --- | --- | --- | --- |
| Direct | 直接生成 | 常识问答、闲聊、格式转换 | 无私域知识、容易编造 |
| RAG | 检索增强生成 | 企业知识库、制度问答、可追溯问答 | 检索不到、检索噪声、单跳能力有限 |
| ReAct | 推理和行动交替 | 多跳问答、工具调用、跨源融合 | 步数失控、工具误用、延迟上升 |
| Plan-and-Execute | 先规划再执行 | 长任务、复杂流程、自动化项目 | 对简单问答过重，计划错误会传播 |
| Reflection | 自我检查和修正 | 高风险生成、代码修复、质量审核 | token 成本高，可能陷入循环 |
| Multi-Agent | 多角色协作 | 研报、复杂项目、需要专业分工的任务 | 协作协议复杂，冲突和循环更难控 |

本章要建立的判断习惯是：先问任务是否需要私域知识，再问是否需要多步工具调用，最后才考虑是否需要多个 Agent。

## 8.2 企业知识库问答：RAG 为主，ReAct 为辅

企业知识库问答助手最常见的推荐模式是：RAG 为主 + ReAct 为辅。

RAG 作为主干，是因为企业知识有四个特点。

第一，数据私域性。企业制度、工单、研发文档、合同、价格表通常不在通用模型训练集中，必须通过检索注入上下文。

第二，可溯源。企业问答经常涉及合规、审计和责任归属，答案不能只说“模型认为”，而要能指出来自哪个文档、哪个版本、哪个 chunk。

第三，知识时效。制度、流程、价格、接口文档都会更新。相比微调，外挂检索的更新成本更低，也更容易回滚。

第四，幻觉抑制。纯 LLM 直答容易把不确定内容说得很像真的；RAG 通过证据约束，能明显降低编造风险。

但纯 RAG 也不是万能的。下面这些场景需要 ReAct 兜底：

| 场景 | 单轮 RAG 的问题 | ReAct 的价值 |
| --- | --- | --- |
| 多跳问答 | 一次检索拿不全证据 | 拆成多次检索后合并 |
| 模糊提问 | 检索词噪声大 | 先澄清或补全实体再检索 |
| 跨源融合 | 单一向量库不够 | 按问题路由不同工具或知识库 |
| 计算型问题 | 检索不出聚合结果 | 调用 SQL、计算器或报表工具 |

所以企业知识库问答的落地架构通常是：

1. 检索层：混合检索，常见组合是 BM25 + 向量 + 元数据过滤，再接 rerank。
2. 路由层：轻量分类器判断问题类型，分到 direct、单轮 RAG 或 ReAct 多步。
3. Agent 层：ReAct 的工具数量要收敛，建议 5 个以内，步数上限 3-5 步。
4. 答案层：强制引用来源；无证据时显式回答“未找到”，禁止编造。
5. 评测与回流：建立评测集，持续看命中率、忠实度、引用正确率和 bad case。

不推荐的模式也要说清楚。

纯 Fine-tune 不适合作为知识库问答主干，因为知识更新需要重训，成本高，难以溯源，也可能遗忘旧能力。

纯 Plan-and-Execute 对一次问答通常太重，延迟高，容错差。

Multi-Agent 辩论或 Reflection 大循环在问答场景里的 ROI 通常偏低，因为 token 和延迟会成倍增长，但不一定显著提升可验证正确率。

## 8.3 为什么需要多个 Agent

单个 Agent 像一个全能选手，什么都会一点，但在复杂任务里很容易出现三个问题：

- 注意力不够集中：同一个提示里既要搜索、分析、写作、审核，角色边界会混。
- 专业职责不清：出了错很难定位是信息搜集错、分析错，还是表达错。
- 流程不可控：复杂任务需要阶段性产物和检查点，单 Agent 容易一口气生成到底。

多 Agent 的价值在于分工协作。以“写一份 AI 行业研报”为例，可以拆成：

| 角色 | 职责 |
| --- | --- |
| Researcher | 搜集行业信息、政策、公司动态、数据点 |
| Analyst | 基于材料提炼趋势、机会、风险和判断 |
| Writer | 把材料组织成结构化报告 |
| Reviewer | 检查事实、结构、完整性和表达质量 |

这不是为了制造复杂度，而是为了让复杂任务有清晰的责任边界和质量关口。

## 8.4 多 Agent 的三种基础协作模式

第一种是 Supervisor 模式。一个主管 Agent 或控制逻辑负责调度多个专家 Agent。专家只做自己擅长的事情，主管决定下一步去哪。它适合客服分流、企业知识库跨域路由、研报团队、数据分析团队。

第二种是 Sequential 模式。多个 Agent 按固定流水线执行，例如“调研 -> 分析 -> 写作 -> 审核”。它简单、稳定、容易调试，适合流程明确的任务。

第三种是 Peer 模式。多个 Agent 平等讨论、互相补充，最后由一个汇总器生成结论。它适合创意讨论、方案比较、复杂评审，但成本和不确定性也最高。

本章练习采用的是 Supervisor/Sequential 混合：流程主线固定，但审核员可以决定是否把报告退回给撰写员修改。

## 8.5 LangGraph 多 Agent 研报团队

第一个练习是 `practice/26_supervisor_research_team.py`。

它用 LangGraph 的 `StateGraph` 管理状态流转，核心状态是 `TeamState`：

```python
class TeamState(TypedDict, total=False):
    topic: str
    research_data: str
    analysis: str
    report: str
    review_feedback: str
    is_approved: bool
    revision_count: int
    use_llm: bool
```

这些字段就是 Agent 之间的“交接单”。

调研员读取 `topic`，写入 `research_data`。分析师读取 `research_data`，写入 `analysis`。撰写员读取 `topic`、`research_data`、`analysis` 和可能存在的 `review_feedback`，写入 `report`。审核员读取 `report`，写入 `review_feedback`、`is_approved` 和 `revision_count`。

控制流如下：

```text
START
  -> researcher
  -> analyst
  -> writer
  -> reviewer
  -> approved ? END : writer
```

这里最值得注意的是审核返工逻辑。真实系统必须避免无限循环，所以示例里用 `revision_count` 限制最多修改次数。这个模式在企业系统中很常见：Agent 可以自我修正，但必须有预算、步数和终止条件。

示例默认使用 mock 输出，是为了保证课前脚本稳定、低成本、可离线讲解。课堂演示时可以加 `--use-llm` 调用真实模型。

## 8.6 分库分索引策略

企业知识库不能简单地把所有内容丢进一个向量库。单库方案会撞上四堵墙。

第一是权限墙。HR、法务、财务、研发数据敏感度不同，必须隔离。不能先检索出机密内容，再在答案阶段过滤。

第二是语料异构墙。长文档、FAQ、代码、工单、表格、图片的切片粒度和 embedding 模型可能完全不同。

第三是更新频率墙。制度可能一年更新一次，产品文档可能每月更新，工单可能分钟级更新。统一索引会拖慢全局。

第四是效果调优墙。同一个关键词在不同业务域含义不同，例如“审批”在财务、研发、法务里指向不同流程。单一向量空间很难兼顾所有域。

推荐切分维度是四维叠加：

| 维度 | 划分依据 | 示例 |
| --- | --- | --- |
| 业务域 | 部门或产品线 | HR、财务、法务、研发、客服 |
| 数据形态 | 文本结构 | 长文档、FAQ、代码、表格、多模态 |
| 敏感等级 | 合规分级 | 公开、内部、机密、绝密 |
| 时效性 | 更新节奏 | 静态制度、动态工单、实时指标 |

实践中通常以业务域为主键，其它维度作为子索引或元数据过滤字段。

关键实施动作包括：

- 每个业务域使用独立 collection 或 index。
- 机密级数据独立集群部署，至少在网络层隔离。
- Embedding 模型按域适配，通用域可用 `bge-m3` 或 `text-embedding-3-large`，代码域可考虑 CodeBERT、jina-code，多语言域要避免中英文混库语义漂移。
- 切片策略按域定制：制度文档按章节和标题语义切，API 文档按函数或接口切，工单整条作为 chunk，表格按行级或表级切并保留表头上下文。
- 统一元数据 Schema，例如 `doc_id, domain, sensitivity, owner_dept, valid_from, valid_to, source_url, version, lang, tags`。
- 多库并行召回后必须 rerank，因为不同库的原始分数不可直接比较。
- 权限过滤必须前置，在向量检索 filter 阶段就带上角色、部门和敏感等级。

`practice/28_multi_index_retrieval.py` 用一个小型 mock 知识库演示这个过程：先路由到业务域，再带敏感等级过滤检索，最后统一 rerank 并输出引用。

## 8.7 Agent 模式路由

`practice/27_agent_mode_router.py` 演示的是一个轻量路由器。

它先判断问题是否需要企业知识。如果不需要，走 direct。如果是单一事实查询，走 RAG。如果出现多跳、比较、跨源或计算型问题，走 ReAct。

课堂代码里的分类规则很简单，但它对应的是一个真实工程原则：不要让所有问题都走最重的 Agent。

一个推荐的三档路由策略是：

| 路由方式 | 适合流量 | 特点 |
| --- | --- | --- |
| 规则路由 | 高频 60% 左右 | 零模型成本、低延迟、可解释 |
| 分类器路由 | 中频 30% 左右 | 可处理更多自然语言变化 |
| Agent 路由 | 长尾 10% 左右 | 能跨域、多跳、调用工具，但成本最高 |

这也是企业知识库问答里控制成本和延迟的关键：让简单问题走简单路径，把复杂能力留给真正复杂的问题。

## 8.8 多 Agent 设计要点

第一，单一职责。每个 Agent 应该有清晰的输入、输出和职责边界。不要让一个“研究员 Agent”同时负责写作、审核和发布。

第二，状态协议清晰。多 Agent 之间不要靠模糊自然语言随意传递，应该用 `State`、JSON、Pydantic schema 或数据库字段明确交接内容。

第三，控制循环。所有返工、反思、讨论都要有最大轮数、最大 token、超时和失败兜底。

第四，工具收敛。ReAct 或多 Agent 系统里工具越多，误选工具的概率越高。工具名、描述、参数和错误返回要足够清楚。

第五，错误可观测。要记录每一步的输入、输出、工具调用、引用来源和路由决策，否则线上 bad case 很难排查。

第六，安全边界前置。权限、敏感等级、用户身份、租户隔离应该在检索和工具调用前完成，而不是等答案生成后再过滤。

## 8.10 本章核心结论

Agent 模式选择的核心不是炫技，而是匹配任务复杂度。

企业知识库问答通常以 RAG 保证可信与可追溯，以 ReAct 处理多跳与跨源长尾，再用前置路由控制成本和延迟。

分库分索引的本质，是把“通用问答”拆成多个“领域问答”子问题。领域内用最适合的切片、embedding、权限和评测策略，领域间用路由和 rerank 统一协调。

多 Agent 适合职责清晰、流程复杂、需要多阶段质量控制的任务。它不是默认选项，而是当单 Agent 或单轮 RAG 已经无法稳定完成任务时，再引入的协作模式。
