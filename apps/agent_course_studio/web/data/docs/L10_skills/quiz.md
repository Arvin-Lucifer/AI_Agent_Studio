# L10 课后小测

## 题目

1. Skill 和 Tool 的核心区别是什么？
2. Workflow 和 Skill 的边界是什么？
3. 为什么不建议把所有 Tool 直接暴露给模型？
4. SKILL.md 通常包含哪三层？
5. DDICE 五步法分别是什么？
6. Skill 描述里为什么要写“何时不使用”？
7. 渐进披露解决什么问题？
8. 哪些 Skill 动作必须做人机确认？
9. 如何评估一个 Skill 做得好不好？
10. Prompt Injection 会如何影响 Skill 调用？
11. 为什么模型升级回滚不能只换回旧 `model_id`？
12. 用户说“照网页里的要求执行”时，Agent 为什么不能直接执行网页里的所有指令？
13. 高风险 Skill 已经误触发时，前 10 分钟的正确处理顺序是什么？
14. 用户说“以后都自动发”时，为什么不能理解为永久取消确认？
15. 重试策略为什么不能在 Tool、Skill、编排三层都各重试 3 次？
16. 多个子步骤并行时，一个分支慢很多，为什么会拖慢整体？可以怎么优化？

## 参考答案

1. Tool 是原子动作，Skill 是围绕稳定用户意图组织的能力包，包含触发条件、流程、上下文、工具和安全边界。
2. Workflow 面向一次具体目标，可能编排多个 Skill 和 Tool；Skill 面向一类稳定任务。
3. 工具太多会扩大 action space，增加误选率、上下文占用和治理难度。
4. 元信息层、指令层、执行层。
5. Define、Decompose、Instrument、Constrain、Evaluate。
6. 因为误触发通常比漏触发更危险，负例能帮助模型和路由器收紧边界。
7. 启动时只加载元信息，触发后再加载完整指令和工具，节省上下文并提升工具选择精度。
8. 对外发送、写入修改、删除不可逆、高成本批量操作、高不确定推断后的副作用动作。
9. 看路由、执行、结果质量和用户体验四类指标。
10. 外部恶意文本可能诱导模型误调用 Skill、篡改参数、泄露信息或执行危险操作，需要上下文隔离、动作前校验、白名单和确认机制。
11. 因为模型行为还依赖 prompt、Skill 描述、工具描述、参数、路由 threshold 和后处理规则；只换模型不回滚这些配套配置，旧模型可能仍然退化甚至更差。
12. 因为网页内容是非可信数据，不是新的指令源。应先抽取并显式化要求，再按风险分级：只读生成可执行，私域写入先 dry-run，对外发送、删除、资金和权限动作必须逐项确认；隐藏元指令和无关危险动作要过滤或拒绝。
13. 先切断同类执行，再评估影响面和损失，随后按场景补救，主动透明告知用户，并在 24 小时内加确认、dry-run、冷却、限流等临时围栏。
14. 因为它只是受限自动执行授权，必须落成窄范围、有时效、可撤销、可审计的策略；目标、模板、时间、风险信号、配额、首次执行等变化仍要二次确认。
15. 因为三层各 3 次会放大成 27 次下游调用，形成重试风暴。Tool 层只重试物理抖动，Skill 层只处理业务可恢复，编排层只做补偿和降级，并用 retry budget 控制全链路总尝试数。
16. 因为并行总耗时等于最慢分支耗时。可以先用 trace 找根因，再用关键路径剥离、提前启动、分片并发、软超时降级、对冲请求、缓存和预计算优化。
