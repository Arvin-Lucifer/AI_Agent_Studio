# 第10讲 Skill 相关：让 Agent 能力模块化复用

本章聚焦 Skill 的基础认知、设计方法、典型案例和工程化治理：从天气查询、代码审查、办公助手渐进加载，到重试分层、并行长尾延迟、自动授权、网页指令边界、误触发止损和模型升级回滚。

## 10.1 从 Chatbot 到 Agent：Skill 是关键跃迁

一个裸 LLM 只能“说话”，但 Agent 需要“做事”。Skill 就是赋予 Agent 行动能力的标准化封装。

| 维度 | Chatbot，无 Skill | Agent，有 Skill |
| --- | --- | --- |
| 能力 | 仅文本生成 | 搜索、计算、调用 API、操作文件 |
| 边界 | 受模型知识截止日期限制 | 可以访问实时数据和外部系统 |
| 协作 | 单轮问答 | 多步规划和工具调用循环 |
| 价值 | 辅助思考 | 替代一部分执行动作 |

Skill 的重点不是“多包一层代码”，而是把一类稳定用户意图变成可发现、可加载、可执行、可评测的能力模块。

## 10.2 Function Calling、MCP、A2A 的位置

老师材料里把三类协议放在同一张图里比较，非常关键。

| 特性 | Function Calling | MCP | A2A |
| --- | --- | --- | --- |
| 解决问题 | 单模型到工具 | 标准化工具接入 | Agent 间协作 |
| 类比 | 给 AI 一本菜谱 | AI 的 USB-C 接口 | AI 的团队协作协议 |
| 连接方式 | 每次请求声明 | 持久双向连接 | 握手 + 任务路由 |
| 跨模型 | 通常绑定具体模型生态 | 模型无关 | 跨厂商 |
| 适用场景 | 快速原型 | 生产级工具集成 | 多 Agent 工作流 |

一句话：

```text
Function Calling 是“怎么调用”。
MCP 是“怎么连接”。
A2A 是“怎么协作”。
```

它们是分层互补关系，不是简单替代关系。

## 10.3 Tool、Skill、MCP 三层架构

### Tool：最小行动单元

Tool 是 Agent 可以调用的原子操作，本质是函数签名 + 执行逻辑。

```json
{
  "name": "get_weather",
  "description": "查询指定城市的天气",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {"type": "string", "description": "城市名称"}
    },
    "required": ["city"]
  }
}
```

Tool 的局限：

- 每个工具独立，缺乏任务上下文。
- 没有明确触发规则，模型不知道什么时候该用。
- 工具多了以后，模型选择困难，tool choice 精度下降。

### Skill：有上下文的工具包

Skill = Tool + 触发条件 + 执行流程 + 上下文知识。

它是一组相关能力的封装，回答的是“我能完成哪类任务”，而不是“我能调用哪个函数”。

| 维度 | Tool | Skill |
| --- | --- | --- |
| 粒度 | 单个函数 | 能力包 |
| 上下文 | 通常无 | 有触发规则和执行指令 |
| 组合性 | 独立调用 | 可编排多个步骤 |
| 发现性 | 需手动注册 | 可通过 Registry 或目录发现 |
| 维护性 | 常耦合在代码中 | 可独立文件夹、可热插拔 |

### MCP：Skill 的传输标准

MCP 定义能力如何被 Agent 发现、连接和使用。它可以承载 Tools、Resources、Prompts 等能力，是 Skill 生态和外部工具系统之间的标准连接层。

## 10.4 SKILL.md 规范

SKILL.md 是当前常见的声明式 Skill 格式。它通常由三层组成：

| 层级 | 内容 | 示例 |
| --- | --- | --- |
| 元信息层 | name、description、trigger、version、tags | `weather-query` |
| 指令层 | Markdown Prompt、使用说明、执行步骤 | 何时使用、何时不使用、输出格式 |
| 执行层 | 脚本、API、Tool、子 Agent | `get_weather`、`get_air_quality` |

一个简化模板：

```markdown
---
name: code-reviewer
description: |
  审查代码变更，检测 Bug、安全漏洞和风格问题。
  当用户提供代码片段或 PR 链接并要求审查时触发。
trigger: 用户提供代码片段、PR 链接，或要求 code review
version: 1.0.0
tags: [code-quality, security, review]
---

# Code Reviewer Skill

## 何时使用
- 用户提交代码片段请求审查。

## 执行步骤
1. 读取用户提供的代码。
2. 检查逻辑、安全、性能和风格。
3. 按严重/警告/建议分级输出。

## 注意事项
- 安全问题一律标为严重。
- 必须给出修复建议。
```

## 10.5 DDICE：Skill 设计五步法

老师材料里给出的五步法是 DDICE：

| 步骤 | 含义 | 要回答的问题 |
| --- | --- | --- |
| Define | 定义场景 | 谁用、什么时候用、替代什么人工动作 |
| Decompose | 拆解任务 | 任务能否拆成稳定子步骤 |
| Instrument | 工具化 | 哪些步骤用 Tool、Prompt、代码或 MCP |
| Constrain | 约束化 | 输入输出 schema、失败兜底、越界限制 |
| Evaluate | 评测闭环 | 金标样本、自动打分、线上回流 |

触发词设计要点：

- 高召回 + 低误触，覆盖同义表达，但排除无关高频词。
- 场景锚定，带上“学生/作业/错题/讲解”等限定词。
- 反例清单，明确写出什么时候不要触发。

## 10.6 Skill 设计五原则

第一，单一职责。每个 Skill 只做一件事，做好一件事。

第二，描述即触发器。description 是 Agent 判断是否调用 Skill 的核心依据。好的描述要写清“何时用 + 做什么 + 何时不用”。

第三，渐进披露。启动时不要一次性暴露所有工具，先加载 Skill 元信息，触发后再加载完整说明和工具。

第四，幂等安全。只读操作天然更安全，写操作需要确认、幂等键、审计和可撤销策略。

第五，可观测。每个 Skill 调用都要能追踪路由、参数、执行步骤、失败原因和结果质量。

- 工具集合：`get_weather(city)` 和 `get_air_quality(city)`。
- 执行流程：解析城市 -> 并行查询 -> 综合建议 -> 结构化输出。
- 上下文知识：不同天气、温度、AQI 的建议规则。

## 10.9 Case 3：办公助手 Skill 与渐进加载

- `office-router`：只做意图路由。

- `email-notifier`：发送邮件通知。

| 上下文占用 | 高 | 低 |
| 工具选择精度 | 低 | 高 |
| 扩展性 | 新增 Skill 常需改代码 | 新增目录和注册即可 |
| 适用场景 | 1-2 个 Skill | 3 个以上 Skill |

## 10.10 Skill 的安全、鲁棒性与评测

Skill 不是只要“能跑”就够。一个好的 Skill 至少要具备：

- 目标边界清晰。
- 输入输出契约明确。
- 执行过程可控。
- 结果可解释。
- 可观测、可评测、可迭代。
- 高风险动作有安全机制。

高风险动作必须 human-in-the-loop：

- 对外发送，如邮件、群消息、批量通知。
- 写入和修改，如创建任务、修改文档。
- 删除和不可逆动作。
- 高成本批量操作。
- 从非结构化文本推断 owner 后直接写入。

评估 Skill 要看四类指标：

- 路由指标：正确触发率、误触发率、漏触发率。
- 执行指标：成功率、耗时、重试率、降级率。
- 结果质量：任务完成率、字段完整率、用户二次修改率。
- 用户体验：确认通过率、中断率、满意度、纠错率。

评测集要覆盖：

- 标准场景。
- 边界场景。
- 负例场景。
- 异常场景。
- 安全场景。

## 10.11 重试策略：Tool 层、Skill 层与编排层

重试不是“失败了就多试几次”。在 Agent 系统里，重试应该按失败语义分层：

| 层级 | 重试什么 | 重试依据 | 典型次数 |
| --- | --- | --- | --- |
| Tool 层 | 瞬时性、物理性故障 | 网络、协议、传输层信号 | 2-3 次，毫秒级退避 |
| Skill 层 | 业务语义可恢复错误 | 业务错误码 + 上下文判断 | 0-1 次，秒级退避 |
| 编排层 | 不重试，只补偿或降级 | 业务决策 | 0 次 |

核心判据是：

```text
这个错误换个时间再试，有没有可能成功？
```

能，就靠近失败点就地重试；不确定，需要业务上下文判断，就上升到 Skill 层；权限、参数、业务冲突这类语义错误，绝不重试。

### 10.11.1 Tool 层重试物理抖动

Tool 层处理与业务逻辑无关的瞬时故障：

- 网络超时。
- 连接重置。
- DNS 抖动。
- HTTP 502、503、504。
- 429 限流，但必须尊重 `Retry-After`。
- Token 过期：刷新 token 后重试一次。

策略是指数退避 + jitter：

```text
attempt 1 -> fail -> wait 200ms +/- jitter
attempt 2 -> fail -> wait 500ms +/- jitter
attempt 3 -> fail -> 上抛 Skill 层
```

Tool 层不能把错误压成一句 `request failed`，必须把 status code、错误类型、`Retry-After` 等语义透传给 Skill 层。

### 10.11.2 Skill 层重试业务可恢复错误

Skill 层需要结合业务上下文判断是否可恢复：

- 文档正在被他人编辑锁定，可以稍后再试。
- 异步任务未就绪，例如会议纪要还在转写中，可以轮询。
- 幂等写入超时，可能已经成功，需要先查再决定。

典型流程：

```text
写 Base 记录返回超时
-> Skill 层用 client_token 查询是否已写入
-> 已写入：返回成功
-> 未写入：再写一次
```

Skill 层重试必须满足：

- 最多 1 次业务重试，或轮询但带总时长上限。
- 写操作必须有 `client_token` 或 `idempotency_key`。
- 下游不支持幂等时，不做写重试。

### 10.11.3 哪些错误不重试

以下错误立即上抛：

- 400 参数错误。
- 401/403 权限或鉴权失败。
- 404 资源不存在。
- 422 参数校验失败。
- 业务规则冲突，例如日程时间冲突、文档已删除。
- 非幂等写操作的未知状态，除非业务层能查询确认。

这些错误重试不会解决，只会放大问题。

### 10.11.4 编排层不叠加重试

编排层看到下游失败，应该做决策，不应该继续包一层循环。

编排层职责：

1. 读取 Skill 返回的 `retriable` 字段。
2. `retriable=true` 时，最多触发一次明确的 Skill 级恢复动作。
3. `retriable=false` 时，走补偿、降级、告知用户路径。

如果 Tool 层 3 次、Skill 层 3 次、编排层 3 次，最坏会变成 27 次调用。这就是重试风暴。

### 10.11.5 关键约束

第一，Retry Budget。

- 全链路总尝试数建议不超过 4 次。
- 编排层默认 0 次重试。
- 下游异常时优先 fail-fast，而不是继续堆重试。

第二，幂等性。

- 非幂等操作重试前必须带 `client_token` 或 `idempotency_key`。
- 不支持幂等时禁止重试。

第三，错误码透传。

- Tool 层必须把原始错误语义传给 Skill 层。
- Skill 层返回要带 `retriable`、`failed_step`、`user_action_required`。

第四，熔断配合。

- 连续 N 次失败，熔断器打开。
- 熔断期间跳过重试，直接 fail-fast。

## 10.12 并行编排中的长尾延迟优化

多个子步骤都能并行，并不代表整体就一定快。并行编排的总耗时通常由最慢分支决定：

```text
parallel_latency = max(branch_latency)
```

所以 P99 往往被那个最慢的分支拖住。优化思路只有两类：要么让慢分支变快，要么让慢分支不阻塞主响应。

### 10.12.1 先诊断慢在哪里

不同根因对应不同策略：

| 根因 | 识别特征 | 适用优化 |
| --- | --- | --- |
| IO 等待 | CPU 空闲，网络或磁盘活跃 | 流式处理、分片、预取 |
| 数据量大 | 耗时随输入规模线性增长 | 分页并发、MapReduce |
| 服务端慢 | 单次调用就慢，本地无法优化 | 异步化、预测性提前触发 |
| 伪并行 | 外层并行，内部仍是顺序 for 循环 | 改成真并发 |

先测再优。没有 trace 数据就谈优化，是盲人摸象。

### 10.12.2 关键路径剥离

最常用的优化方式，是把慢步骤从“必须完成才能响应”降级为“后台继续”。

适用场景：

- 生成配图。
- 归档。
- 统计。
- 长报告润色。
- 其他不影响主决策的结果。

模式：

```text
快分支完成 -> 先返回主响应
慢分支继续 -> 标记 pending
慢分支完成 -> IM/通知/WebSocket/轮询后补
```

关键是区分 `blocking`、`non_blocking` 和 `optional`。

### 10.12.3 提前启动

慢步骤如果只依赖用户输入，就不要等前置步骤全部完成。

坏流程：

```text
校验 -> 创建文档 -> 拉模板 -> 拉权限
```

好流程：

```text
入口同时启动：校验、拉模板、拉权限
校验通过后直接创建文档
```

提前启动适合模板预取、权限预拉、素材预处理、候选检索。

### 10.12.4 分片并发

慢的根因是“量大”时，把一个慢任务拆成多个小任务：

- 1000 条记录写 Base -> 拆成 10 个 100 条并发写。
- 大文档解析 -> 按章节并行处理。
- 大批量检索 -> 按 shard 并发召回再合并。

约束：

```text
分片数 x 单片 QPS <= 下游配额
```

Fan-out 不受控会把优化变成限流事故。

### 10.12.5 软超时与降级

给慢步骤设软超时，不是硬 kill。

超时后：

- 先返回降级结果。
- 标记慢结果 `pending`。
- 慢结果异步完成后覆盖更新。

例子：

```text
Minutes 精要获取超过 5s
-> 先返回逐字稿
-> 精要完成后追加更新
```

### 10.12.6 对冲请求与缓存预计算

对冲请求用于消除只读请求的长尾尖刺：

- 主请求到 P95 还没返回时，发起第二个相同请求。
- 取先返回的结果。
- 只适合只读、幂等、下游有冗余容量的场景。

缓存与预计算适合输出可复用的慢步骤：

- 按 `(user_id, 参数指纹)` 做 LRU 缓存。
- 周期性任务用 scheduler 提前计算。
- 热门模板、权限、配置提前预取。

反模式：

- 只看平均耗时，不看 P95/P99。
- 所有并行分支都 join 后才返回。
- “并行”函数里套顺序 for 循环。
- Fan-out 不考虑下游 QPS。
- Hedging 用在非幂等写操作。

## 10.13 “以后都自动发”：受限自动授权

用户说：

```text
以后都自动发。
```

这句话不是“取消确认”，而是“授予一段受限自动执行权限”。系统要做的不是从此无脑执行，而是把口头授权翻译成有范围、有时效、可撤销、可审计的策略对象。

### 10.13.1 先判断能不能接

不是所有“自动发”都能兑现。先过四个维度：

| 维度 | 可以自动化 | 需降级或拒绝 |
| --- | --- | --- |
| 影响范围 | 私人、小范围，如自己日历、个人任务、固定小群 | 群发、对外、跨部门广播 |
| 可逆性 | 可撤回、可删除、可修改 | 不可逆，如打款、合同、删除数据 |
| 合规风险 | 内部常规操作 | 财务、权限变更、审批流 |
| 用户身份 | 本人操作本人资源 | 代他人操作、涉及他人隐私 |

右侧任意一项命中，就不能接受“永久自动”。要么保留单次确认，要么走审批，要么拒绝自动化。

### 10.13.2 把口头授权落成策略对象

一句话授权必须转成规则对象：

```json
{
  "action": "send_im_notification",
  "scope": {
    "target": "group_id=project-daily",
    "content_template": "日报模板",
    "time_window": "workday 09:00-10:00"
  },
  "limits": {
    "max_per_day": 1,
    "max_total": 30
  },
  "expiry": "2026-09-16",
  "revoke_hint": "说“停止自动发日报”可随时撤销",
  "audit": true
}
```

关键设计：

- 范围必须窄：不是“以后发消息都自动”，而是“以后这类消息到这个目标自动”。
- 默认有期限：例如 90 天或次数上限，到期重新确认。
- 保留撤销路径：明确告诉用户怎么关掉。
- 每次执行留痕：用户随时能查“你替我自动做了什么”。

### 10.13.3 标准回复话术

不要只说“好的，以后自动发”。要复述边界、时效和撤销方式：

```text
好的，已记住：之后工作日早上给 XX 群发日报时自动执行，不再每次确认。
授权范围仅限此场景，30 天后或发满 20 次会再次跟你确认一次。
想随时关闭，说“停止自动发日报”即可。
```

这段话完成四件事：

1. 复述：防止用户以为授权更宽。
2. 限定：明确范围，避免越权。
3. 时效：主动给出过期条件。
4. 兜底：保留撤销入口。

### 10.13.4 仍需二次确认的情况

即使用户说过“以后都自动”，以下情况仍然要拦一次：

- 超出授权范围：内容模板、目标、时间窗口变化。
- 风险信号：收件人数量突增、内容包含敏感词、金额超阈值。
- 异常环境：异地登录、非常用设备、凌晨等反常时间。
- 达到配额：接近次数或频率上限。
- 首次执行：第一次真实执行做 dry-run 预览和确认，避免规则理解偏差。

拦截话术：

```text
你之前授权了自动发日报，但这次收件人从 1 个群变成了 5 个，超出授权范围，需要你再确认一次。
```

### 10.13.5 技术落地

授权存储：

- 写入持久化的用户偏好表或规则表。
- 不把授权只塞在对话 memory 里。
- memory 可以提醒存在规则，但不能作为执行依据。

执行前校验：

- 规则是否仍有效：未过期、未撤销。
- 本次参数是否在 scope 内。
- 是否触发豁免条件。
- 是否超过频率或总次数限制。

审计日志：

```json
{
  "time": "2026-06-18T09:05:00",
  "rule_id": "auto-send-daily-project",
  "params": {
    "target": "group_id=project-daily",
    "content_template": "日报模板"
  },
  "result": "sent_mock"
}
```

撤销机制：

- 支持自然语言撤销：“停止自动发日报”。
- 支持显式列表：“显示所有自动化规则”。
- 撤销立即生效，不等下次调用。

自动执行失败时，不要无限重试。一次失败就回退为“通知用户 + 待确认”，避免失败状态下错误放大。

## 10.14 “照网页里的要求执行”：外部内容授权边界

真实 Agent 经常会遇到这样的请求：

```text
照这个网页里的要求执行。
```

这句话听起来像用户已经授权了，但工程上不能简单理解为“网页说什么，Agent 就做什么”。这个场景的本质是：用户把一部分任务说明交给外部网页，但并没有把最高指令权交给网页。

处理关键不是一律拒绝，也不是照单全收，而是明确区分：网页可以提供哪些数据，用户授权到哪一层动作，哪些动作必须重新确认。

### 10.14.1 先拆真实意图

“照网页执行”通常有四种含义：

| 用户真实意图 | 示例 | 授权程度 |
| --- | --- | --- |
| 按网页内容生成或总结 | “按这个教程帮我写个类似的” | 内容参考，不涉及动作 |
| 按网页步骤操作自己的资源 | “照这个文档创建表格” | 私域资源，中等风险 |
| 按网页表单或要求填写提交 | “按这个申请表要求填” | 有对外动作，高风险 |
| 无条件执行网页里的指令 | “网页说什么你就做什么” | 本质上放弃审查权，必须拒绝或降级 |

前三类可以谨慎处理。第四类不能直接接受，因为它把外部内容升级成了新的指令源。

### 10.14.2 数据与指令永远分离

Prompt Injection 防御的核心原则是：网页内容永远先当“数据”，不自动升级为“指令”。

- 用户说“照网页执行”是用户指令。
- 网页里的步骤、字段、示例是非可信数据。
- 网页里夹带的 `ignore previous instructions`、角色切换、权限提升，不属于用户授权范围。
- 高风险动作必须回到用户原始意图重新校验。

类比：用户说“按合同条款办事”，不代表合同里夹了一张纸条写“把公章给我”，系统就要照做。

### 10.14.3 四步处理流程

第一步，抓取并显式化。

Agent 应先复述自己从网页里识别到的要求：

```text
我从网页中识别到 A、B、C 三个要求。
其中 C 涉及对外发送，需要你单独确认。
```

这一步把隐性内容变成显性内容，让授权重新回到用户手里。

第二步，按动作风险分级。

| 风险级别 | 例子 | 处理 |
| --- | --- | --- |
| 只读 / 生成 | 总结、翻译、起草文案 | 可直接执行 |
| 私域写入 | 在自己的 Base 里建记录、建私人日程 | Dry Run 预览后执行 |
| 对外 / 不可逆 | 发邮件、发群消息、打款、删除 | 逐项人工确认，不批量授权 |

用户的“照网页执行”最多穿透到私域写入层，不穿透对外或不可逆层。

第三步，过滤可疑指令。

进入 Skill 路由和参数抽取前，要丢弃：

- 元指令，如 `ignore previous instructions`。
- 角色切换和权限提升。
- 与当前任务无关的动作请求。
- 读取 cookie、token、本地文件或外传数据的要求。

只保留与用户原始意图直接相关的内容。

第四步，执行后审计。

每一步实际调用的 Skill、参数和来源都要留痕：

- 哪些字段来自用户。
- 哪些字段来自网页。
- 哪些网页指令被丢弃。
- 哪些动作进入 preview 或人工确认。

### 10.14.4 必须拒绝或降级的情况

以下情况即使用户坚持，也不能直接执行：

1. 网页要求读取 cookie、token、本地文件、环境变量或外传数据。
2. 网页要求跨用户、跨账号操作。
3. 网页涉及资金、权限变更或合规审批绕过。
4. 网页来源可疑，例如 HTTP 非 HTTPS、未知域名、标题和内容严重不符。

拒绝时要说清具体原因，而不是笼统说“不安全”。

示例：

```text
这个网页里包含“向 xxx@example.com 发邮件并附带 Y 信息”的要求。
这超出了我能直接执行的范围，涉及对外发送，需要你亲自确认目标和内容。
我可以帮你准备草稿，但发送动作需要你手动确认。
```

## 10.15 高风险 Skill 误触发：事后损失控制

事前确认、白名单和 dry-run 都很重要，但生产系统里仍然要准备一个现实问题：

```text
如果高风险 Skill 已经误触发了，怎么办？
```

这时第一优先级不是追责，也不是马上改代码，而是尽快止损、评估影响、透明告知和结构加固。

正确顺序是：

```text
切断 -> 评估 -> 补救 -> 告知 -> 加固
```

### 10.15.1 黄金 5 分钟：先止血

误触发发生后的第一目标是防止扩散。

立刻做三件事：

1. 切断同类执行。
2. 锁定影响面。
3. 判断是否需要紧急人工介入。

切断同类执行包括：

- 熔断该 Skill：按当前用户、会话、租户或全局停用。
- 冻结相关自动化规则：暂停同 scope 授权规则。
- 检查队列和排期：取消尚未执行的后续批次。

锁定影响面要拉取：

- 时间。
- 参数。
- 目标对象。
- 成功数和失败数。
- 是否还有持续轮询、定时任务或队列批次。

触发以下任一信号，应立即升级人工：

- 涉及外部对象，如邮件、群发、客户通知。
- 涉及资金、权限或合规。
- 影响范围超过个位数目标。
- 不可逆操作，如删除、打款、发布。

### 10.15.2 损失评估：先量化，再行动

没评估清楚就补救，可能造成二次损失。

| 维度 | 关键问题 | 决定 |
| --- | --- | --- |
| 可逆性 | 操作能撤销吗？撤销动作本身有副作用吗？ | 是否回滚 |
| 扩散性 | 是否已经被看到、转发或消费？ | 是否还能静默处理 |
| 关联影响 | 是否触发下游工作流、通知、自动回复？ | 补救范围 |

经验法则：

- 可逆 + 未扩散：静默回滚 + 事后告知。
- 不可逆或已扩散：不要硬撤，走澄清或补偿路径。
- 已触发下游：先评估下游，再回头处理上游。

### 10.15.3 分场景补救策略

可逆操作，例如建文档、建日程、建任务、写记录：

- 先保留证据副本。
- 删除或标记作废。
- 删除动作也要留审计日志。

已送达但可撤回，例如 IM、邮件：

- 撤回窗口内立即撤回。
- 超时或已读时发补充说明。
- 外部邮件用 `[更正]` 或 `[Correction]` 标题发送更正邮件。

金额或权限类：

- 立即走退款、权限回收、订单取消等人工反向流程。
- 不要用另一个自动化去修复自动化的错。
- 同步合规、风控、财务或权限系统 owner。

触发下游工作流：

- 沿链路回溯：`Skill A -> Skill B -> Webhook -> 第三方系统`。
- 先停最下游，防止继续扩散。
- 第三方系统不可控时，联系对方 owner。

### 10.15.4 透明告知用户

无论损失大小，都要主动告诉用户发生了什么，不能静默处理。

四段式告知：

1. 发生了什么。
2. 已采取措施。
3. 残余影响。
4. 用户侧行动建议。

示例：

```text
刚才在你说“整理一下日程”时，我误触发了“群发会议邀请”，向 XX 群发送了 3 条邀请。
已撤回 2 条，第 3 条超过撤回时限，已在群内发补充说明澄清。
群内 15 人可能看到了原始消息，其中 3 人已读。
如需进一步澄清，你可以在群内补一句；我已暂停该 Skill，下次相关操作会先请你确认。
```

不要隐瞒，不要淡化，不要把责任推给“模型理解错了”。系统责任不能甩给模型。

### 10.15.5 24 小时内启用临时围栏

完整复盘前，先加临时防线：

- 该 Skill 降级为需确认。
- 相似语义请求全部 dry-run。
- 同用户其他高风险 Skill 进入预防性冷却。
- 该 Skill 限流到安全水位。

这些是事中围栏，不用等 1-3 天复盘结束。

### 10.15.6 复盘产出

止血完成后再复盘。目标不是追责，而是让同类问题不再发生。

| 问题 | 产出 |
| --- | --- |
| 为什么模型会误触发？ | 用户 query 与 Skill 描述的语义重叠点 |
| 为什么没被前置拦住？ | 前置校验、置信度门槛、确认机制缺口 |
| 为什么损失这么大？ | 限流、审计、撤销、熔断是否到位 |
| 下次怎么防？ | 具体代码、配置、流程改动 |

改进优先级：

1. 让误触发不发生：Skill 描述更精确，意图分类器加置信度门槛。
2. 让误触发发生了也没事：dry-run、二次确认、配额上限、熔断阈值。
3. 让误触发发生了能快速恢复：审计日志、一键撤销、补偿脚本。

## 10.16 模型升级回滚：Skill 发布治理

模型升级回滚比传统服务回滚复杂。传统服务常见问题是“崩了”，模型升级更常见的问题是“不一样了”：没有报错，但 Skill 路由、输出风格、工具参数、拒答边界或安全判断悄悄退化。

回滚设计的核心是让“发现 -> 决策 -> 执行”三步都有依据。

### 10.16.1 版本即快照

一个可回滚的模型版本，不应该只包含 `model_id`，而应该包含完整快照：

- 模型标识和权重版本。
- Prompt 模板。
- 工具描述。
- Skill 列表和 Skill description。
- temperature、top_p、max_tokens、超时。
- 路由或分类器 threshold。
- 后处理规则。

只换模型不换配套，是伪回滚。很多线上“回滚无效”，原因是 prompt 或 Skill 描述已经跟着新模型改过，旧模型使用新配置反而更差。

### 10.16.2 流量可切分

发布系统要支持按用户、租户、会话、Skill 维度切流。切流应通过配置中心完成，不需要重启，不需要改代码，并支持细到 1% 的灰度比例。

### 10.16.3 指标可对照

回滚决策需要新旧并跑数据：

- Skill 召回率。
- 误触发率。
- 任务完成率。
- 用户终止率。
- 延迟 P50/P95/P99。
- 安全事件和对外误发率。

高风险升级应先用 shadow mode：新版本接收真实流量，但输出不展示给用户，只做对照评估。

### 10.16.4 四档回滚

| 档位 | 动作 | 适用场景 |
| --- | --- | --- |
| L1 参数调节 | 调 threshold、改 prompt 小片段、关闭某 Skill | 单一 Skill 偶发问题 |
| L2 局部切流 | 问题 Skill 或问题场景切回旧版 | 特定能力退化 |
| L3 灰度回退 | 新版本灰度比例降到 0% | 整体质量下降但无严重合规事故 |
| L4 全量回滚 | 所有流量切回旧版 | 严重事故、数据损坏、合规风险 |

大多数问题优先 L1/L2。直接 L4 往往是过度反应。

### 10.16.5 回滚副作用

回滚不是零成本：

- 多轮会话可能上下文断裂。
- 新版本创建的异步任务可能需要按版本标签继续执行。
- 新版 memory schema 旧版本可能读不了。
- 新版本错误输出可能污染缓存。

因此回滚时要考虑会话保持、任务版本标签、memory 兼容和缓存清理。

### 10.16.6 回滚后的闭环

回滚完成后必须：

1. 封存触发回滚的 case。
2. 定位根因：模型、prompt、Skill 描述、threshold、后处理还是外部系统。
3. 加入回归集。
4. 缩小下次发布粒度。
5. 判断是否永久保留旧版本通道。

## 10.18 核心结论

Skill 不是简单的 API 包装，而是 Agent 认知外部世界、组织外部能力的标准化接口。

掌握 Tool -> Skill -> MCP 三层架构，就掌握了 Agent 开发的核心工程范式：

- 用 Function Calling 入门工具调用。
- 用 SKILL.md 规范沉淀可复用能力。
