# 模型升级回滚：不只是换回旧版本

模型升级回滚和传统服务回滚最大的差别是：问题往往不是“崩了”，而是“不一样了”。不崩的退化最难发现，也最难定义回滚标准。

回滚设计的核心，是让“发现 -> 决策 -> 执行”三步都有可操作依据。

## 1. 回滚的三个前置条件

没有下面三件事，回滚只能靠感觉。

### 1.1 版本即快照，而不是单一参数

一个可回滚的“模型版本”应包含：

- 模型标识：model id 和权重版本。
- Prompt 模板。
- 工具描述。
- Skill 列表和 Skill 描述。
- 关键参数：temperature、top_p、max_tokens、超时。
- 路由或分类器 threshold。
- 后处理规则：过滤、校验、格式化。

只换模型不换配套，属于伪回滚。线上很多“回滚无效”，根因是 prompt 或 Skill 描述已经跟着新模型改过，老模型使用新 prompt 反而效果更差。

### 1.2 流量可切分，切分可原子化

要求：

- 能按用户、租户、会话、Skill 维度独立切流。
- 切流不需要重启、不需要改代码。
- 配置中心推送后即时生效。
- 切流比例可细到 1%，且不同维度可叠加。

### 1.3 有新旧并跑的对照数据

回滚决策不能拍脑袋，需要数据：

- 核心指标：Skill 召回率、误触发率、任务完成率、用户终止率、延迟 P50/P95。
- 影子流量：新版本上线前，新旧同跑，但只采纳旧版输出。
- 用户反馈信号：点赞点踩、重新提问率、显式投诉。

## 2. 四档回滚策略

回滚不是只有“全量切回”一种动作。

| 档位 | 动作 | 适用场景 | 恢复时间 |
| --- | --- | --- | --- |
| L1 参数调节 | 调 threshold、改 prompt 小片段、关闭某 Skill | 单一 Skill 偶发问题 | 分钟级 |
| L2 局部切流 | 问题 Skill 或问题场景切回旧版 | 特定能力退化，其他正常 | 分钟级 |
| L3 灰度回退 | 新版本灰度比例从 X% 降到 0% | 整体质量下降但有指标支撑 | 小时级 |
| L4 全量回滚 | 所有流量切回旧版 | 严重事故、数据损坏、合规风险 | 小时级 |

优先从 L1/L2 处理。直接 L4 往往是过度反应，会放大用户感知。

## 3. 回滚触发机制

### 3.1 自动触发：硬红线

以下情况应立即自动回滚：

- 错误率突增超过 N 倍基线。
- P99 延迟翻倍。
- Skill 成功率低于基线 10 个百分点以上。
- 安全事件：prompt injection 拦截率下降、对外误发率上升。

自动触发必须有冷却机制：回滚后 T 分钟内不自动重新发布，避免反复横跳。

### 3.2 半自动触发：软红线

指标跌破但未到硬红线时：

- 触发告警。
- oncall 在 10 分钟内决策。
- runbook 里预置“如果 A 指标下降 B%，执行 C 动作”。

### 3.3 定性触发：体验问题

用户反馈集中出现时：

- 建立快速 case 收集通道。
- 每收到 N 条同类反馈触发复核。
- 建立 20-50 条关键场景金标 case。
- 任意关键 case 回归失败，立即 L2 回滚相关能力。

## 4. 回滚副作用控制

回滚不是零代价。

### 4.1 上下文断裂

用户进行中的多轮会话应保持当前版本，新会话走目标版本，避免同一会话里模型风格和能力突变。

### 4.2 异步任务

已经由新版本创建的定时任务或长时任务，旧版本不一定能处理。任务数据应带版本标签，执行时按标签路由。

### 4.3 Memory 兼容

新版本写入的 memory 结构，旧版本可能无法解析。需要保证 memory schema 前后兼容，或回滚时同步回退 memory 读取层。

### 4.4 缓存污染

新版本错误输出如果进入缓存，回滚后仍可能返回错误。回滚时要清理相关缓存 key。

## 5. 发布前必须准备的逃生通道

发布前就要准备好：

- 一键回滚按钮：配置中心有显眼入口，权限明确，不需要走发布流程。
- 回滚 runbook：写清每档回滚的操作步骤、负责人、预期恢复时间、验证清单。
- 对照评测集：上线前新旧版本跑同一套评测并归档。
- 通信模板：对内 oncall 群通告，对用户的话术预案。

## 6. 发布策略决定回滚难度

### 6.1 渐进灰度

推荐：

```text
1% -> 5% -> 20% -> 50% -> 100%
```

每级都要有观察窗口和明确推进/回退标准。

### 6.2 按维度灰度

- 按用户风险分层：低风险用户先，高风险企业关键账号最后。
- 按 Skill 分层：只读 Skill 先，写入或对外类 Skill 最后。
- 出问题时回滚半径更小。

### 6.3 影子发布

新版本接收真实流量，但输出不展示给用户。对比影子输出和线上输出差异，适合高风险 Skill 升级。

## 7. 回滚后必须做什么

回滚完成后不是结束，而是开始：

1. 封存问题样本：保存触发回滚的 case、时间、参数、输出。
2. 复盘根因：模型变化、prompt 不兼容、Skill 描述配套未更新，还是后处理问题。
3. 加入回归集：触发回滚的 case 必须进入发布前金标集。
4. 缩小下次发布粒度：例如从 20% 直接到 100% 改为 20% -> 50% -> 100%。
5. 评估是否永久保留旧版本通道：高风险场景可以双版本并存。

## 8. 反模式清单

- 只回滚模型，不回滚 prompt、Skill 描述、threshold。
- 全量发布没有灰度。
- 回滚必须走完整发布流程。
- 凭感觉决策，没有指标对照。
- 回滚后不复盘。
- 回滚、修复、再发、再回滚，反复横跳。

## 9. 心智模型

回滚设计的本质不是“如何退回”，而是让每次发布都可观测、可切分、可逆。

真正成熟的系统里，回滚应该是日常可用的档位，而不是按下去天就塌的核按钮。

设计顺序：

```text
版本即快照
  -> 流量可切分
  -> 指标可对照
  -> 触发有分级
  -> 执行有自动化
  -> 发布用灰度
```

最容易被忽略的是：把回滚入口建在日常工具里，让它足够便宜，便宜到可以频繁使用。
