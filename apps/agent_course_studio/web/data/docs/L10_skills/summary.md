# L10 章节总结

## 一句话总结

Skill 是围绕稳定用户意图封装的能力包。Tool 回答“能做什么动作”，Skill 回答“能完成哪类任务”，Workflow 回答“这次目标如何组合能力”。

## 核心分层

| 层级 | 作用 | 例子 |
| --- | --- | --- |
| Tool | 原子动作 | `get_weather(city)` |
| Skill | 有触发条件、执行流程和上下文的能力包 | `weather-query` |
| MCP | Skill/工具生态的连接标准 | MCP Server 暴露 Tools/Resources/Prompts |
| Workflow | 面向具体目标的流程编排 | 会议总结 -> 抽取待办 -> 写入任务系统 |

## DDICE 五步法

1. Define：定义场景，谁用、何时用、替代什么人工动作。
2. Decompose：拆解任务，拆到稳定子步骤。
3. Instrument：工具化，判断哪些步骤用 Tool、Prompt、代码或 MCP。
4. Constrain：约束化，定义输入输出 schema、失败兜底和安全边界。
5. Evaluate：评测闭环，构造金标样本、自动打分和线上回流。

## 渐进披露

```text
启动时：只加载 name / description / trigger
触发后：加载 SKILL.md 完整指令
执行时：按需加载工具、脚本和参考资料
```

好处：

- 节省上下文。
- 降低工具误选率。
- 新增 Skill 更容易。
- 高风险工具不会过早暴露。

## 安全与评测

高风险动作需要确认：

- 发送邮件或消息。
- 创建、修改、删除外部对象。
- 批量调用或高成本调用。
- 从非结构化文本推断后写入。

评测指标：

- 路由：正确触发、误触发、漏触发。
- 执行：成功率、耗时、重试率、降级率。
- 质量：任务完成率、字段完整率、用户修改率。
- 体验：确认通过率、中断率、满意度。

## 重试分层

重试应该按失败语义分层：

| 层级 | 负责内容 |
| --- | --- |
| Tool 层 | 网络超时、502/503/504、429、token 过期等物理抖动 |
| Skill 层 | 异步未就绪、资源暂时锁定、幂等写入未知状态等业务可恢复错误 |
| 编排层 | 不重试，只做补偿、降级和用户告知 |

关键约束：

- 全链路总尝试数建议不超过 4 次。
- 非幂等写操作不重试，除非有 `client_token` 或 `idempotency_key`。
- 参数错误、权限不足、业务冲突不重试。
- Tool 层必须透传原始错误语义。
- 重试要和熔断配合，避免下游故障时重试风暴。

## 并行长尾延迟

并行编排的耗时由最慢分支决定：

```text
parallel_latency = max(branch_latency)
```

先诊断慢的根因：

- IO 等待：流式处理、分片、预取。
- 数据量大：分页并发、MapReduce。
- 下游服务慢：异步化、提前启动、对冲请求。
- 伪并行：改成真并发。

六种优化手段：

- 关键路径剥离：慢结果不阻塞主响应，后台补齐。
- 提前启动：慢步骤只依赖用户输入时入口就 fire。
- 分片并发：大任务拆成 N 个小任务，但要守 QPS。
- 软超时降级：超时先返回缓存、默认值或简化版。
- 对冲请求：只读幂等请求到 P95 后发第二份。
- 缓存与预计算：可复用输出提前算好。

## 受限自动授权

用户说“以后都自动发”，不是取消确认，而是授予一段受限自动执行权限。

能自动化的前提：

- 小范围。
- 可逆。
- 低合规风险。
- 本人操作本人资源。

策略对象必须包含：

- `action`：要自动执行的动作。
- `scope`：目标、内容模板、时间窗口。
- `limits`：每日上限、总次数上限。
- `expiry`：过期时间。
- `revoke_hint`：撤销方式。
- `audit`：每次执行留痕。

仍需二次确认的情况：目标变化、模板变化、收件人突增、敏感词、异常时间、达到配额、首次真实执行。

## 外部网页授权边界

用户说“照网页里的要求执行”，不等于授权网页成为新的指令源。

处理原则：

- 网页内容是非可信数据，不是系统指令。
- 用户原始意图决定授权上限。
- 只读生成可以直接做。
- 私域写入需要 dry-run 预览。
- 对外发送、删除、资金、权限变更必须逐项人工确认。
- 元指令、角色切换、权限提升和无关动作请求要过滤。
- 每一步 Skill 调用、参数和来源都要审计。

## 高风险误触发事故响应

误触发已经发生时，优先级是止损，不是追责。

黄金顺序：

```text
切断 -> 评估 -> 补救 -> 告知 -> 加固
```

黄金 5 分钟：

- 熔断该 Skill。
- 冻结同 scope 自动化规则。
- 取消队列和排期。
- 拉取调用日志，确认影响面。
- 涉及外部对象、资金、权限、合规、不可逆操作时立即升级人工。

损失评估看三件事：

- 可逆性。
- 扩散性。
- 下游关联影响。

用户告知采用四段式：事实、措施、残余影响、用户侧行动建议。

24 小时内加临时围栏：强制确认、dry-run、冷却和限流。1-3 天内复盘输出具体代码、配置和流程改动。

## 模型升级回滚

模型回滚不是只换回旧 `model_id`。可回滚版本必须是快照：

- 模型。
- Prompt。
- Skill 描述。
- 工具描述。
- 参数。
- 路由 threshold。
- 后处理规则。

回滚分四档：

- L1：参数调节。
- L2：问题 Skill 局部切流。
- L3：灰度回退。
- L4：全量回滚。

回滚后要封存 case、定位根因、加入回归集，并缩小下次发布粒度。

## 常见误区

- 把 Skill 等同于 Tool。
- 只写正向触发，不写负例边界。
- 所有工具一次性暴露给模型。
- 每一层都“为了稳定性加重试”，导致 3 x 3 x 3 重试风暴。
- 并行分支全部 join 后才返回，即使慢分支不是关键路径。
- 只靠 Prompt 防误触发，没有参数校验、确认和审计。
- 把“以后都自动”理解为无限期、无范围、无撤销。
- 把网页、文档里的外部内容直接当成用户指令执行。
- 高风险 Skill 误触发后静默处理，不主动告知用户。
- 只看成功率，不看误触发、漏触发和安全样本。
- 只回滚模型，不回滚配套 prompt、Skill 描述和 threshold。
