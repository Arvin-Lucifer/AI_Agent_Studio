# L10 面试题库：Agent Skill 设计与治理

说明：本题库基于老师提供的 “Agent 开发面试 - Skill” 材料整理，并补充了课程代码中的实战表达。补充内容用于帮助回答更完整，不替代原材料观点。

## 一、基础认知题

### 1. 什么是 Agent 中的 Skill？它和 Tool、Workflow 有什么区别？

Skill 是面向某类稳定用户意图的能力封装。它通常不只是一次工具调用，而是围绕任务目标组织的一组输入约束、执行流程、错误处理、结果结构和使用规则。

- Tool 回答“我能做什么动作”，例如读文件、查天气、写数据库。
- Skill 回答“我能完成哪类任务”，例如天气出行建议、代码审查、会议纪要总结。
- Workflow 回答“为了完成这次目标，如何组合能力”，可能编排多个 Skill 和 Tool。

### 2. 为什么 Agent 系统需要 Skill，而不是把所有能力都直接暴露成 Tool？

原因：

- 直接暴露大量 Tool 会扩大 action space，增加误选率。
- 重复流程逻辑可以封装到 Skill 中，例如校验、重试、确认和降级。
- Skill 更利于治理和评测，可以统计某类场景的成功率和失败原因。
- Skill 能提供安全边界，对高风险动作增加确认、白名单和审计。
- Skill 解耦用户意图和底层实现，模型只需做意图理解和技能选择。

### 3. 一个好的 Skill 应具备哪些基本属性？

至少包括：

- 目标边界清晰。
- 输入输出契约明确。
- 执行过程可控，支持校验、重试、降级和确认。
- 结果可解释，失败时说明失败点和下一步建议。
- 可观测、可评测、可迭代。
- 高风险动作有安全机制。

## 二、设计题

### 4. 设计一个“会议纪要总结并同步待办”的 Skill，给出输入输出定义。

输入建议：

- `meeting_source`：会议文本、链接或妙记来源。
- `summary_style`：高管版、执行版、技术版。
- `extract_todos`：是否抽取待办。
- `todo_sink`：写入飞书、Jira 或仅输出。
- `participants`：辅助 owner 归属。
- `language`：输出语言。
- `confirm_before_write`：写入前是否确认。

输出建议：

- `summary`
- `key_decisions`
- `risks`
- `action_items[]`，包含 `task_desc/owner/due_date/confidence`
- `write_result`，包含 `created/skipped/failed/reason`

关键点：识别待办和写入任务系统要分阶段；写入属于副作用操作，最好支持 dry-run 或 preview。

### 5. Skill 粒度应该怎么定？过大和过小有什么问题？

Skill 粒度应围绕稳定用户意图定义，而不是围绕单个后端 API。

过小的问题：

- Skill 数量膨胀。
- 模型难以选择。
- 流程逻辑散落。

过大的问题：

- 参数过多。
- 内部逻辑重。
- 失败点难定位。
- 一个 Skill 覆盖多个意图，路由准确率下降。

原则：一个 Skill 应能闭环完成一类高频、边界稳定、输入输出可标准化的任务。

### 6. Skill 描述文案怎么写，才能让模型更容易正确调用？

建议包含：

- 能力说明。
- 触发示例。
- 负例边界。
- 输入前提。
- 副作用说明。

高风险 Skill 要明确“发送、删除、批量修改前需要确认”。

## 三、调用与编排题

### 7. 用户说“帮我整理这周会议结论并创建待办”，设计成一个 Skill，还是多个 Skill 编排？

取决于稳定性。如果这是高频稳定复合意图，输入来源、待办系统和确认流程都稳定，可以封装成高阶 Skill。如果来源多样、目标系统不统一、团队规则差异很大，更适合多个 Skill 编排。

判断标准：

- 是否高频复用。
- 输入输出是否稳定。
- 是否需要独立失败恢复。
- 是否存在多系统分支。
- 是否需要多个确认点。

### 8. Skill 编排中，串行和并行如何决策？

核心看数据依赖和失败影响范围。

- 独立读取类任务适合并行，例如并行拉取多份会议材料。
- 后一步依赖前一步结果时必须串行，例如先抽取 owner 再创建任务。
- 还要考虑限流、调用成本、幂等性和失败合并策略。

追问：如果多个子步骤都能并行，但其中一个慢得多，如何优化？

参考回答：

并行编排的总耗时由最慢分支决定，所以要先用 trace 定位慢分支根因：IO 等待、数据量大、下游服务慢，还是内部伪并行。

优化分两类：

- 让慢分支变快。
- 让慢分支不阻塞主响应。

常用手段：

- 关键路径剥离：如果慢分支不是主响应必需，先返回快结果，慢结果标记 pending 后补。
- 提前启动：慢步骤只依赖用户输入时，在编排入口就启动，不等前置校验和快步骤结束。
- 分片并发：数据量大时拆成多个 shard 并行处理，但要控制 `分片数 x QPS <= 下游配额`。
- 软超时降级：超过软超时先返回缓存、默认值或简化版，真实结果异步覆盖。
- 对冲请求：只读幂等请求到 P95 仍未返回时发第二份请求，取先返回结果。
- 缓存和预计算：可复用输出按参数指纹缓存，周期性任务提前算好。

优秀回答会强调：必须先有 trace、关键路径标记、限流背压和后补一致性设计。

### 9. Skill 内部依赖多个 Tool，其中一个步骤失败了怎么办？

先分类失败：

- 临时错误：有限重试、指数退避。
- 可降级错误：返回部分结果并说明缺失。
- 需要用户介入：中止副作用操作，请求补充信息或授权。
- 致命错误：终止，不返回伪造结果。

返回值应结构化，例如 `success/partial_success/failed_step/retryable/user_action_required`。

### 10. 重试策略应该放在 Tool 层还是 Skill 层？

两层都要，但职责不同。不能在每一层都随手加 3 次重试，否则会出现 `3 x 3 x 3 = 27` 次调用的重试风暴。

Tool 层负责物理层抖动：

- 网络超时、连接重置、DNS 抖动。
- HTTP 502/503/504。
- 429 限流，但必须尊重 `Retry-After`。
- Token 过期，刷新后重试一次。

Tool 层策略：指数退避 + jitter，最多 2-3 次，总耗时有上限，并且必须把 status code、错误类型、`Retry-After` 等原始语义透传给 Skill 层。

Skill 层负责业务语义可恢复错误：

- 文档暂时锁定。
- 异步任务未就绪。
- 幂等写入超时后，先用 `client_token` 查询是否已成功，再决定是否重试一次。

Skill 层策略：最多 1 次业务重试，或带总时长上限的轮询；写操作必须具备 `idempotency_key`。

不重试的情况：

- 400、401、403、404、422。
- 参数校验失败、权限不足。
- 业务规则冲突。
- 非幂等写操作的未知状态。

编排层原则上不重试，只根据 Skill 返回的 `retriable` 做补偿、降级或告知用户。优秀回答会强调 retry budget、幂等性、错误码透传和熔断。

## 四、安全与鲁棒性题

### 11. 哪些 Skill 动作必须做人机确认？

通常包括：

- 对外发送类：邮件、群消息、批量通知。
- 写入修改类：创建任务、修改文档、批量更新。
- 删除不可逆类。
- 高成本批量调用。
- 高不确定推断后的写入，例如从会议文本推断 owner 后直接创建任务。

### 12. 如何防止模型误触发高风险 Skill？

从四层防护：

- 触发层：完善描述、加入负例、提高高风险意图阈值。
- 参数层：强校验参数完整性、枚举值、格式和对象存在性。
- 策略层：副作用操作确认、dry-run、权限、配额、白名单。
- 执行层：幂等键、审计日志、可撤销机制、沙箱验证。

不能只依赖 Prompt。

### 13. 用户说“以后都自动发”，如何设计授权边界？

不能把它理解成取消确认，而是要把一次口头授权落成受限自动执行策略。

先判断能不能接：

- 小范围、私人或固定小群可以考虑。
- 群发、对外、跨部门广播要降级为单次确认。
- 打款、删除数据、合同、权限变更、审批流应拒绝口头永久授权。
- 代他人操作、涉及他人隐私也不能直接自动。

能接时要落成持久化规则对象，而不是只存在对话 memory 里：

- `action`：具体动作。
- `scope`：目标、内容模板、时间窗口。
- `limits`：每天最多几次、总共最多几次。
- `expiry`：过期时间，例如 30 天或 90 天。
- `revoke_hint`：撤销方式。
- `audit`：每次执行留痕。

执行前每次都要校验：

- 规则未过期、未撤销。
- 本次目标、模板、时间在 scope 内。
- 没有收件人突增、敏感词、异常设备、异常时间等风险信号。
- 未超过频率和总次数上限。

首次真实执行仍建议 dry-run 预览和确认。用户回复不能只说“好的”，要复述授权范围、说明时效和撤销方式。

优秀回答会强调：用户要的是“别每次都烦我”，不是“从此失控”。

### 14. 高风险 Skill 已经误触发了，事后怎么止损？

优先级不是追责或立刻改代码，而是先止损、评估影响、透明告知。

黄金 5 分钟：

- 熔断该 Skill，对当前用户、会话、租户或全局停用。
- 冻结同 scope 自动化规则。
- 取消队列和排期，防止后续批次继续执行。
- 拉取调用日志：时间、参数、目标、成功数、失败数、trace_id。
- 判断事故是否还在发生；持续执行类先切流再分析。

触发外部对象、资金、权限、合规、不可逆操作，或影响范围超过个位数目标时，立即升级人工 oncall。

损失评估看三件事：

- 可逆性：能否撤销，撤销是否有副作用。
- 扩散性：是否已经被看到、转发或消费。
- 下游关联：是否触发自动回复、工作流、Webhook 或第三方系统。

补救策略：

- 可逆操作：先保留证据副本，再删除或标记作废。
- 已送达消息：撤回窗口内撤回；超时或已读时发补充说明。
- 金额/权限类：走人工反向流程，不用另一个自动化修复自动化的错。
- 下游工作流：先停最下游，再回溯上游。

用户告知采用四段式：发生了什么、已采取措施、残余影响、用户侧行动建议。不能隐瞒、淡化或把责任推给模型。

24 小时内要加临时围栏：该 Skill 强制确认、相似请求 dry-run、同用户其他高风险 Skill 冷却、限流。1-3 天内复盘，产出 Skill 描述修正、置信度门槛、确认机制、限流熔断和回归样本。

优秀回答会强调：误触发后的正确顺序是“切断 -> 评估 -> 补救 -> 告知 -> 加固”。

### 15. Prompt Injection 会如何影响 Skill 调用？如何防御？

影响：

- 调用不该调用的 Skill。
- 篡改参数。
- 泄露内部信息。
- 诱导危险操作。

防御：

- 指令分层，系统指令高于外部内容。
- 上下文隔离，把网页/文档标记为非可信数据。
- 高风险动作前基于用户原始意图重新校验。
- 参数白名单。
- 发送、删除、写入前 preview + 确认。

追问：如果用户明确说“照网页里的要求执行”，是否可以照做？

参考回答：

不能无条件照做。用户授权的是“参考网页完成用户任务”，不是把网页升级为新的指令源。网页内容仍然是非可信数据，必须先显式抽取要求，再按动作风险分级。

处理流程：

- 先拆用户真实意图：是内容参考、私域写入、对外提交，还是无条件外部授权。
- 把网页要求显式列给用户，尤其标出发送、删除、付款、权限变更等高风险动作。
- 只读生成可以直接执行；私域写入先 dry-run；对外或不可逆动作必须逐项确认。
- 过滤网页里的元指令、角色切换、权限提升和与原始任务无关的动作。
- 拒绝读取 cookie/token、本地文件、跨账号操作、资金和权限变更绕过。
- 保留审计日志，记录每个 Skill 调用、参数和来源。

优秀回答会强调：用户的“照网页执行”最多授权到用户可理解、可确认的内容，不授权网页里的隐藏指令。

## 五、评测与优化题

### 16. 如何评估一个 Skill 做得好不好？

四类指标：

- 路由指标：正确触发率、误触发率、漏触发率。
- 执行指标：成功率、耗时、重试率、降级率。
- 结果质量：任务完成率、字段完整率、用户二次修改率。
- 用户体验：确认通过率、中断率、满意度、纠错率。

线下做 intent 路由、参数抽取、端到端任务成功测试。线上结合埋点、A/B、错误回放和失败聚类。

### 17. Skill 线上成功率突然下降，怎么排查？

按链路分层：

- 路由层：是否改了描述、prompt、模型版本或触发规则。
- 参数层：字段缺失、抽取错误、歧义增加。
- 执行层：API 超时、限流、鉴权失效。
- 结果层：后处理或 schema 兼容性变化。
- 数据层：流量结构变化、长尾 case 增加。

手段：分步骤日志、错误码分布、失败样本聚类、case 回放、变更前后对比。

追问：如果是模型升级导致问题，如何设计回滚？

参考回答：

模型升级回滚不是简单换回旧模型。一个可回滚版本必须是完整快照，包含 model id、prompt 模板、工具描述、Skill 列表和描述、temperature、top_p、max_tokens、路由 threshold、后处理规则等。只换模型不换配套配置，通常是伪回滚。

回滚前要具备三件事：

- 版本即快照。
- 流量可按用户、租户、会话、Skill 维度切分。
- 有新旧并跑或 shadow mode 对照数据。

回滚动作应分级：

- L1 参数调节：调 threshold、改 prompt 小片段、关闭某 Skill。
- L2 局部切流：问题 Skill 或问题场景切回旧版。
- L3 灰度回退：新版本灰度比例降到 0%。
- L4 全量回滚：严重事故、数据损坏或合规风险时所有流量切回旧版。

还要控制回滚副作用：多轮会话不要中途切模型，异步任务带版本标签，memory schema 保持兼容，回滚时清理新版本错误缓存。回滚后要封存 case、定位根因、加入回归集，并缩小下次发布粒度。

优秀回答会强调：回滚设计的本质是让发布可观测、可切分、可逆，而不是出事后手动换模型。

### 18. 如何构造 Skill 评测集？

至少覆盖：

- 标准场景。
- 边界场景：参数缺失、模糊表达、多意图、超长输入。
- 负例场景：不该触发。
- 异常场景：权限不足、对象不存在、接口超时、脏数据。
- 安全场景：prompt injection、越权对象、未确认写入。

数据来源：线上 query 脱敏采样、人工构造难例、失败案例回灌、对抗样本生成。

## 六、判分提示

好答案：

- 先定义边界，再谈实现。
- 会明确 trade-off。
- 能把模型问题转化为工程机制问题。
- 经常提到确认、校验、幂等、可观测和评测闭环。

风险信号：

- 把 Skill 等同于 Tool。
- 过于依赖“加 Prompt”。
- 缺少副作用、安全和审计意识。
- 只看成功率，不谈误触发和漏触发。
