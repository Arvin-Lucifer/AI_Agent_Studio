# “以后都自动发”：授权边界与落地

用户说“以后都自动发”时，不是在取消确认，而是在授予一个受限的自动执行权限。

正确处理方式是把这句口头授权翻译成：

```text
窄范围 + 有时效 + 可撤销 + 可审计
```

的策略对象，而不是从此无脑执行。

## 1. 先判断：这句话能不能接

不是所有“自动发”都能自动化。先过三道闸。

| 维度 | 可以自动化 | 需降级或拒绝 |
| --- | --- | --- |
| 影响范围 | 私人、小范围，例如自己日历、个人任务、固定小群 | 群发、对外、跨部门广播 |
| 可逆性 | 可撤回、可删除、可修改 | 不可逆，例如打款、合同、删除数据 |
| 合规风险 | 内部常规操作 | 财务、权限变更、审批流 |
| 用户身份 | 本人操作本人资源 | 代他人操作、涉及他人隐私 |

右侧任意一项命中，就不接受“永久自动”。要么保留单次确认，要么走审批，要么直接拒绝。

## 2. 把口头授权落成结构化策略

一句话授权必须转成规则对象，否则下一次执行时边界会变成争议。

示例：

```json
{
  "action": "send_im_notification",
  "scope": {
    "target": "group_id=project-daily",
    "content_template": "日报模板",
    "time_window": "workday 09:00-10:00"
  },
  "limits": {
    "max_per_day": 1,
    "max_total": 30
  },
  "expiry": "2026-09-16",
  "revoke_hint": "说“停止自动发日报”可随时撤销",
  "audit": true
}
```

关键设计：

- 范围必须窄：不是“以后发消息都自动”，而是“以后这类消息到这个目标自动”。
- 默认有期限：例如 90 天或次数上限，到期重新确认，避免祖传授权。
- 保留撤销路径：明确告诉用户怎么关闭。
- 每次执行留痕：用户随时能查“你替我自动做了什么”。

## 3. 回复用户的标准话术

不要只说“好的，以后自动发”。要复述边界、时效和撤销方式。

推荐：

```text
好的，已记住：之后工作日早上给 XX 群发日报时自动执行，不再每次确认。
授权范围仅限此场景，30 天后或发满 20 次会再次跟你确认一次。
想随时关闭，说“停止自动发日报”即可。
```

这段话做了四件事：

1. 复述：防止用户以为授权更宽。
2. 限定：明确范围，避免越权。
3. 时效：主动给出过期条件。
4. 兜底：保留撤销入口。

## 4. 仍需二次确认的豁免场景

即使用户已经授权“以后自动”，下面情况仍然要拦一次：

1. 超出授权范围：内容模板、目标、时间窗口变化。
2. 风险信号：收件人数量突增、内容包含敏感词、金额超阈值。
3. 异常环境：异地登录、非常用设备、凌晨等反常时间。
4. 达到配额：接近次数或频率上限。
5. 首次执行：第一次真实执行应做 dry-run 预览和确认，避免规则理解偏差。

拦截话术：

```text
你之前授权了自动发日报，但这次收件人从 1 个群变成了 5 个，超出授权范围，需要你再确认一次。
```

## 5. 技术落地关键点

### 5.1 授权存储

- 写入持久化的用户偏好表或规则表。
- 不把授权只塞在对话 memory 里。
- memory 可以提醒“存在这条规则”，但不能作为执行依据。

### 5.2 执行前校验

每次自动执行前都跑 checklist：

- 规则是否仍有效：未过期、未撤销。
- 本次参数是否在 scope 内。
- 是否触发豁免条件。
- 是否超过频率或总次数限制。

### 5.3 审计日志

每次自动执行必须记录：

```json
{
  "time": "2026-06-18T09:05:00",
  "rule_id": "auto-send-daily-project",
  "params": {
    "target": "group_id=project-daily",
    "content_template": "日报模板"
  },
  "result": "sent_mock"
}
```

### 5.4 撤销机制

- 支持自然语言撤销，例如“停止自动发日报”。
- 支持查看列表，例如“显示所有自动化规则”。
- 撤销立即生效，不等下次调用。

### 5.5 降级路径

自动执行失败时，不要无限重试。一次失败就回退为“通知用户 + 待确认”，避免失败状态下错误放大。

## 6. 反模式清单

- 把“以后都自动”理解为无限期、无范围、无撤销。
- 授权信息只存对话上下文，换个会话就失效或越权。
- 不告诉用户撤销方式。
- 不做执行前范围校验，把“发日报”的授权用来“发季报”。
- 对打款、删数据、外发、权限变更接受口头永久授权。

## 7. 面试表达

如果面试官问：“用户说以后都自动发，你怎么设计？”

可以这样回答：

> 我不会把它理解成取消确认，而是把它落成受限自动执行策略。先判断能不能接：小范围、可逆、低合规风险、本人资源才可自动化；群发、对外、跨部门、打款、删除、权限和审批流要保留单次确认或走审批。能接的授权要落成持久化规则对象，包含 action、scope、limits、expiry、revoke_hint 和 audit。每次执行前校验未过期、未撤销、参数在 scope 内、没有异常风险信号。首次执行仍做 dry-run 预览。用户侧回复要复述范围、说明时效和撤销方式。自动执行失败时降级为通知用户待确认。

核心句：

```text
“以后都自动发”是用一次确认换取一段受限期内的免确认，不是从此失控。
```
