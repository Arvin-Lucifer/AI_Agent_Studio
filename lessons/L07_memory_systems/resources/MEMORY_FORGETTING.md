# 记忆遗忘机制

长期记忆必须支持遗忘。遗忘不是缺陷，而是主动的记忆治理。

## 为什么要遗忘

1. 存储和索引成本会持续增长。
2. 低质量旧记忆会污染检索结果。
3. 用户偏好和事实会变化。
4. 隐私与合规要求用户可删除个人数据。

## 八类策略

| 策略 | 一句话 | 适用场景 |
| --- | --- | --- |
| TTL | 超过时间自动过期 | 临时事实、新闻、短期任务 |
| 滑动窗口 | 只保留最近 N 条 | 短期上下文 |
| 衰减函数 | 记忆强度随时间下降，命中后增强 | 长期偏好、人格化记忆 |
| LRU/LFU | 淘汰最近最少使用或最少访问 | 容量有限场景 |
| 重要性打分 | 低重要性优先遗忘 | 长期事实流 |
| 摘要压缩 | 多条折叠成摘要 | 长对话和长期任务 |
| 去重/冲突合并 | 重复合并，新事实覆盖旧事实 | 偏好变化、画像更新 |
| 显式删除 | 用户要求删除时执行 | 隐私合规 |

## 按记忆类型匹配

| 记忆类型 | 推荐策略 | 原因 |
| --- | --- | --- |
| 会话短期上下文 | 滑动窗口 + 摘要 | token 有限 |
| 用户画像 | upsert 覆盖 + 软删除 | 最新真值优先 |
| 偏好 | 衰减 + 一致性合并 | 偏好会演变 |
| 事实流 | 重要性打分 + LRU + TTL | 数量大、质量不均 |
| 知识图谱三元组 | 一致性合并 + 显式删除 | 关系需精确 |
| 隐私字段 | 显式删除 + 强制 TTL | 合规优先 |

## 小代价落地方案

给每条记忆增加 metadata：

```json
{
  "created_at": "...",
  "last_access": "...",
  "access_count": 3,
  "importance": 7,
  "expire_at": "..."
}
```

检索后更新 `last_access` 和 `access_count`。定期清理过期记忆。容量超限时按低重要性、低访问频次和低新鲜度淘汰。

## 合规提醒

软删除便于审计，但不等于真正删除。涉及隐私法规时，还要考虑向量索引、缓存、日志、备份和下游副本。
