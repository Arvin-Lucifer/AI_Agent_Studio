# L11 面试题速查：Agent 评测和部署

## 来源说明

- 题目来源：基于老师提供的 Agent 评估与调优材料、部署材料，以及本章配套代码整理。
- 整理方式：不是逐字摘抄，而是把课堂内容整理成面试可表达的问答。
- 补充边界：CI 门禁、日志字段、SSE 事件设计、Nginx 细节和成本监控属于工程化补充。

## 一、评估与调优

### 1. 为什么 Agent 需要系统化评估？

因为 Agent 的失败往往不是程序崩溃，而是回答错误、工具误调、幻觉、安全越界或成本失控。只手工问几个问题无法覆盖边界场景，也无法判断改 prompt 或换模型后是否整体退化。

### 2. Agent 评估有哪些核心维度？

准确性、完整性、相关性、安全性、效率/成本、鲁棒性。生产系统还会进一步拆出引用正确率、工具选择准确率、参数准确率、拒答正确率和用户满意度。

### 3. 如何设计 Agent 评测集？

每条 case 至少包含问题、类别、期望工具、期望关键词、不应出现的内容。评测集要覆盖正常事实问答、闲聊、边界输入、Prompt Injection、无答案拒答和多子问题完整性。

### 4. 为什么只做关键词匹配不够？

关键词匹配会有假阳性和假阴性。比如“不是 5 天”仍命中“5天”；“5 个工作日”语义正确但可能没命中。更成熟的方法是结合语义相似度、LLM-as-Judge、引用校验和人工抽检。

### 5. 为什么要评估工具调用？

Agent 的过程会影响成本、安全和答案可信度。最终答案看起来正确，不代表工具选择正确。评估时要看是否该调工具、是否调对工具、参数是否正确、是否过度调用和失败后是否正确降级。

### 6. 安全检查如何计分？

严重安全问题应作为 hard gate。泄露 system prompt、越权调用工具、对外发送未确认信息等问题不能被平均分掩盖。可以单独记录安全通过率，并在安全失败时阻断发布。

### 7. Agent 常见翻车场景有哪些？

幻觉、死循环、工具误调、格式错乱、Prompt 注入、成本失控和延迟过高。每类问题对应不同修复方式，不能一律靠“加一句 prompt”解决。

### 8. 如何修复幻觉？

要求只基于检索结果回答，无证据时明确拒答；答案中注明来源；对关键事实做引用校验；把无答案和证据不足样例加入评测集。

### 9. 如何避免工具调用死循环？

设置最大步数，检测重复工具和重复参数，要求每次 observation 推动状态变化。连续失败时不要无限重试，应降级、澄清或中止。

### 10. 如何做调优闭环？

先跑 baseline，定位失败类型，做最小修复，再跑同一套评测集。修复后的 bad case 要进入回归集，防止下一次改动把旧问题带回来。

## 二、部署与服务化

### 11. 为什么需要把 Agent 部署成 API？

本地脚本只能开发者自己用。真实应用需要多用户访问、网页或 App 调用、健康检查、监控日志、成本统计、权限控制和 7x24 稳定运行。

### 12. FastAPI 封装 Agent 时要注意什么？

请求和响应要结构化；`session_id` 用来隔离会话；`/health` 必须轻量；接口要限制输入长度；异常要返回可理解错误；不要把密钥或内部堆栈暴露给用户。

### 13. `/health` 为什么不能调用模型？

健康检查会被负载均衡器频繁调用。如果它调用模型，会增加成本、延迟和外部依赖。模型服务抖动时也可能让健康检查误判，导致实例被错误摘除。

### 14. 流式输出怎么实现？

常见做法是 SSE 或 WebSocket。SSE 更简单，适合服务端单向推送 token、工具调用事件和完成事件。LangChain/LangGraph 可以通过 `astream_events` 获取模型和工具事件。

### 15. Nginx 代理 SSE 要注意什么？

必须关闭 `proxy_buffering` 和 `proxy_cache`，并设置较长的 `proxy_read_timeout`。否则后端逐块输出，浏览器仍可能等缓冲后才看到内容。

### 16. Gunicorn 和 Uvicorn 分别做什么？

Uvicorn 是 ASGI server，负责运行 FastAPI。Gunicorn 负责多 worker 进程管理、重启和进程监督。常见组合是 `gunicorn app:app --worker-class uvicorn.workers.UvicornWorker`。

### 17. 部署后如何监控成本？

记录请求数、输入 token、输出 token、模型、工具调用次数、延迟和估算成本。发现成本上升时，要看是否上下文过长、工具循环、RAG top_k 过大或简单请求走了复杂链路。

### 18. 部署后如何做 Prompt 注入防御？

输入侧检测注入模式和长度；外部内容当数据；工具层做权限和参数校验；高风险动作确认；输出侧避免泄露 system prompt、密钥、cookie 和内部配置。

### 19. Agent API 上线前检查哪些内容？

健康检查、限流、超时、日志、错误处理、密钥管理、安全评测、成本监控、回滚方案、Nginx 配置、SSE 是否可用、评测集是否通过。

### 20. 如何把评测接入 CI/CD？

发布前跑固定评测集。平均分低于阈值、安全 case 失败、关键场景失败或延迟/成本超过阈值时阻断发布。评测结果保存为 JSON，便于对比和复盘。
