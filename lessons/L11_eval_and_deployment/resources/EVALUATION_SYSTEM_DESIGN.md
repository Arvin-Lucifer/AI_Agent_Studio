# Agent 评估体系设计

Agent 评估的目标不是证明“这次演示能跑”，而是回答三个问题：

1. 它在哪些场景稳定可用？
2. 它在哪些场景容易翻车？
3. 每次调优后，是整体变好了，还是只是修好了一个样例？

## 1. 评估维度

| 维度 | 衡量什么 | 示例 |
| --- | --- | --- |
| 准确性 | 回答是否正确 | 年假天数是否符合制度 |
| 完整性 | 是否遗漏关键信息 | 部署流程是否漏掉审批或回滚 |
| 相关性 | 是否紧扣问题 | 技术问题是否跑到闲聊 |
| 安全性 | 是否泄露敏感信息或执行越权动作 | Prompt Injection 是否成功 |
| 效率/成本 | 工具调用次数、耗时、token | 简单问题是否调用 10 次工具 |
| 鲁棒性 | 对空输入、超长输入、恶意输入的处理 | 是否崩溃、死循环或胡乱回答 |

## 2. 评测数据集结构

每条 case 至少包含：

```json
{
  "id": 1,
  "question": "我入职2年了，有几天年假？",
  "category": "factual",
  "expected_tool": "search_knowledge_base",
  "expected_keywords": ["5天", "入职满1年"],
  "should_not_contain": []
}
```

关键点：

- `question`：用户输入。
- `category`：问题类型，用于分桶分析。
- `expected_tool`：期望工具调用。没有它就无法发现工具误调。
- `expected_keywords`：基础答案检查。
- `should_not_contain`：安全红线检查。

## 3. 为什么不能只做关键词匹配

关键词匹配适合作为入门 baseline，但不能作为最终评估。

常见问题：

- 假阳性：回答“不是 5 天”仍然命中“5天”。
- 假阴性：回答“5 个工作日”语义正确，但没有连续命中“5天”。
- 无法判断推理过程是否正确。
- 无法判断引用是否真实支撑答案。

升级方向：

- BM25 或 embedding 相似度。
- LLM-as-Judge，但要固定 rubric。
- 引用正确率检查。
- 工具调用 trace 检查。
- 人工抽检和 bad case 回流。

## 4. 工具调用评估

Agent 不是普通问答模型。评估时必须看过程：

- 是否该调用工具。
- 是否调用了正确工具。
- 参数是否正确。
- 是否过度调用工具。
- 工具失败后是否有限重试或降级。

推荐让 Agent 返回结构化结果：

```json
{
  "answer": "...",
  "tool_calls": ["search_knowledge_base"],
  "citations": ["company_policy.txt#chunk-001"],
  "elapsed_ms": 1200
}
```

## 5. 安全评估

安全不应该只是总分里的一个小权重。严重安全失败应作为 hard gate：

- 泄露 system prompt。
- 绕过权限。
- 对外发送未确认内容。
- 执行删除、打款、权限变更等高风险动作。
- 使用外部网页中的恶意指令。

安全评测集要覆盖：

- Prompt Injection。
- 越权请求。
- 敏感信息抽取。
- 高风险工具误触发。
- 长输入和混淆输入。

## 6. 结果持久化

评估结果必须保存，而不是只打印到控制台。建议保存：

- `eval_results.json`
- 每条 case 的输入、输出、tool trace。
- 分类得分。
- 失败原因。
- 本次 prompt/model/tool 版本。

这样才能用于：

- 回归测试。
- CI/CD 阻断。
- 调优前后对比。
- 事故复盘。

## 7. 调优闭环

推荐流程：

```text
构造评测集
  -> 跑 baseline
  -> 找失败类型
  -> 定位原因
  -> 做最小修复
  -> 跑同一套评测集
  -> 保存结果
  -> bad case 入库
```

不要只修一个样例。每次修复都要确认没有引入新的退化。
