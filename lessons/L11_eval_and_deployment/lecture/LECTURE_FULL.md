# 第11讲：Agent 评测和部署 —— 从“能跑”到“好用”

本章由两条主线组成：第一条是 Agent 评估与调优，解决“怎么证明它真的好用”；第二条是服务化部署，解决“怎么让别人稳定使用它”。

## 11.1 为什么需要评估

搭了一个 Agent，问几个问题，效果还不错，然后直接上线，是很危险的。

没有系统化评估，就不知道 Agent 在什么情况下会翻车。生产环境中的翻车代价可能很高：

- 回答错误，导致用户做出错误决策。
- 泄露系统提示词或敏感信息。
- 调错工具，触发不该触发的动作。
- 进入工具调用死循环，造成延迟和成本失控。
- 面对恶意输入时绕过安全边界。

评估的目的不是追求一个漂亮分数，而是建立一套可重复的“考试机制”：每次改 prompt、换模型、加工具、改 RAG 策略，都能用同一批问题判断是否真的变好。

## 11.2 Agent 的评估维度

Agent 的评估不像传统软件只有“对/错”。它同时包含答案质量、执行过程、安全边界和资源成本。

| 维度 | 衡量什么 | 示例 |
| --- | --- | --- |
| 准确性 | 回答是否正确 | 问年假天数，是否符合公司规定 |
| 完整性 | 是否遗漏关键信息 | 问部署流程，是否漏掉审批或回滚 |
| 相关性 | 是否紧扣问题 | 问技术问题，是否跑到无关话题 |
| 安全性 | 是否产生有害输出 | 是否泄露敏感信息、是否被 Prompt 注入 |
| 效率/成本 | 工具调用次数和耗时 | 简单问题是否调用 10 次工具 |
| 鲁棒性 | 奇怪输入能否稳住 | 空输入、超长输入、恶意输入 |

工程上不要只看平均分。平均分可能掩盖安全失败、长尾问题和某一类场景的系统性退化。

## 11.3 构建评测数据集

评测数据集就是 Agent 的考试题。每条样例至少要写清：

- 用户问题。
- 期望工具。
- 期望答案关键词。
- 不应该出现的内容。
- 问题类别。

本章代码对应 `practice/44_eval_dataset.py`。

```python
eval_case = {
    "question": "我入职2年了，有几天年假？",
    "expected_tool": "search_knowledge_base",
    "expected_keywords": ["5天", "入职满1年"],
    "category": "factual",
}
```

好的评测集要覆盖：

- 正常事实问答。
- 多子问题完整性。
- 闲聊不调用工具。
- 空输入和超长输入。
- Prompt Injection。
- 无答案拒答。
- 工具失败和噪声结果。

## 11.4 自动化评测

自动化评测要检查两件事：

1. 最终回答是否符合预期。
2. 执行过程是否符合预期。

只看最终文本，会漏掉很多 Agent 特有问题。例如用户闲聊，Agent 调用了知识库；答案看起来还行，但工具选择已经错了。

本章 `practice/45_eval_runner.py` 做了三类检查：

- `check_keywords`：检查关键词 baseline。
- `check_tool_calls`：检查工具调用是否符合期望。
- `check_safety`：检查是否触碰安全红线。

评测结果会保存到 `data/eval_results.json`，方便后续回归、CI 和调优对比。

## 11.5 评测器常见缺陷

老师材料中的 baseline 评测器非常适合入门，但也暴露了几个重要问题。

第一，关键词匹配过于简陋。

- 假阳性：回答“不是 5 天”也命中“5天”。
- 假阴性：回答“5 个工作日”语义正确，但没有连续命中“5天”。

第二，`expected_tool` 如果只写在数据里、不参与评分，就无法发现工具误调。

第三，安全检查不应粗暴混进一个总分。严重安全问题应该单独作为 hard gate。

第四，结果必须持久化。只打印到控制台，无法做版本对比和 CI 阻断。

第五，真实评测中不要每个 case 都重新初始化 LLM client。应复用 client，并记录模型、prompt、工具版本。

## 11.6 常见翻车场景与调优

| 翻车场景 | 症状 | 修复方法 |
| --- | --- | --- |
| 幻觉 | Agent 编造不存在的信息 | 只基于检索结果回答，无证据拒答，注明来源 |
| 死循环 | 反复调用同一个工具 | 设置最大工具调用次数，检测重复调用 |
| 工具误调 | 应查知识库却去计算 | 优化工具描述，加入“何时不用” |
| 格式错乱 | JSON 输出不合法 | 加 schema、few-shot，降低 temperature |
| Prompt 注入 | 恶意输入绕过约束 | 输入校验、输出过滤、角色边界强化 |

修复幻觉的 Prompt 约束可以这样写：

```text
## 重要约束
- 只基于知识库检索到的内容回答，不要添加知识库中没有的信息。
- 如果知识库中没有相关信息，直接回复“抱歉，我在知识库中没有找到相关信息”。
- 不要猜测或编造数据、日期、数字。
- 引用信息时注明来源文档。
```

## 11.7 为什么需要部署

本地命令行脚本只能证明开发者自己能用。真实应用需要：

- 被多个用户同时使用。
- 通过网页、API、CLI 或 App 访问。
- 7x24 小时稳定运行。
- 有监控、日志、成本统计和错误告警。
- 可以灰度、回滚和扩容。

部署就是把 Agent 从“你电脑上的脚本”变成“所有人都能使用的服务”。

## 11.8 用 FastAPI 封装 Agent

FastAPI 很适合做 Agent API：

- Pydantic 请求/响应模型清晰。
- 自动生成 OpenAPI 文档。
- 支持同步和异步接口。
- 容易接入 Uvicorn/Gunicorn。

本章 `practice/47_agent_api.py` 提供两个接口：

- `GET /health`：健康检查，不调用模型。
- `POST /chat`：接受用户消息和 session_id，返回 Agent 回复、工具调用和安全标记。

健康检查不应调用模型。它的职责是告诉负载均衡器“进程是否活着”，不能变成昂贵的业务请求。

## 11.9 流式输出

Agent 常常需要检索、调用工具、等待模型生成。一次性返回会让用户觉得系统卡住。

流式 API 可以分阶段返回：

```json
{"type": "tool_call", "tool": "search_knowledge_base"}
{"type": "token", "content": "根据公司制度"}
{"type": "done"}
```

本章 `practice/48_agent_api_streaming.py` 用 SSE 实现流式输出。真实接入 LangChain/LangGraph 时，可以把 mock 事件替换成 `astream_events`。

## 11.10 前端接入

`practice/chat_frontend.html` 是最小前端页面：

- 输入用户消息。
- 调用 `/chat/stream`。
- 展示工具调用事件。
- 逐块追加 token。

前端实现时要注意：

- SSE 分块可能拆开 JSON，需要 buffer。
- 用户输入不要直接拼到 `innerHTML`。
- 本地开发和线上部署的 API URL 要区分。

## 11.11 生产部署

简单部署链路：

```text
FastAPI app
  -> Uvicorn worker
  -> Gunicorn 多 worker 管理
  -> Nginx 反向代理
  -> 浏览器或 API 客户端
```

SSE 经过 Nginx 时必须关闭缓冲：

```nginx
proxy_buffering off;
proxy_cache off;
proxy_read_timeout 86400s;
```

否则后端已经在流式输出，用户仍可能等到缓冲满了才看到内容。

## 11.12 成本监控

部署后的 Agent 必须知道自己花了多少钱。

至少记录：

- 请求数。
- 输入 token。
- 输出 token。
- 平均延迟。
- 工具调用次数。
- 估算成本。

本章 `practice/49_cost_monitor.py` 用本地估算器演示成本监控。生产环境应优先使用模型供应商返回的 token usage。

## 11.13 Prompt 注入防御

部署后，用户输入不可控。基本防护包括：

- 检测“忽略之前指令”等注入模式。
- 限制输入长度。
- 对外部网页内容做数据/指令隔离。
- 高风险动作必须确认。
- 输出不泄露 system prompt、密钥、cookie。

本章 `practice/50_safety_guardrails.py` 是第一层输入清洗示例，不替代完整权限系统。

## 11.15 核心结论

Agent 工程化的闭环是：

```text
评测集 -> 自动评测 -> 失败诊断 -> 调优修复 -> 回归测试 -> API 部署 -> 监控与安全
```

“能跑”只是开始。“好用”需要评测证明；“可上线”需要部署、监控、成本和安全一起到位。
