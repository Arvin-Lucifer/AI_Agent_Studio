# L03 专题：工具结果不可靠或噪声很大时怎么办

工具返回的是 observation，不是 truth。成熟 Agent 要做的不是“拿到结果就说”，而是先把 observation 处理成可信 evidence。

## 一、面试官在考什么

1. 是否意识到工具输出不等于事实。
2. 是否理解噪声的不同来源。
3. 是否会做结果治理，而不是把脏数据直接喂给模型。
4. 是否知道什么时候继续推理，什么时候停下来。
5. 是否有生产化评估思维。

## 二、噪声类型

| 类型 | 典型问题 |
| --- | --- |
| 召回噪声 | 搜回来很多弱相关结果 |
| 结构噪声 | 网页广告、导航栏、页脚、OCR 错字、表格错位 |
| 冲突噪声 | 多个系统返回关键字段不一致 |
| 数据质量问题 | 缺字段、错字段、旧版本、过期日期 |

## 三、结果治理

### 1. 清洗

- DOM 提纯。
- 正文抽取。
- 去 boilerplate。
- 删除重复段落。
- PDF/OCR 去页眉页脚、合并断行、恢复表格。

### 2. 过滤

只保留和当前问题真正相关的片段。

例子：用户问“退款什么时候到账”，就从退款政策文档中抽取到账时间相关段落，而不是把整篇文档塞给模型。

### 3. 归一化

把不同来源、不同格式的数据整理成统一结构。

例子：

- `high_risk`、`HIGH`、`risk_level=3`、`高风险客户` -> `risk_level: high`
- `2026/4/1`、`April 1, 2026` -> `2026-04-01`

## 四、Rerank

不要迷信 top1。对候选结果做二次排序：

- 语义相关度。
- 时间新鲜度。
- 来源权威性。
- 文档类型。
- 字段完整性。
- 与用户上下文匹配度。

加分表达：

```text
当工具返回噪声较大时，我不会直接依赖初始召回结果，而是会增加 reranking 和 evidence selection，把召回问题转成高质量证据选择问题。
```

## 五、多源交叉验证

适合：

- 高风险业务结论。
- 数值型结论。
- 多系统信息容易不一致。
- 外部网页信息真实性不稳定。

原则：

- 来源权威性有层级。
- 最新版本优先。
- 结构化系统优先于论坛 / 聊天。
- 正式政策优先于个人经验。

## 六、冲突检测

如果多个工具结果在关键字段上不一致，要显式标注。

例子：

```text
退款系统显示处理中，工单系统显示已审批。通常以退款主系统为准，我建议再确认最新流水。
```

用户信任的不是永远自信的答案，而是能发现不一致、说明不一致、给出可信优先级并保守输出的系统。

## 七、证据约束

要求模型：

- 只基于给定 evidence 作答。
- 证据不足时明确说不足。
- 不得补充证据中没有的数值、日期、结论。
- 每个关键结论绑定来源片段。

推荐输出结构：

```json
{
  "answer": "...",
  "supporting_evidence": [],
  "confidence": "low|medium|high",
  "conflicts": [],
  "need_clarification": false
}
```

## 八、保守回答

应该保守的场景：

- 结果来源互相矛盾。
- 检索结果弱相关。
- 时效性无法确认。
- 关键字段缺失。
- 高风险场景。

例子：

```text
我找到了相关政策草案和讨论记录，但没有看到明确的正式生效通知，因此目前不能确认它已正式生效。
```

## 九、面试金句

在 Agent 系统里，工具返回的是 observation，不是 truth；真正的能力在于把 observation 变成可信 evidence。
