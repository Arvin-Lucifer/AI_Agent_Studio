## 基础概念

### 1. 什么是 RAG？解决什么问题？

RAG 是 Retrieval-Augmented Generation，即检索增强生成。它在模型生成答案前，先从外部知识库检索相关资料，再把资料作为上下文交给模型回答。

解决：

- 知识更新。
- 私有知识访问。
- 降低幻觉。
- 来源可追溯。

### 2. 一个典型 RAG 系统包含哪些模块？

数据接入、数据清洗、文档切分、embedding、索引存储、query rewrite、retrieval、rerank、上下文构造、生成答案、引用与评估。

### 3. RAG 和 Fine-tuning 有什么区别？

- RAG：推理时检索外部知识，适合事实知识、频繁更新、引用溯源。
- Fine-tuning：训练时改变模型参数，适合固定任务能力、输出风格、格式遵循。

两者可以结合：RAG 提供事实，微调塑造行为。

### 4. Embedding 的作用是什么？

Embedding 把文本转成向量，让语义相近的文本在向量空间中距离更近。RAG 用它做文档向量化、query 向量化和语义检索。

### 5. 什么是 Chunk？为什么要切分？

Chunk 是文档切出来的小片段。切分是为了适配上下文窗口、提高检索精度、降低成本、支持引用和溯源。

## 工程实现与优化

### 6. Chunk size 如何设置？

没有固定答案，要结合文档类型、上下文窗口、检索目标和评估结果。

- 太大：噪声多、上下文浪费。
- 太小：语义断裂、召回碎片化。

常见策略：固定长度、按段落标题、语义切分、overlap。

### 7. 向量检索和关键词检索有什么区别？为什么要 Hybrid Search？

向量检索擅长语义匹配，但对错误码、函数名、产品型号等精确词不敏感。关键词检索擅长精确匹配，但对同义表达不敏感。Hybrid Search 结合两者，常用于企业知识库和技术文档。

### 8. Reranker 的作用是什么？

Retriever 负责高召回，Reranker 负责高精度。常见流程是先召回 Top50，再 rerank 选 Top5 放入上下文。

### 9. RAG 答案不准确如何排查？

按链路排查：

- 数据源是否有答案。
- 文档解析是否丢表格/图片/标题。
- chunk 是否合适。
- 检索 TopK 是否命中。
- reranker 是否排序正确。
- 上下文是否太噪。
- prompt 是否约束基于上下文。
- LLM 是否错误整合证据。

### 10. 如何评估 RAG？

检索指标：

- Recall@K。
- Precision@K。
- MRR。
- NDCG。

生成指标：

- Answer Correctness。
- Faithfulness / Groundedness。
- Completeness。
- Citation Accuracy。
- Refusal Accuracy。

线上指标：满意度、点赞点踩、人工质检、追问率、无答案率、延迟、成本。

### 11. Query Rewrite 有什么作用？

补全上下文、统一术语、扩展同义词、拆解复杂问题、生成多路 query。风险是偏离原意、过度扩展、继承错误上下文。

## Agentic RAG

### 12. Agent 场景下的 RAG 和普通问答 RAG 有什么区别？

普通 RAG 是一次性“检索 -> 生成”。Agentic RAG 中，Agent 可以多步规划、多轮检索、选择知识源、调用其他工具、维护中间状态，并判断证据是否足够。

### 13. 什么是 Agentic RAG？

Agentic RAG 是让检索成为 Agent 决策的一部分。常见架构包括 Router-based RAG、Multi-step RAG、Corrective RAG、Self-RAG、Tool-augmented RAG。

### 14. 多跳问题如何处理？

使用问题拆解、迭代检索、结构化中间状态、证据图和最终综合。不能只靠单次 TopK 检索。

### 15. 企业级 RAG 权限控制如何设计？

原则：用户只能检索和看到自己有权限访问的内容。

方案：

- chunk 继承原文档 ACL。
- 索引层权限过滤。
- 上下文注入前再次校验。
- 权限变更后增量更新索引。
- 引用链接也要做权限检查。
- 缓存按用户或权限集合隔离。
- 记录审计日志。

### 16. 如何处理冲突和过期信息？

维护文档元数据：更新时间、版本号、状态、权威等级、Owner、适用范围。检索排序加入时间和权威权重。冲突时显式说明，不要强行合并。

### 17. 如何降低 RAG 延迟和成本？

缓存、减少不必要的 LLM 调用、控制 TopK、两阶段检索、上下文压缩、模型分层、并行检索、索引优化。

### 18. 如何防止 RAG Prompt Injection？

把检索内容视为数据而不是指令。使用内容检测、工具权限控制、引用隔离、输出约束、高风险二次确认、最小权限和审计。

## 综合设计

### 19. 设计企业内部知识库 Agent RAG 系统

回答应覆盖：

- 离线链路：数据接入、解析、清洗、chunk、权限同步、embedding、索引。
- 在线链路：意图识别、权限识别、rewrite、hybrid search、rerank、上下文构造、生成、引用、日志。
- Agent：选择知识源、多轮检索、证据判断、追问。
- 权限：ACL、过滤、缓存隔离、审计。
- 评估：Recall@K、MRR、Faithfulness、Citation Accuracy、线上反馈。
- 运维：版本治理、增量索引、过期检测、监控报警。

### 20. 知识库没有答案怎么办？

不要强行回答。应根据检索分数、reranker 阈值和上下文支持度判断无答案，明确说明无法从现有资料确认，并引导用户补充信息或反馈知识缺口。

### 21. 你做过的 RAG 项目中效果最差的问题类型是什么？

可从多跳、术语、长文档、表格、PDF、幻觉、召回噪声、无答案乱答等方向回答，并说明具体优化。

### 22. 为什么检索结果是对的，最终答案还是错的？

检索对只是原料对，模型仍可能被噪声干扰、截断关键上下文、不会多跳推理、误用冲突版本、脱离上下文回答。优化方向：细切片、rerank、上下文压缩、强 prompt、多跳拆解、答案校验。

### 23. 如何处理表格、图片、PDF 等非纯文本数据？

表格保留表头和单元格关系，转 Markdown/JSON；图片 OCR + 视觉描述；PDF 做版面解析、标题层级、页眉页脚清洗、页码和 bbox 保留。不要简单转纯文本。

## RAG、PageIndex、LLM Wiki 选型

| 方案 | 一句话 | 适合 |
| --- | --- | --- |
| RAG | 现查现用，通用成熟 | 大规模、频繁更新、开放域文档问答 |
| PageIndex | 基于文档结构和推理导航 | 单篇超长结构化文档、法律/财报/规范 |
| LLM Wiki | 知识编译成可维护 Wiki | 个人/小团队长期知识沉淀 |

决策：

- 数据规模大、更新频繁：RAG。
- 单篇长文档、高可解释：PageIndex。
- 小团队、长期沉淀、极简透明：LLM Wiki。
