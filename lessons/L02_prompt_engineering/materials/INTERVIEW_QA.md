## 1. 如何改写一个模糊 Prompt？

原 prompt：

```text
帮我总结这篇文章，越详细越好。
```

改写思路：

- 补充角色：你是研究助理、代码 Reviewer、客服质检员等。
- 明确任务目标：总结、抽取、改写、分类、生成计划。
- 约束输出结构：标题、要点、证据、不确定信息。
- 加上长度和风格限制。
- 明确禁止编造，缺少依据时要标注。

参考回答：

```text
我会把模糊 prompt 改成“角色 + 任务 + 输入边界 + 输出结构 + 质量约束”。例如总结文章时，不会只写“越详细越好”，而是要求输出一句话结论、3-5 个核心观点、关键证据、适用读者和不确定信息，并限制字数、禁止添加原文没有的信息。
```

## 2. Prompt 很好但 token 太贵怎么办？

回答要点：

- 删除重复规则，保留真正影响行为的高优先级约束。
- 把稳定规则模板化，不在每次请求里重复拼接无关内容。
- few-shot 只保留覆盖边界的代表样本。
- 长历史做摘要或检索，只放当前任务需要的上下文。
- 分层模型：简单任务用便宜模型，复杂任务再升级。

## 3. 如何评估 Prompt 效果？

不要只看单次演示，应建立固定测试集。

指标：

- 正确率。
- 格式稳定性。
- 越界率。
- 幻觉率。
- 成本。
- 延迟。
- 人工满意度或业务指标。

高质量回答：

```text
我会先固定一组覆盖正常、边界、异常和越权输入的测试集，每次改 prompt 都跑同一批样例，比较正确率、格式稳定性、越界率、幻觉率、成本和延迟。失败样本会进入回归集，避免下一版 prompt 把旧问题改回来。
```

## 4. 怎么降低幻觉？

回答要点：

- Prompt 明确要求“基于给定资料回答”。
- 缺少依据时必须说不知道。
- 要求输出证据字段或引用来源。
- 对事实类问题接入 RAG 或工具。
- 对关键结论做二次校验。

## 5. Few-shot 越多越好吗？

不是。Few-shot 的核心不是数量，而是覆盖关键边界。

选择样本时关注：

- 是否覆盖正常、边界、异常输入。
- 输出格式是否一致。
- 是否存在相互冲突。
- token 成本是否可接受。

## 6. CoT 一定更好吗？

不一定。

适合：

- 多步推理。
- 数学、逻辑、规划。
- 需要拆解问题的复杂任务。

不适合：

- 简单分类。
- 低延迟任务。
- 不适合暴露详细推理的场景。

生产表达可以改成“先分析再给结论，但只输出结论和必要依据”。

## 7. JSON 输出不稳定怎么办？

回答要点：

- 降低 temperature。
- 给出明确 JSON schema。
- 使用 few-shot 展示合法输出。
- 用解析器做字段校验。
- 失败后有限重试或兜底提取。
- 优先使用 SDK 或模型提供的结构化输出能力。

## 8. System Prompt 很长，User Prompt 也很长，怎么分配上下文？

回答框架：

- System 放稳定身份、边界、安全规则和输出契约。
- User 放当前任务和当前输入。
- 历史对话做摘要或检索。
- 示例少而精。
- 重要业务规则优先级高于风格性描述。
