## 二、Chain-of-Thought Prompting 示例

### 1. 数学推理（基础 CoT）

```text
请一步一步思考后给出答案。

问题：一个书架有5层，第1层放了3本书，从第2层开始，每一层比前一层多放2本书。问书架上一共有多少本书？

思考过程：
- 第1层：3本
- 第2层：3 + 2 = 5本
- 第3层：5 + 2 = 7本
- 第4层：7 + 2 = 9本
- 第5层：9 + 2 = 11本
- 总计：3 + 5 + 7 + 9 + 11 = 35本

答案：35本
```

```text
思考过程：（留空供学员练习）
```

### 2. 逻辑推理（中级 CoT）

```text
请逐步分析并给出结论。

问题：公司有A、B、C、D四个人竞聘经理职位，已知：
1. 如果A当选，则B也当选
2. B和C中只能有一人当选
3. C和D要么都当选，要么都不当选
4. D当选了

请问最终谁当选了？

思考过程：
- 由条件4：D当选
- 由条件3：C和D同进退 -> C也当选
- 由条件2：B和C只能选一个 -> B不当选
- 由条件1的逆否命题：B不当选 -> A不当选
- 结论：C和D当选，A和B未当选

答案：C和D当选。
```

练习：

```text
问题：有5个人（甲乙丙丁戊）坐成一排，满足以下条件：
1. 甲不坐在最左边
2. 乙不坐在最右边
3. 甲和丙之间恰好隔一个人
4. 丁紧挨着戊

请给出一种可行的座位排列。

思考过程：（留空供学员练习）
```

### 3. 多步骤任务规划（Agent CoT）

```text
你是一个 AI Agent，面对复杂任务时需要先规划再执行。

任务：用户说"帮我分析我们团队上周的 OKR 完成情况，生成一份报告发到团队群里"

思考过程：
1. 分解任务：
   a) 获取团队 OKR 数据
   b) 分析完成情况并生成报告
   c) 将报告发送到团队群

2. 依赖关系分析：
   - b 依赖 a 的结果
   - c 依赖 b 的结果
   - 必须按顺序执行

3. 工具选择：
   - a -> 调用 OKR 系统 API 获取数据
   - b -> 本地数据分析 + 文档生成
   - c -> 调用 IM 发送接口

4. 缺失信息检查：
   - "我们团队"指哪个团队？ -> 需确认或从上下文推断
   - "上周"的时间范围？ -> 可计算
   - "团队群"是哪个群？ -> 需确认

5. 决策：先确认缺失信息，再按顺序执行。

回复："我来帮你完成这个任务。请确认两个信息：
1）团队名称或 OKR 所在的项目空间？
2）报告发送的目标群聊名称是？"
```

### 4. 代码调试（CoT + 自我纠错）

```python
def find_duplicates(lst):
    seen = {}
    duplicates = []
    for item in lst:
        if item in seen:
            duplicates.append(item)
        seen[item] = True
    return duplicates
```

输入：`[1, 2, 3, 2, 4, 3, 2]`  
期望输出：`[2, 3]`  
实际输出：`[2, 3, 2]`

分析要点：

1. 当一个元素重复出现 3 次及以上时，会被多次加入 `duplicates`。
2. 修复方式：加入 `duplicates` 前检查是否已存在，或使用 `set` 去重。

修复后代码：

```python
def find_duplicates(lst):
    seen = set()
    duplicates = set()
    for item in lst:
        if item in seen:
            duplicates.add(item)
        seen.add(item)
    return list(duplicates)
```

### 5. Zero-Shot CoT

最简洁的 CoT 触发方式，是在提示末尾加上引导语：

```text
问题：一个农场有鸡和兔共35只，脚共94只，请问鸡和兔各有多少只？

请一步一步思考（Let's think step by step）：
```

这种写法不需要给出示例，模型也会更倾向于展开推理步骤。

## 三、使用建议总结

| 技术 | 适用场景 | 核心要点 |
| --- | --- | --- |
| Few-Shot | 格式规范、分类、转换、结构化输出 | 示例要典型且多样，覆盖边界情况 |
| CoT | 数学推理、逻辑判断、多步决策 | 展示中间步骤，降低跳跃性错误 |
| Few-Shot + CoT | 复杂推理 + 固定格式输出 | 在示例中同时展示推理过程和输出格式 |
| Zero-Shot CoT | 快速原型、通用推理增强 | 加“请一步步思考”即可快速触发推理 |

## 四、设计原则

### Few-Shot 示例设计原则

1. 多样性：覆盖不同类型输入，包括正例、负例和边界情况。
2. 一致性：各示例输出格式必须统一。
3. 渐进性：从简单到复杂排列，帮助模型学习层次。
4. 数量适中：通常 2-5 个即可，过多会占用 context window。

### CoT 设计原则

1. 显式化：每个推理步骤都写出来，不跳步。
2. 可验证：每步的中间结果可以被检查。
3. 结构化：用编号或分层组织推理过程。
4. 收束明确：推理链最终收敛到明确结论。

## 五、Few-Shot 数量与质量的平衡

核心原则：用最少的示例传递最大的信息增量。

| 维度 | 示例多 | 示例少 |
| --- | --- | --- |
| 准确率 | 模式更稳定 | 可能不稳定 |
| 上下文占用 | 大，挤压输入/输出空间 | 小，留出更多空间 |
| 延迟与成本 | 更高 | 更低 |
| 过拟合风险 | 模型可能照抄示例 | 较低 |

| 任务类型 | 建议数量 | 理由 |
| --- | ---: | --- |
| 二分类 | 1-2 个 | 模型已有较强先验 |
| 多分类 | 每类 1 个，共 3-5 个 | 需要覆盖类别边界 |
| 结构化输出 | 1-2 个 | 重点是格式，不是语义 |
| 工具调用决策 | 每种工具至少 1 个 | 让模型学会路由边界 |
| 复杂推理 | 2-3 个 | 需要展示推理路径和收束方式 |
