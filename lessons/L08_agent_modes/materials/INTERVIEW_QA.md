# L08 面试题库：Agent 模式、多 Agent 与企业知识库问答

说明：本题库基于老师第 8 讲“多 Agent 协作系统”和补充材料“企业知识库问答助手设计要点：Agent 模式选型与分库分索引策略”整理，并做了面试表达层面的结构化补充。补充内容用于帮助回答更完整，不替代原讲义观点。

## 一、基础题

### 1. 什么是 Agent 模式选型？

Agent 模式选型是指根据任务复杂度、知识来源、工具需求、可追溯要求、成本和延迟，选择 Direct、RAG、ReAct、Plan-and-Execute、Reflection 或 Multi-Agent 等模式。

面试回答要点：

- 不同任务不应默认上最复杂的 Agent。
- 企业问答通常先判断是否需要私域知识。
- 需要多步检索、跨源融合或计算时，再升级到 ReAct。
- 需要角色分工和多阶段质量控制时，再考虑多 Agent。

### 2. 企业知识库问答为什么推荐 RAG 为主？

因为企业知识通常有私域性、时效性和合规溯源要求。RAG 可以把外部知识检索进上下文，并通过引用 chunk、文档版本和来源来约束答案。

关键词：

- 企业文档不在通用模型训练集中。
- 制度、价格、流程、接口会更新。
- 答案需要可追溯、可审计。
- 检索证据能降低幻觉。

### 3. 为什么还要叠加 ReAct？

纯 RAG 适合单轮事实查询，但多跳、模糊、跨源、计算型问题经常需要拆解和多步工具调用。ReAct 可以让 Agent 先思考下一步，再调用检索、SQL、计算器或其它工具。

典型场景：

- “P0 工单响应时间和 API 发布审批有什么共同要求？”
- “请比较 HR 年假政策和财务报销流程的审批差异。”
- “统计近 30 天某产品线的高优工单数量。”

### 4. 为什么不推荐纯 Fine-tune 做企业知识库问答主干？

纯 Fine-tune 有三个问题：知识更新需要重训，答案难以溯源，模型可能遗忘或混淆知识。企业知识库问答更适合用检索系统做知识更新和版本管理。

### 5. 什么是 Supervisor 多 Agent 模式？

Supervisor 模式是由一个主管或控制逻辑负责调度多个专家 Agent。专家 Agent 只负责自己的子任务，主管决定任务流向、是否返工、何时结束。

适合：

- 多部门客服分流。
- 企业知识库跨域路由。
- 研报生成团队。
- 数据分析和报告审核流程。

## 二、中等题

### 6. Supervisor、Sequential、Peer 三种多 Agent 模式有什么区别？

| 模式 | 特点 | 适合场景 | 风险 |
| --- | --- | --- | --- |
| Supervisor | 主管调度专家 | 跨域任务、动态路由 | 主管判断错误会影响全局 |
| Sequential | 固定流水线 | 流程明确的任务 | 灵活性较低 |
| Peer | 平等讨论后汇总 | 创意、评审、方案比较 | 成本高、冲突难控 |

回答时可以结合本章代码：调研员、分析师、撰写员、审核员构成了流水线，审核不通过再回到撰写员，体现了 Supervisor/Sequential 混合。

### 7. 多 Agent 系统为什么要有明确的 State？

因为 State 是 Agent 之间的交接协议。它明确每个 Agent 读取什么、写入什么、谁负责哪个字段。没有 State，多个 Agent 只能靠自然语言随意传递，容易出现信息丢失、责任不清、无法审计和难以调试。

本章 `TeamState` 包含：

```text
topic, research_data, analysis, report, review_feedback,
is_approved, revision_count, use_llm
```

### 8. 多 Agent 如何避免无限循环？

常见做法：

- 设置最大轮数，例如最多返工 2-3 次。
- 设置最大 token、超时和预算。
- 每轮必须产生结构化状态变化。
- 多次失败后降级为人工审核或返回部分结果。
- 记录 trace，方便定位循环原因。

本章代码用 `revision_count` 防止 Writer 和 Reviewer 无限返工。

### 9. 企业知识库为什么需要分库分索引？

单库方案会遇到四类问题：

- 权限墙：不同部门和敏感等级必须隔离。
- 语料异构墙：制度、代码、工单、表格需要不同切片和 embedding。
- 更新频率墙：冷数据和热数据不应共享同一更新节奏。
- 效果调优墙：同一关键词在不同业务域语义不同。

因此推荐按业务域为主键，叠加数据形态、敏感等级和时效性。

### 10. 多库召回后为什么要 rerank？

不同库的向量分数、BM25 分数、embedding 模型和切片策略可能不同，原始分数不可直接比较。统一 rerank 可以把来自不同库的候选文档放到同一个排序标准下，提升证据选择质量。

## 三、偏复杂题

### 11. 如何设计企业知识库问答的整体架构？

推荐分层：

1. 路由层：判断 direct、单轮 RAG、ReAct 多步。
2. 检索层：BM25 + 向量 + 元数据过滤 + rerank，按业务域分库。
3. Agent 层：必要时调用检索、SQL、计算器、业务 API。
4. 答案层：强制引用来源，无证据时拒答。
5. 评测回流：跟踪命中率、忠实度、引用正确率、权限拦截率和 bad case。

工程约束：

- 工具数量收敛到 5 个以内。
- ReAct 步数上限建议 3-5 步。
- 检索前置权限过滤。
- 日志记录路由、检索、引用和错误。

### 12. 如何设计分库分索引的元数据 Schema？

建议统一字段：

```text
doc_id, domain, sensitivity, owner_dept, valid_from, valid_to,
source_url, version, lang, tags
```

作用：

- `domain` 支持业务域路由。
- `sensitivity` 支持权限过滤。
- `valid_from/valid_to/version` 支持时效和版本控制。
- `source_url/doc_id` 支持答案溯源。
- `tags/lang` 支持检索增强和多语言治理。

### 13. 权限过滤应该放在哪里？

应该尽量前置到检索阶段。用户身份、角色、部门、租户和敏感等级应该进入检索 filter，而不是检索后才在答案层过滤。

原因：

- 避免越权内容进入模型上下文。
- 避免敏感内容出现在日志和 trace 中。
- 降低提示注入或模型泄露风险。
- 满足最小可见原则。

答案层仍然可以做二次检查，但不能替代检索前过滤。

### 14. 如何判断一个问题走 RAG 还是 ReAct？

可以从四个维度判断：

- 是否单一事实查询。是，则优先单轮 RAG。
- 是否需要跨多个业务域。是，倾向 ReAct 或多库路由。
- 是否需要计算、聚合或调用数据库。是，倾向 ReAct。
- 是否问题模糊，需要澄清或实体补全。是，先澄清或用 ReAct 改写。

示例：

- “入职 3 年有几天年假？”适合 RAG。
- “P0 工单响应时间和 API 发布审批有什么共同要求？”适合 ReAct。

### 15. 为什么不建议所有问题都走多 Agent？

因为多 Agent 会增加 token、延迟、错误传播、状态管理和调试复杂度。高频简单问答走 direct 或 RAG 更稳定。多 Agent 应该用于复杂产出、专业分工、质量审核或跨角色协作，而不是所有问题的默认路径。

## 四、系统设计题

### 16. 设计一个企业知识库问答助手，你会怎么做？

可以按以下结构回答：

需求目标：

- 支持 HR、财务、研发、客服等知识库问答。
- 答案必须可追溯。
- 用户只能看到自己有权限的内容。
- 多跳和跨源问题能被处理。

架构设计：

- 路由层：规则路由覆盖高频问题，分类器覆盖中频问题，Agent 路由处理长尾。
- 检索层：按业务域分库，BM25 + 向量混合检索，metadata filter 前置权限。
- Rerank 层：多库候选统一重排。
- Agent 层：RAG 为主，ReAct 处理多跳、跨源和计算。
- 答案层：引用 `doc_id/version/source_url`，无证据拒答。
- 评测层：维护黄金问答集、引用正确率、越权测试和 bad case 回流。

落地节奏：

1. 按业务域粗分 3-5 个核心库，统一 Schema，规则路由跑通。
2. 增加敏感等级隔离和权限前置过滤，接入 rerank。
3. 上线分类器路由和 ReAct 路由，处理跨域长尾。
4. 按业务域独立评测和 A/B 实验，形成域 owner + 平台治理。

### 17. 如何设计“分库分索引 + 路由”的落地方案？

建议回答：

- 业务域是主键，例如 HR、Finance、Engineering、Support。
- 数据形态决定切片和 embedding，例如制度按章节，API 按接口，工单整条，表格保留表头。
- 敏感等级决定物理隔离和鉴权策略。
- 时效性决定更新频率和索引刷新方式。
- 路由层先用规则覆盖高频，再用分类器处理自然语言变化，最后用 Agent 路由处理长尾。
- 多库并行召回后统一 rerank，答案必须带引用。

### 18. 如何评估这个企业知识库问答系统？

评估指标：

- 检索命中率：正确文档是否进入 top-k。
- 忠实度：答案是否只基于证据。
- 引用正确率：引用是否真的支持答案。
- 拒答正确率：无证据时是否拒答。
- 权限拦截率：越权问题是否不会泄露。
- 延迟和成本：不同模式下的响应时间和 token 成本。
- 用户满意度和 bad case 类型分布。

## 五、追问题

### 19. 如果 RAG 检索不到答案怎么办？

优先拒答或请求澄清，不编造。可以尝试查询改写、扩展业务域、提升召回 top-k、使用 ReAct 多步检索，仍无证据时回答“未找到相关依据”。

### 20. 如果多个知识库结果冲突怎么办？

按版本、有效期、来源权威性和业务域 owner 决策。答案中说明冲突，并优先引用最新且权威的来源；无法判断时请求人工确认。

### 21. 如果用户问题同时涉及机密库和公开库怎么办？

先根据用户权限确定可见范围，只检索允许访问的库。如果用户无权访问机密库，即使公开库里有间接线索，也不能泄露机密内容。答案可以说明“当前权限下未找到可引用依据”。

### 22. 如果多 Agent 结果不一致怎么办？

不要让 Agent 无限争论。可以让 Reviewer 根据证据、引用、规则和置信度做裁决；无法裁决时进入人工审核或返回多个有来源的候选结论。

### 23. 如何把本章代码改成生产系统？

需要替换或增强：

- mock 知识库替换为真实向量库、BM25、数据库和文档平台。
- 规则路由升级为规则 + 分类器 + Agent 路由。
- 增加用户身份、租户、权限和审计。
- 增加真实 rerank 模型。
- 增加评测集、监控、trace 和告警。
- 对所有 Agent 设置超时、最大步数、异常兜底和人工接管。
