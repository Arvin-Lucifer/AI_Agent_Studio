# 企业知识库问答助手设计要点

本文档整理第 8 章补充材料中的两个核心设计问题：

1. Agent 模式选型。
2. 分库分索引策略。

## 1. Agent 模式选型

### 1.1 推荐模式

推荐使用：

```text
RAG 为主 + ReAct 为辅
```

RAG 负责把企业私域知识可靠地注入上下文，ReAct 负责处理单轮 RAG 难以覆盖的多跳、跨源、模糊和计算型长尾问题。

### 1.2 为什么 RAG 是主干

企业知识库问答有四个关键要求：

- 数据私域性：企业制度、合同、工单、研发文档通常不在通用模型训练集中。
- 可溯源：合规、审计和责任归属要求答案能追溯到文档、版本和 chunk。
- 知识时效：制度、流程、价格和接口会变化，检索更新比微调更新更轻。
- 幻觉抑制：相比纯 LLM 直答，检索证据能明显降低编造风险。

### 1.3 为什么叠加 ReAct

| 场景 | 单轮 RAG 表现 | ReAct 价值 |
| --- | --- | --- |
| 多跳问答 | 一次检索拿不全证据 | 拆解为多次检索后比对 |
| 模糊提问 | 检索词噪声大 | 先澄清或补全实体再检索 |
| 跨源融合 | 单一向量库不够 | 按问题路由不同工具 |
| 计算型问题 | 检索不出聚合结果 | 调用 SQL 或计算工具 |

### 1.4 不推荐的模式

纯 Fine-tune：

- 知识更新需要重训。
- 成本高。
- 不可溯源。
- 容易遗忘或混淆知识。

纯 Plan-and-Execute：

- 对一次问答过重。
- 延迟高。
- 计划错误会传播到后续步骤。

Multi-Agent 辩论或 Reflection 大循环：

- token 和延迟成倍增长。
- 问答场景 ROI 通常偏低。
- 如果没有证据约束，多个 Agent 也可能一起编造。

### 1.5 落地架构

```text
用户问题
  -> 路由层
      -> direct
      -> single RAG
      -> ReAct multi-step
  -> 检索层
      -> BM25
      -> vector search
      -> metadata filter
      -> rerank
  -> Agent 层
      -> search tool
      -> SQL/calculation tool
      -> business API
  -> 答案层
      -> citations
      -> no evidence => 未找到
  -> 评测与回流
```

工程建议：

- 检索层使用混合检索和 rerank。
- 路由层先用规则和分类器，只有长尾问题交给 Agent 路由。
- Agent 工具集收敛到 5 个以内。
- ReAct 步数上限控制在 3-5 步。
- 答案必须引用来源。
- 无证据时显式回答“未找到”，禁止编造。
- 建立评测集，持续看命中率、忠实度、引用正确率和权限拦截。

## 2. 分库分索引策略

### 2.1 为什么要分库

单库方案会同时遇到四堵墙。

权限墙：

- HR、法务、财务、研发的数据敏感度不同。
- 必须做到最小可见原则。
- 机密内容不能进入无权限用户的检索上下文。

语料异构墙：

- 长文档、代码、工单、表格、多模态内容的最佳切片策略不同。
- 不同语料可能需要不同 embedding 模型。

更新频率墙：

- 制度可能年更。
- 产品文档可能月更。
- 工单可能分钟级更新。
- 统一索引会拖慢全局刷新。

效果调优墙：

- 同一关键词在不同业务域含义不同。
- 单一向量空间难以同时服务所有领域。

### 2.2 切分维度

| 维度 | 划分依据 | 示例 |
| --- | --- | --- |
| 业务域 | 部门或产品线 | HR、财务、法务、研发、客服 |
| 数据形态 | 文本结构 | 长文档、FAQ、代码、表格、多模态 |
| 敏感等级 | 合规分级 | 公开、内部、机密、绝密 |
| 时效性 | 更新节奏 | 静态制度、动态工单、实时指标 |

推荐以业务域为主键，其它维度作为子索引或字段过滤。

### 2.3 关键实施动作

#### 2.3.1 索引层物理隔离

- 每个业务域独立 collection 或 index。
- 机密级数据独立集群部署。
- 至少在网络层和鉴权层隔离。

#### 2.3.2 Embedding 模型按域适配

- 通用域：`bge-m3`、`text-embedding-3-large`。
- 代码域：CodeBERT、jina-code。
- 多语言域：使用多语模型，避免中英文混库导致语义漂移。

#### 2.3.3 切片策略按域定制

| 数据类型 | 推荐切片 |
| --- | --- |
| 制度文档 | 按章节 + 标题语义切，512-1024 token |
| API 文档 | 按函数或接口切，签名、示例、参数放同一 chunk |
| 工单 | 整条作为一个 chunk |
| 表格 | 行级或表级切片，保留表头上下文 |
| FAQ | 一问一答或相近问法合并 |

#### 2.3.4 元数据 Schema 统一

推荐字段：

```text
doc_id, domain, sensitivity, owner_dept, valid_from, valid_to,
source_url, version, lang, tags
```

这些字段分别服务于溯源、路由、权限、版本、时效、多语言和标签过滤。

#### 2.3.5 路由层三档策略

| 路由方式 | 覆盖流量 | 特点 |
| --- | --- | --- |
| 规则路由 | 高频 60% | 零模型成本、低延迟、可解释 |
| 分类器路由 | 中频 30% | BERT/LLM 映射到 1-3 个候选域 |
| Agent 路由 | 长尾 10% | 跨域、多跳、动态工具调用 |

#### 2.3.6 召回融合

多库并行召回后，要使用 rerank 模型统一重排。

原因是不同库的分数不可比：

- embedding 模型可能不同。
- 切片大小不同。
- BM25 和向量分数尺度不同。
- 各库数据分布不同。

#### 2.3.7 权限过滤

权限过滤要前置：

- 在向量检索 filter 阶段带上角色、部门、租户、敏感等级。
- 机密库使用独立鉴权服务。
- 使用短时 token。
- 审计日志记录谁在什么时候检索了哪些域和文档。

答案阶段可以做二次检查，但不能替代检索前过滤。

## 3. 优势汇总

| 维度 | 收益 |
| --- | --- |
| 效果 | 领域内召回更准，常见情况下召回率可比单库高 15%-30% |
| 安全合规 | 满足最小可见原则，便于等保、SOX、GDPR 等审计 |
| 成本 | 简单问题走轻量路径，整体推理成本可下降 |
| 运维 | 故障爆炸半径小，单库变更不影响全局 |
| 迭代 | 各域独立评测和 A/B 实验 |
| 扩展 | 新域接入只需增加 collection 和路由规则 |

## 4. 落地节奏

阶段一：

- 按业务域粗分 3-5 个核心库。
- 统一 metadata schema。
- 用规则路由跑通端到端闭环。

阶段二：

- 补充敏感等级隔离。
- 接入权限前置过滤。
- 接入 rerank。

阶段三：

- 上线分类器路由。
- 增加 Agent 路由，处理跨域长尾。
- 引入 SQL、报表或业务 API 工具。

阶段四：

- 按域独立评测调优。
- 建立域 owner + 平台团队的双层治理。
- bad case 回流到切片、路由、检索、rerank 和提示词策略。

## 5. 核心结论

Agent 模式方面：

- 用 RAG 保证答案可信与可溯源。
- 用 ReAct 处理多跳、跨源和计算型长尾。
- 用前置路由控制成本和延迟。

分库分索引方面：

- 把“通用问答”拆成多个“领域问答”子问题。
- 领域内使用最适合的切片、embedding、权限和评测策略。
- 领域间通过路由层、rerank 和统一答案层协调。
