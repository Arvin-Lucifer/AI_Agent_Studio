# Agent 模式选型速查

四种经典 Agent 设计模式的完整教辅讲解见 [Agent 设计模式](../../../teaching_support/AGENT_DESIGN_PATTERNS.md)。本文件更偏工程选型速查。

## 1. 先用问题复杂度筛选

```text
是否需要私域/实时知识？
  否 -> Direct
  是 -> 是否单一事实查询？
        是 -> RAG
        否 -> 是否需要多步、跨源、计算或工具？
              是 -> ReAct
              否 -> RAG + 查询改写

是否需要长流程规划？
  是 -> Plan-and-Execute

是否需要多角色专业分工或质量审核？
  是 -> Multi-Agent

是否需要高风险自检？
  是 -> Reflection，但必须限制轮数和预算
```

## 2. 模式对比表

| 模式 | 输入 | 中间过程 | 输出 | 成本 | 可控性 |
| --- | --- | --- | --- | --- | --- |
| Direct | 用户问题 | 单次模型生成 | 答案 | 低 | 中 |
| RAG | 问题 + 检索证据 | 检索、重排、生成 | 带引用答案 | 中 | 高 |
| ReAct | 问题 + 工具 | 思考、行动、观察循环 | 多步综合答案 | 中高 | 中 |
| Plan-and-Execute | 复杂目标 | 计划、执行、修正 | 任务结果 | 高 | 中 |
| Reflection | 初稿 | 自检、修订 | 修订结果 | 高 | 中 |
| Multi-Agent | 复杂任务 | 多角色协作 | 综合产物 | 高 | 取决于协议 |

## 3. 企业问答默认策略

企业知识库问答建议从这条路径开始：

```text
规则路由
  -> 单轮 RAG
  -> 无证据拒答
  -> 多跳/跨源/计算问题升级 ReAct
  -> 复杂产出型任务再考虑 Multi-Agent
```

不要默认把所有问题交给多 Agent。多 Agent 可以提升复杂任务质量，但对简单问答会增加延迟、token 成本和调试难度。

## 4. 什么时候升级到 ReAct

从 RAG 升级到 ReAct 的典型信号：

- 问题包含比较、共同点、分别、跨部门、多个实体。
- 需要多次检索才能拿全证据。
- 需要调用 SQL、计算器、报表 API 或业务系统。
- 用户问题模糊，需要先补全实体或澄清业务域。
- 检索结果冲突，需要进一步查版本、有效期或权威来源。

## 5. 什么时候升级到 Multi-Agent

从单 Agent 升级到多 Agent 的典型信号：

- 任务有明显阶段：调研、分析、写作、审核。
- 任务需要多个专业视角：法务、财务、研发、安全。
- 输出需要质量门禁：事实核查、合规检查、风格检查。
- 单 Agent 提示词过长，职责混杂，结果不稳定。

## 6. 生产约束

无论选择哪种模式，都要有这些约束：

- 最大步数。
- 最大 token。
- 超时和重试。
- 工具白名单。
- 权限前置过滤。
- 引用和 trace。
- bad case 评测回流。
