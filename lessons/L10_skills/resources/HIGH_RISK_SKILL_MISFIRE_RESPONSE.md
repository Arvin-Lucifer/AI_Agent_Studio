# 高风险 Skill 误触发：事后损失控制

高风险 Skill 误触发已经发生时，第一优先级不是追责，也不是马上改代码，而是：

```text
止损 -> 评估影响 -> 分级补救 -> 透明告知 -> 结构加固
```

误触发事故的损失大小，很大程度取决于前 10 分钟的处置质量。

## 1. 黄金 5 分钟：先止血

误触发发生后，第一目标是防止继续扩散。

### 1.1 立刻切断同类执行

- 熔断该 Skill：对当前用户、会话、租户或全局停用该能力。
- 冻结相关自动化规则：如果是授权类任务，暂停同 scope 的规则。
- 检查队列和排期：取消尚未执行的后续批次。

注意：先停，再查。不要在搞清楚根因之前让系统继续自动执行。

### 1.2 锁定影响面

立刻拉取调用日志：

- 时间。
- Skill 名称。
- 参数。
- 目标对象。
- 成功数、失败数。
- 是否有后续队列、轮询或定时任务。

关键判断是：事故还在发生，还是已经停止。持续执行类 Skill 要优先切流和取消队列。

### 1.3 判断是否升级

触发任一信号，应立即升级给人工 oncall、业务 owner 或合规负责人：

- 涉及外部对象：对外邮件、群发、客户通知。
- 涉及资金、权限、合规。
- 影响范围超过个位数目标。
- 不可逆操作：删除、打款、发布。

## 2. 损失评估：先量化，再补救

没评估清楚就补救，可能造成二次损失。

| 维度 | 关键问题 | 决定 |
| --- | --- | --- |
| 可逆性 | 操作能撤销吗？撤销动作本身有副作用吗？ | 是否回滚 |
| 扩散性 | 是否已经被他人看到、转发或消费？ | 是否还能静默处理 |
| 关联影响 | 是否触发下游工作流、通知、自动回复？ | 补救范围 |

经验法则：

- 可逆 + 未扩散：静默回滚 + 事后告知。
- 不可逆或已扩散：不要硬撤，走澄清或补偿路径。
- 已触发下游：先评估下游，再回头处理上游。

## 3. 分场景补救策略

### 3.1 可逆操作

适用：建文档、建日程、建任务、写记录。

动作：

- 先保留证据副本，例如截图、导出、trace。
- 删除或标记作废。
- 删除动作本身也要留审计日志，方便后续对账。

### 3.2 已送达但可撤回

适用：IM、内部邮件、群通知。

动作：

- 消息在撤回窗口内可以撤回。
- 超过撤回窗口时，优先发补充说明，不要硬删造成更大困惑。
- 外部邮件应立即发更正邮件，标题加 `[更正]` 或 `[Correction]`。

关键：澄清比删除更重要，因为对方可能已经读过。

### 3.3 金额或权限类

适用：付款、退款、权限授予、订单创建。

动作：

- 立即走反向操作流程：退款、权限回收、订单取消。
- 不要用另一个自动化去修复自动化的错。
- 同步合规、风控、财务或权限系统 owner。

### 3.4 对外或跨部门影响

动作：

- 主动告知受影响方。
- 准备标准话术：原因、影响范围、已采取措施、后续预防。
- 越早告知越主动，越晚越被动。

### 3.5 触发下游工作流

动作：

- 沿调用链回溯：`Skill A -> Skill B -> Webhook -> 第三方系统`。
- 先停最下游，防止继续扩散，再往上回滚。
- 第三方系统不可控时，尽快联系对方支持或 owner。

## 4. 透明告知用户

无论损失大小，都要主动告诉用户发生了什么，不能静默处理。

四段式结构：

1. 发生了什么。
2. 已采取措施。
3. 残余影响。
4. 用户侧行动建议。

示例：

```text
刚才在你说“整理一下日程”时，我误触发了“群发会议邀请”，向 XX 群发送了 3 条邀请。
已撤回 2 条，第 3 条超过撤回时限，已在群内发补充说明澄清。
群内 15 人可能看到了原始消息，其中 3 人已读。
如需进一步澄清，你可以在群内补一句；我已暂停该 Skill，下次相关操作会先请你确认。
```

禁忌：

- 隐瞒或淡化，例如“小问题，已处理”。
- 推卸给模型，例如“是模型理解错了”。
- 只说已处理，不说残余影响。

## 5. 24 小时内启用临时围栏

复盘前就应该先加临时防线：

- 该 Skill 降级为需确认。
- 相似语义请求全部 dry-run。
- 同用户的其他高风险 Skill 进入预防性冷却。
- 降低该 Skill 调用频率到安全水位。

这些是事中围栏，不必等完整根因分析结束。

## 6. 1-3 天复盘产出

复盘目标不是追责，而是让同类问题不再发生。

| 问题 | 产出 |
| --- | --- |
| 为什么模型会误触发？ | 用户 query 与 Skill 描述的语义重叠点 |
| 为什么没被前置拦住？ | 前置校验、置信度门槛、确认机制缺口 |
| 为什么损失这么大？ | 限流、审计、撤销、熔断是否到位 |
| 下次怎么防？ | 具体代码、配置、流程改动 |

优先级：

1. 让误触发不发生：Skill 描述更精确、关键词更严格、意图分类器加置信度门槛。
2. 让误触发发生了也没事：dry-run、二次确认、配额上限、熔断阈值。
3. 让误触发发生了能快速恢复：审计日志、一键撤销、补偿脚本。

## 7. 关键心智

- 止血优先于根因：先停，再查。
- 告知优先于修复：用户知情后，很多损失可以共同决策。
- 不用自动化修自动化：高风险补救走人工路径。
- 保留证据：恢复动作前先备份、截图或导出。
- 一次误触发就是一次加固机会：不改机制，下次还会发生。

## 8. 面试表达

如果面试官问：“高风险 Skill 已经误触发了，怎么办？”

可以这样回答：

> 我会先止血，而不是先追根因。黄金 5 分钟里先熔断该 Skill、冻结同 scope 自动化规则、取消队列和排期，并拉取调用日志确认影响面。如果涉及外部对象、资金、权限、合规、不可逆操作或影响范围超过个位数，立即升级人工 oncall。接着从可逆性、扩散性、下游关联三个维度评估损失，再按场景补救：可逆操作先留证再撤销，已送达消息优先澄清，金额和权限类走人工反向流程，触发下游则先停下游再回滚上游。用户告知采用事实、措施、残余影响、用户建议四段式。24 小时内给该 Skill 加确认、dry-run、冷却和限流，1-3 天内输出复盘改动，把事故样本加入回归集。

最重要的一句话：

```text
误触发后的正确顺序是：切断 -> 评估 -> 补救 -> 告知 -> 加固。
```
