# 重试策略：Tool 层、Skill 层与编排层的边界

重试策略不是“哪里失败就哪里多试几次”。正确答案是：Tool 层和 Skill 层都需要重试，但职责完全不同；编排层原则上不重试，只做补偿、降级和用户告知。

如果三层都随手加 3 次重试，很容易写出：

```text
3 x 3 x 3 = 27 次调用
```

这种代码看起来“更稳定”，实际会把下游打挂。

## 1. 分层原则：按失败语义划分

| 层级 | 重试什么 | 重试依据 | 典型次数 |
| --- | --- | --- | --- |
| Tool 层 | 瞬时性、物理性故障 | 网络、协议、传输层信号 | 2-3 次，毫秒级退避 |
| Skill 层 | 业务语义可恢复错误 | 业务错误码 + 上下文判断 | 0-1 次，秒级退避 |
| 编排层 | 不重试，只补偿或降级 | 业务决策 | 0 次 |

核心判据：

```text
这个错误换个时间再试，有没有可能成功？
```

- 能：靠近失败点就地重试，通常在 Tool 层。
- 不确定，需要业务上下文判断：上升到 Skill 层。
- 权限、参数、业务冲突等纯语义错误：绝不重试，直接上抛。

## 2. Tool 层该重试什么

Tool 层处理物理层抖动，要求错误和业务逻辑无关，并且重试幂等安全。

适合 Tool 层重试：

- 网络超时。
- 连接重置。
- DNS 抖动。
- HTTP 502、503、504。
- 429 限流，但必须尊重 `Retry-After`。
- Token 过期：刷新 token 后重试一次。

策略：

```text
attempt 1 -> fail -> wait 200ms +/- jitter
attempt 2 -> fail -> wait 500ms +/- jitter
attempt 3 -> fail -> 上抛 Skill 层
```

约束：

- 最多 2-3 次。
- 指数退避 + jitter。
- 总耗时通常不超过 5 秒。
- Tool 层必须保留原始错误语义，不能压成 `request failed`。

## 3. Skill 层该重试什么

Skill 层处理业务语义可恢复错误。它需要结合上下文判断是否重试。

适合 Skill 层处理：

- 依赖资源暂时不可用，例如文档正在被他人编辑锁定。
- 异步任务未就绪，例如会议纪要还在转写中。
- 幂等写入的“看似失败但可能已成功”。

例子：

```text
写 Base 记录返回超时
-> Skill 层用 client_token 查询是否已写入
-> 已写入：返回成功
-> 未写入：再写一次
```

策略：

- 最多 1 次业务重试。
- 轮询必须带总时长上限。
- 写操作必须具备 `client_token` 或 `idempotency_key`。

## 4. 不建议重试的错误

遇到这些错误，无论哪一层都应立即上抛：

- 4xx 客户端错误，除 408 和 429 外。
- 400 参数错误。
- 401/403 权限或鉴权失败。
- 404 资源不存在。
- 422 参数校验失败。
- 业务规则冲突，例如日程时间冲突、文档已删除。
- 非幂等写操作的未知状态，除非业务层能查询确认。

这些错误重试不会解决问题，只会放大问题。

## 5. 编排层的职责

编排层不应该继续重试下游调用。

它看到下游失败后应该：

1. 读取 Skill 返回的 `retriable` 字段。
2. 如果 `retriable=true`，最多触发一次明确的 Skill 级恢复动作。
3. 如果 `retriable=false`，走补偿、降级、告知用户路径。

编排层重试本质上是在已经重试过的基础上再重试，是重试风暴的常见来源。

## 6. 关键设计约束

### 6.1 Retry Budget

全链路要有重试预算。

```text
总尝试数 = Tool 层尝试 x Skill 层尝试 x 编排层尝试
```

建议规则：

- 全链路总尝试数不超过 4 次。
- 编排层默认 0 次重试。
- 下游异常时优先 fail-fast，而不是继续堆重试。

### 6.2 幂等性

幂等是重试的前提。

- 非幂等操作重试前必须带 `client_token` 或 `idempotency_key`。
- 下游不支持幂等时禁止重试，只能上抛。

### 6.3 错误码透传

Tool 层必须把原始错误语义透传给 Skill 层：

- HTTP status。
- 错误类型。
- `Retry-After`。
- 是否未知写入状态。
- 是否支持幂等查询。

否则 Skill 层无法判断能不能继续重试。

### 6.4 熔断配合

重试和熔断是一对：

- 连续 N 次失败，熔断器打开。
- 熔断期间跳过重试，直接 fail-fast。
- 避免下游已经挂了，上游还在疯狂重试。

## 7. 速查表

| 错误场景 | 重试层 | 次数 | 备注 |
| --- | --- | --- | --- |
| 网络超时 | Tool | <= 3 | 指数退避 |
| HTTP 503/504 | Tool | <= 2 | 带 jitter |
| 429 限流 | Tool | 按 `Retry-After` | 不要硬重试 |
| Token 过期 | Tool | 1 | 刷新后重试 |
| 异步任务未就绪 | Skill | 轮询 | 带总超时 |
| 幂等写超时 | Skill | 1 | 先查询后重试 |
| 400 参数错 | 不重试 | 0 | 立即上抛 |
| 403 权限 | 不重试 | 0 | 立即上抛 |
| 业务冲突 | 不重试 | 0 | 交给用户决策 |

## 8. 面试表达

如果面试官问：“重试策略应该放在 Skill 层还是 Tool 层？”

可以这样回答：

> 两层都要，但职责不同。Tool 层处理网络、协议、传输这类物理层抖动，例如超时、502/503/504、429、token 过期，使用指数退避加 jitter，最多 2-3 次。Skill 层处理业务语义可恢复错误，例如异步任务未就绪、资源暂时锁定、幂等写入超时后的查询确认，通常最多 1 次或带总时长上限的轮询。参数错误、权限不足、业务冲突和非幂等未知写入不重试。编排层原则上不重试，只根据 `retriable` 做补偿、降级或告知用户。全链路要有 retry budget，总尝试数最好不超过 4 次，并配合幂等键、错误码透传和熔断。

核心句：

```text
Tool 层管物理抖动，Skill 层管业务可恢复，编排层只决策不叠重试。
```
