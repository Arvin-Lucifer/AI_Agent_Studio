# 并行编排中的长尾延迟优化

多个子步骤都能并行时，整体耗时往往不是平均值决定的，而是由最慢的那个分支决定的。

```text
parallel_latency = max(branch_latency)
```

这就是 tail latency。一个分支拖到 P99，整个 Skill 的 P99 也会被拖上去。

优化目标有两个：

1. 让慢分支变快。
2. 让慢分支不阻塞整体交付。

## 1. 先诊断：慢的根因决定策略

没有 trace 数据就谈优化，基本是在猜。

| 根因 | 识别特征 | 适用优化 |
| --- | --- | --- |
| IO 等待 | CPU 空闲，网络或磁盘活跃 | 流式处理、分片、预取 |
| 数据量大 | 耗时随输入规模线性增长 | 分页并发、MapReduce |
| 服务端慢 | 单次调用就慢，本地无法优化 | 异步化、预测性提前触发 |
| 伪并行 | 外层并行，内部仍是顺序 for 循环 | 改成真并发 |

先测量：

- 每个子步骤的 P50/P95/P99。
- 是否有内部串行循环。
- CPU、网络、磁盘、下游 API 的等待比例。
- 慢步骤结果是否是主响应的关键依赖。

## 2. 核心优化手段

### 2.1 关键路径剥离

把慢步骤从“必须完成才能响应”降级为“后台继续”。

适用：慢步骤不是主决策依据，例如生成配图、归档、统计、长报告润色。

模式：

- 先返回快分支结果。
- 响应里标记 `pending`。
- 慢分支完成后通过 IM、通知、前端轮询或 WebSocket 补齐。

示例：

```text
用户请求生成会议纪要 + 生成配图
-> 会议纪要 2s 返回
-> 配图 18s 后追加
```

### 2.2 提前启动

慢步骤如果只依赖用户输入，就不要等前置步骤全部完成。

适用：模板预取、权限预拉、素材预处理、候选检索。

坏流程：

```text
校验 -> 创建文档 -> 拉模板 -> 拉权限
```

好流程：

```text
入口同时启动：校验、拉模板、拉权限
校验通过后直接创建文档
```

### 2.3 分片并发

慢的根因是“量大”时，把一个慢任务拆成多个小任务。

例子：

- 1000 条记录写 Base，拆成 10 个 100 条并发写。
- 大文档解析，按章节并行处理。
- 大批量检索，按 shard 并发召回再合并。

约束：

```text
分片数 x 单片 QPS <= 下游配额
```

分片并发不是越多越好，超过下游配额会把优化变成限流事故。

### 2.4 软超时 + 降级

给慢步骤设置软超时，不是硬 kill。

超时后：

- 先返回降级结果。
- 标记慢结果 `pending`。
- 慢结果异步完成后覆盖更新。

示例：

```text
Minutes 精要获取超过 5s
-> 先返回逐字稿
-> 精要完成后追加更新
```

### 2.5 对冲请求

当慢步骤达到 P95 分位还没返回时，发起第二个相同请求，取先返回的结果。

适用：

- 只读请求。
- 幂等请求。
- 下游有冗余容量。
- 长尾尖刺来自偶发实例抖动。

成本：

- 增加一次调用成本。
- 需要去重和取消慢副本。
- 不能用于非幂等写操作。

### 2.6 缓存与预计算

慢步骤输出可复用时，应优先缓存或提前算好。

方式：

- 按 `(user_id, 参数指纹)` 做 LRU 缓存。
- 周期性任务用 scheduler 预计算。
- 热门模板、权限、配置提前预取。

适用：模板渲染、权限信息、常用统计、周期性报告。

## 3. 选择策略的判断树

```text
慢步骤是否主响应必需？
  否 -> 关键路径剥离 / 软超时降级
  是 -> 慢的原因是什么？
        IO 等待 -> 流式 / 预取 / 提前启动
        数据量大 -> 分片并发 / MapReduce
        下游服务慢 -> 对冲请求 / 异步化
        内部伪并行 -> 改真并发
```

## 4. 关键工程约束

### 4.1 Trace 先行

优化前要能看到每个子步骤：

- `step_name`
- `start_time`
- `end_time`
- `duration_ms`
- `dependency`
- `critical_path`
- `result_state`

没有 trace，就不知道该优化哪条路径。

### 4.2 明确关键路径

每个子步骤都要标注：

- `blocking`：阻塞主响应。
- `non_blocking`：可以后台完成。
- `optional`：可丢弃或降级。

### 4.3 限流与背压

Fan-out 和 hedging 都会增加下游压力。必须配合：

- 并发上限。
- QPS 上限。
- 队列长度上限。
- 熔断和降级。

### 4.4 结果一致性

慢结果后补时，要考虑：

- 前端如何更新。
- 用户是否需要通知。
- 旧结果是否会被覆盖。
- 后补失败是否要提示。

## 5. 反模式

- 只看平均耗时，不看 P95/P99。
- 所有并行分支都 join 后才返回，即使有些不是关键结果。
- “并行”函数里套顺序 for 循环。
- Fan-out 不考虑下游 QPS，导致限流。
- Hedging 用在非幂等写操作。
- 软超时后没有后补机制，用户永远拿不到完整结果。

## 6. 面试表达

如果面试官问：“多个子步骤都能并行，但其中一个慢得多，如何优化？”

可以这样回答：

> 并行编排的总耗时由最慢分支决定，所以先用 trace 定位慢分支根因：IO 等待、数据量大、下游服务慢，还是内部伪并行。然后看慢结果是否在关键路径上。如果不是主响应必需，就做关键路径剥离，先返回快结果，慢结果 pending 后补；也可以设置软超时，超时先返回缓存、默认值或简化版。若慢是因为数据量大，就分片并发或 MapReduce，但要受 QPS 和并发上限约束。若慢来自下游服务长尾，可以提前启动或对只读幂等请求做 hedging。若结果可复用，用缓存和预计算。整个过程中要有 trace、关键路径标记、限流背压和后补一致性设计。

核心句：

```text
要么让慢分支变快，要么让慢分支不阻塞主响应。
```
