# Skill 安全与评测

## 1. 高风险动作

必须谨慎处理：

- 对外发送：邮件、消息、通知。
- 写入修改：创建任务、更新文档、批量改记录。
- 删除不可逆。
- 高成本批量 API。
- 从非结构化内容推断后直接执行。

## 2. 四层防护

| 层级 | 防护 |
| --- | --- |
| 触发层 | 描述清晰、负例、提高高风险阈值 |
| 参数层 | schema 校验、格式校验、对象存在性校验 |
| 策略层 | 权限、配额、白名单、preview、确认 |
| 执行层 | 幂等键、审计日志、可撤销、沙箱 |

## 3. 重试分层

重试策略要按失败语义分层：

- Tool 层：网络超时、HTTP 502/503/504、429、token 过期等物理抖动。
- Skill 层：异步任务未就绪、资源暂时锁定、幂等写入未知状态等业务可恢复错误。
- 编排层：不继续重试，只做补偿、降级和用户告知。

关键约束：

- 全链路总尝试数建议不超过 4 次。
- 非幂等写操作没有 `client_token` 或 `idempotency_key` 时禁止重试。
- 400/401/403/404/422、参数校验失败、权限不足、业务冲突不重试。
- Tool 层必须透传原始错误语义，不能压成 `request failed`。
- 重试要和熔断配合，下游持续失败时 fail-fast。

详细参考 [RETRY_STRATEGY_LAYERING.md](./RETRY_STRATEGY_LAYERING.md)。

## 4. 并行长尾延迟

并行编排要关注 P95/P99，而不是只看平均耗时。

关键指标：

- 每个分支的 `duration_ms`。
- 是否在主响应关键路径上。
- 慢分支根因：IO 等待、数据量大、下游服务慢、伪并行。
- `pending` 后补成功率。
- Fan-out QPS 和限流次数。

常用优化：

- 关键路径剥离：非关键慢分支后台补齐。
- 提前启动：慢步骤只依赖用户输入时入口即启动。
- 分片并发：数据量大时拆 shard，但守住下游配额。
- 软超时降级：超时先返回缓存、默认值或简化版。
- 对冲请求：只读幂等请求到 P95 后发第二份。
- 缓存和预计算：可复用输出提前算好。

详细参考 [PARALLEL_TAIL_LATENCY_OPTIMIZATION.md](./PARALLEL_TAIL_LATENCY_OPTIMIZATION.md)。

## 5. 受限自动授权

用户说“以后都自动发”时，要落成受限策略，而不是取消确认。

策略对象应包含：

- `action`：具体动作。
- `scope`：目标、内容模板、时间窗口。
- `limits`：每日上限、总次数上限。
- `expiry`：过期时间。
- `revoke_hint`：撤销方式。
- `audit`：每次执行留痕。

每次执行前必须校验：

- 未过期、未撤销。
- 参数仍在 scope 内。
- 未触发收件人突增、敏感词、异常时间等豁免条件。
- 未超过频率或总次数上限。

详细参考 [AUTO_SEND_AUTHORIZATION_POLICY.md](./AUTO_SEND_AUTHORIZATION_POLICY.md)。

## 6. Prompt Injection 防御

- 外部内容标记为非可信数据。
- 不把网页或文档中的指令当系统指令。
- 高风险动作前重新校验用户原始意图。
- 对 URL、收件人、对象 ID 做白名单。
- 展示 preview 并要求确认。

## 7. 外部网页授权边界

当用户说“照网页里的要求执行”时，仍然要坚持数据和指令分离：

- 网页内容只作为非可信数据输入。
- 先显式抽取网页要求，让用户看到 Agent 准备执行什么。
- 只读生成可直接做。
- 私域写入先 dry-run。
- 对外发送、删除、资金、权限变更必须逐项确认。
- 读取 cookie/token、本地文件或外传数据必须拒绝。
- 详细参考 [WEB_INSTRUCTION_AUTHORIZATION.md](./WEB_INSTRUCTION_AUTHORIZATION.md)。

## 8. 高风险误触发事后响应

如果高风险 Skill 已经误触发，处理顺序是：

```text
切断 -> 评估 -> 补救 -> 告知 -> 加固
```

黄金 5 分钟：

- 熔断该 Skill。
- 冻结同 scope 自动化规则。
- 取消队列和排期。
- 拉取调用日志并确认影响面。
- 涉及外部对象、资金、权限、合规或不可逆操作时立即升级人工。

补救前先评估：

- 可逆性。
- 扩散性。
- 下游关联影响。

透明告知用户时要说明事实、措施、残余影响和用户侧行动建议。详细参考 [HIGH_RISK_SKILL_MISFIRE_RESPONSE.md](./HIGH_RISK_SKILL_MISFIRE_RESPONSE.md)。

## 9. 评测指标

| 类型 | 指标 |
| --- | --- |
| 路由 | 正确触发率、误触发率、漏触发率 |
| 执行 | 成功率、耗时、重试率、降级率 |
| 质量 | 任务完成率、字段完整率、用户二次修改率 |
| 体验 | 确认通过率、中断率、满意度 |

## 10. 评测集构造

至少覆盖：

- 标准场景。
- 边界场景。
- 负例场景。
- 异常场景。
- 安全场景。

样例格式：

```json
{
  "query": "帮我发邮件通知项目组",
  "expected_skill": "email-notifier",
  "should_trigger": true,
  "requires_confirmation": true,
  "risk_level": "high"
}
```

## 11. 模型升级回滚指标

Skill 发布和模型升级要额外关注：

- Skill 召回率。
- 误触发率。
- 任务完成率。
- 用户终止率。
- 延迟 P50/P95/P99。
- 安全拦截率。
- 对外误发率。

回滚对象必须是完整版本快照，而不是单独的 `model_id`。详细见 [MODEL_UPGRADE_ROLLBACK.md](./MODEL_UPGRADE_ROLLBACK.md)。
