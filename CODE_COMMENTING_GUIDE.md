# 课程代码注释规范

这份规范用于已经整理的代码，也用于后续新增课程代码。目标不是把每一行都翻译成中文，而是让学员能读懂关键模块为什么这样写、出了问题该往哪里看。

## 基本原则

- 每个练习脚本开头用模块 docstring 说明学习目标、核心流程和运行方式。
- 课程代码需要“导读型注释”：不只解释这一行做什么，还要告诉学员这段代码应该怎么看。
- 注释优先解释“为什么”和“边界条件”，不要重复描述显而易见的语法。
- 关键流程前加短注释：环境加载、client 构造、prompt/schema、模型调用、解析兜底、工具执行、并发、错误恢复。
- 函数名已经清楚时不用硬写注释；函数承担课程知识点时补一行 docstring。
- 不在注释里写真实密钥、账号、内部地址或不可公开的配置。

## 导读型注释要求

- 文件开头要给出阅读路线，例如“先看模型构造，再看工具定义，最后看 Agent 调用”。
- 每个关键代码块前要解释它对应的课程概念，例如 `@tool` 对应工具 schema，`thread_id` 对应会话隔离。
- 框架封装处要解释它替代了前面哪段手写逻辑，例如 `create_agent` 替代工具调用循环。
- mock 数据旁边要说明真实项目里会替换成什么，例如 API、数据库、搜索服务或文件系统。
- 安全边界旁边要说明风险来源，例如路径穿越、任意代码执行、上下文过长或跨用户记忆泄漏。
- 调试开关旁边要说明观察目标，例如 trace 用来看 tool_calls，debug query 用来看模型如何改写搜索词。

## Agent 代码必须讲清的模块

- `messages`：说明它保存了哪些角色消息，以及为什么它就是最小上下文/短期记忆。
- `system prompt`：说明它约束角色、边界、输出格式或工具使用策略。
- 结构化输出：说明 prompt 只是约定，代码仍要解析、校验和处理失败。
- `tools` schema：说明这是给模型看的工具说明书，不是实际执行函数。
- 工具注册表：说明模型返回工具名后，代码如何路由到真实函数。
- 工具结果回填：说明 `tool_call_id` 如何把一次工具调用和结果对应起来。
- 重试与超时：说明网络/API 调用会失败，课程代码如何做最小恢复。
- 并发工具调用：说明哪些工具可以并发，以及为什么仍要保留可追踪的结果顺序。

## 后续章节检查清单

- 新增 Python 练习脚本前 30 行内能看出本文件的学习目标。
- 新增 Python 练习脚本前 40 行内要能看出推荐阅读顺序或关键观察点。
- 读者能从注释中理解本章关键概念，而不需要先看完整讲义。
- 关键框架 API 附近要说明“它帮我们省掉了什么手写逻辑”。
- 所有外部调用都有最小错误处理说明，例如超时、限流、解析失败、工具失败。
- 所有 mock 数据都说明它模拟的真实系统来源，例如数据库、API、消息队列或本地文件。
- 如果代码引入框架，至少注释入口、状态对象、节点/链路、回调或评估逻辑。
